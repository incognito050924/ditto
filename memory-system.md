# 목적
1. Agentic Coding 에 도움을 줄 수 있는 장기 기억 메모리 시스템을 만든다. (코드 베이스에 대한 이해를 잘해야 함)
2. 프로젝트/프로덕트 개발을 위한 지식 관리 시스템을 구축한다. (개인/팀/조직)
3. claude code (codex) 사용하여 에이전트 또는 사용자 (사용자 표면 활용)는 지식 베이스를 기반한 정보에 접근하거나 컨설팅 질문등에 대한 답을 얻을 수 있다.
4. 단순 데이터의 아카이브가 아닌 방대한 양의 장서와 자료를 소장한 도서관과 그 안에서 원하는 자료를 delivery 하거나 사실 관계를 확인하거나, 결정에 도움이 되는 답변을 제공할 수 있는 사서/curator가 있는 시스템을 목표로 한다.
5. 시스템의 모든 결과는 명확한 근거나, 논리적 모순/거짓이 없어야 하며, 확실하지 않은 것에 대해서 섣부른 가정이나 추측은 배제하거나 명확하게 드러내야 한다. 또한 기존의 의사결정들이나 규칙들이 항상 반영하여야 한다.
6. 장기 기억 메모리로 세션동안만 유지되는 컨택스트를 확장 및 보완한다.

# 목표
1. LLM, 에이전트의 할루시네이션을 최소화하는 것 (LLM 은 추론모델로 입력에 대해서 적당한 결과를 만들어 내는 모델임, 여기에는 그 어떤 사실 관계나 근거가 없음. 하지만 )
2. 사람의 기억은 휘발성이 있고, 신뢰도 역시 강하다고 볼 수는 없다. 다만, 적절한 맥락에서는 신뢰도 회복 가능하고, 여러 환경에서의 적응이 가능하기 때문에 성능에 비해 높은 신뢰도를 달성할 수 있다로 생각함. 하지만, AI 는 용량의 제한이 인간에 비할 바 없이 크고, 휘발성이 현저히 적고, 많은 처리가 가능하다. 하지만 한 번 생겨난 편향이나, 이로 인해 발생한 결과들에 대해서는 계속 쌓여 SLOP 이 될 수 있다(자가 교정 능력이 거의 없는 수준이지만 다른 이점이 너무 큼). 그래서 하네스로 어느 정도 교정이 발생할 가능성을 만들어 신뢰성을 더 높임 (여러가지 근거들을 활용할 수 있도록)
3. 프로젝트/프로덕트의 코드베이스는 -요즘 같은 Agentic coding 이 만연한 시대에는 덜 하지만, 위에서 언급했듯 LLM 은 의도가 없음- 단순히 코드 산출물이 아니다. 요건 정의부터 기획, 설계, 구현 그 모든 단계를 수 없이 반복하면서 쌓아온 수 많은 의사 결정들, 개발자들의 고민이나 의도들이 녹아져 있다. 하지만 이러한 것들은 숨겨져 있고, 잘 드러나거나 알아내기가 어렵다. 이것을 개발자 단 한명, 친한 소수 몇 명만 알고 있다면 그 순간에도 그 밖의 누군가는 이들의 의도와 부딪히는 작업을 하고 있을 수 있다. 그래서 모든 의도, 의사결정, 모호성등을 투명하게 드러낼 수 있는 방법이 필요하다.
4. 장기 기억 메모리이지만 한번에 로드 되는게 아니라 필요한 시점에 필요한 부분만 점진적으로 획득할 수 있도록 한다. (토큰 및 컨택스트 윈도우 비용 최적화, context rot 은 반드시 피해야 함)

# 고려 사항
- RAG 도입: 단순 조회와 서빙이라면 도입 효과가 떨어지지만 특정 지식이나 가치를 만들어내려면 rag 필요 
- GraphDB 도입: 기억이나 컨택스트는 고맥락이 필요함. 고맥락 작업은 주로 어떤 사실들이 이 작업과 어떤 관계가 있는지 파악하는 것(e.g. coding 또한 엔티티/함수/데이터 간의 흐름이나 관계를 설계하는 것)이므로 관계를 표현하는 graph db 를 적용해보는것은 어떨까 (memgraph/neo4j, memgraph 는 vectordb, rag 기능도 확장으로 사용 가능)  
- 개인 영역과 팀/조직을 위한 영역 분리 고려
- 코드 베이스를 개발해나가는데 주로 사용되기 때문에 git을 배제할 수 없다. 하지만 git 은 로컬과 원격으로 구분되어 있다보니 이 경계에 대한 고려도 필요함. 예를 들어 지식은 내 로컬에 있는 내용을 최신화 되어 있지만 다른 사람들은 푸시되지 않은 최신 코드베이스가 아니므로 최신의 지식이 맞지 않을 수 있음. 또는 다른 브랜치로 스위치하는 경우도 마찬가지 (ditto memory 의 신선도 2축이 이것때문, 커밋 해시를 활용해야 함, 지식에 커밋 해시를 라벨링 하던가)

사람의 단기 기억과 장기 기억
기억의 신선도에 따라 컨택스트에 중요도가 높을 확률
장기 기억은 RAG로 활용
단기기억은 Graph로 활용
기억이나 컨택스트의 중요도는 시간에 따라 감소하는 형태로 표현하는게 합리적인지 확인 필요
기억은 관계가 중요,
절대 잊지 않는 기억 시스템 vs. 시간 흐름에 따라 잊히거나 편향이 생길수도 있는 시스템 
기억은 장면/사실에 약간의 컨택스트가 포함된 것
