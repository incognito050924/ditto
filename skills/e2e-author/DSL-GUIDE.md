# DITTO E2E 여정 DSL 작성 가이드

> **목적**: 사용자 여정(서비스를 쓰는 사람이 밟는 화면 흐름)을 전용 마크업(DSL)으로 적는 법을 알려준다. 이렇게 적은 여정은 DITTO가 실제 브라우저에서 반복 실행되는 Playwright 테스트로 변환한다 — 여러분이 Playwright 코드를 알 필요는 없다.
> **소비자**: DITTO 사용자. 개발 지식이 없어도 따라 쓸 수 있어야 한다.
> **갱신 조건**: DSL 문법(front-matter 필드, 단계 동사, 확인 문형, 조건·블록·케이스 규칙)이 바뀔 때 이 문서를 함께 갱신한다. 문법 버전: `ditto_journey: v1`.

---

## 1. 시작하기 — 첫 여정 파일 만들기

여정 하나 = 마크다운 파일 하나. 대상 프로젝트 저장소의 `e2e/journeys/` 아래에 만든다.

```
e2e/
  journeys/
    login-basic.journey.md        ← 여정 파일 (사람이 쓴다)
    blocks/
      blk-login.block.md          ← 재사용 블록 (사람이 쓴다, §6)
  generated/
    login-basic.spec.ts           ← 생성된 테스트 (DITTO가 쓴다 — 직접 수정 금지)
```

- 여정 파일 이름: `<아무 이름>.journey.md`
- 파일 구조: 맨 위 front-matter(`---`로 감싼 정보 블록) + 본문(번호 매긴 단계 목록) + 필요하면 `## 케이스` 테이블

가장 작은 여정은 이렇게 생겼다.

```markdown
---
ditto_journey: v1
id: jrn-login-basic
name: 기본 로그인
description: 등록된 사용자가 이메일·비밀번호로 로그인할 수 있다. 깨지면 모든 사용자가 서비스에 못 들어온다.
surfaces:
  - page:/login
uses_blocks: []
flaky_history: []
---

1. [s1] 방문: /login
2. [s2] 입력: "이메일" 칸에 user@example.com
3. [s3] 입력: "비밀번호" 칸에 secret123
4. [s4] 클릭: "로그인" 버튼
5. [s5] 확인: url contains /dashboard
```

직접 처음부터 쓸 필요는 없다 — DITTO와의 대화로 함께 작성하는 것이 기본 흐름이다. 여정의 의도(무엇을, 왜)는 여러분이 내고, 코드 사실(화면 요소의 셀렉터, 경로)은 에이전트가 조사해 채운다.

## 2. front-matter — 7개 필드

| 필드 | 필수 | 뜻 | 예 |
|---|---|---|---|
| `ditto_journey` | 필수 | 문법 버전 표식. 항상 `v1` | `ditto_journey: v1` |
| `id` | 필수 | 안정 식별자. `jrn-` + 소문자-하이픈(kebab). 한번 정하면 바꾸지 않는다 | `jrn-checkout-coupon` |
| `name` | 필수 | 사람이 보고 식별하는 이름 | `쿠폰 적용 결제` |
| `description` | 필수 | 이 여정의 목적·가치 — **왜 깨지면 안 되는가** | `쿠폰 할인이 결제 금액에 반영된다. 깨지면 매출 직결.` |
| `surfaces` | 필수 | 이 여정이 지나가는 표면 목록. `page:<경로>` / `api:<METHOD> <경로>` / `component:<저장소 경로\|glob>` 세 형태만 허용. **코드 변경과의 기계 교차(영향 추림)는 `component:`만 한다** — 값은 컴포넌트 이름이 아니라 저장소 파일 경로/glob이어야 매치된다. `page:`/`api:`는 사람·에이전트가 영향을 식별하는 메타데이터다 | `- page:/checkout` `- api:GET /api/coupons/validate` `- component:src/components/coupon/**` |
| `uses_blocks` | 필수 | 본문에서 참조하는 재사용 블록 id 목록. 안 쓰면 `[]` | `- blk-login` |
| `flaky_history` | 필수 | 간헐 실패(flaky) 판정 이력. 새 여정은 `[]` — **DITTO가 기록**하므로 직접 채우지 않는다. 이 필드는 digest 계산에서 제외되므로 기록이 쌓여도 테스트 재생성이 필요하지 않다 | `[]` |

## 3. 본문 — 단계 줄 쓰는 법

단계 한 줄의 모양:

```
N. [s<번호>] (조건)? <동사>: <목적어>
```

- `N.` — 마크다운 번호 목록.
- `[s<번호>]` — 단계의 안정 id (`[s1]`, `[s2]`, …). 테스트 실패가 "어느 단계에서"로 보고될 때 이 id가 쓰인다. 단계를 중간에 끼워 넣어도 기존 id는 바꾸지 않는다.
- `(조건)` — 선택. 이 단계를 특정 케이스·변수 상황에서만 실행한다 (§7).
- `<동사>: <목적어>` — 아래 7개 동사 중 하나 + 콜론 + 무엇에 대해서인지.

### 동사 7개

| 동사 | 뜻 | 예 |
|---|---|---|
| `방문` | 페이지로 이동한다 | `방문: /checkout` |
| `클릭` | 요소를 클릭한다 | `클릭: "결제하기" 버튼` |
| `입력` | 입력칸에 값을 친다 | `입력: "쿠폰 코드" 칸에 {coupon}` |
| `선택` | 드롭다운·라디오 등에서 고른다 | `선택: "배송 방법"에서 택배` |
| `대기` | 요소나 상태가 나타날 때까지 기다린다 | `대기: "주문 요약" 영역 visible` |
| `확인` | 기대 결과를 검증한다 — 목적어는 §4의 확인 5형 중 하나 | `확인: "총 결제금액" contains {expected_total}` |
| `블록` | 재사용 블록을 통째로 실행한다 (§6) | `블록: blk-login (user={user}, password=secret123)` |

### 대상(target) 적는 법 — 3가지

클릭·입력·확인 등의 목적어에서 화면 요소를 가리킬 때:

| 방식 | 예 | 언제 |
|---|---|---|
| 사람 라벨 (따옴표) | `"로그인" 버튼`, `"쿠폰 코드" 칸` | **기본값.** 화면에 보이는 그대로 적는다 — 실제 셀렉터 해석은 에이전트의 몫 |
| `[data-testid=…]` | `[data-testid=coupon-input]` | 개발자가 테스트 식별자를 달아 둔 경우 (가장 안정적) |
| CSS 셀렉터 | `#summary .total` | 셀렉터를 아는 경우만. 모르면 사람 라벨로 충분하다 |

## 4. 확인(assertion) — 5가지 문형

`확인:` 동사의 목적어는 반드시 아래 5형 중 하나다. 자유 문장(예: "결제가 잘 된다")은 기계가 판정할 수 없어 미검증으로 남는다.

| 문형 | 뜻 | 예 |
|---|---|---|
| `<target> contains <텍스트>` | 요소가 해당 텍스트를 포함한다. 텍스트 자리에 `{변수}` 가능 | `확인: "총 결제금액" contains 9,000원` |
| `<target> visible` | 요소가 화면에 보인다 | `확인: "주문 완료" 메시지 visible` |
| `<target> hidden` | 요소가 화면에 안 보인다 | `확인: "로딩 스피너" hidden` |
| `<target>` | 요소가 존재한다 (target만 단독으로) | `확인: [data-testid=order-id]` |
| `url contains <텍스트>` | 현재 주소(URL)에 해당 텍스트가 포함된다 | `확인: url contains /orders/` |

## 5. 데이터 케이스 — 한 여정을 여러 데이터로 검증

같은 여정을 다른 입력값(유효 쿠폰/만료 쿠폰/쿠폰 없음…)으로 반복 검증하려면, 본문 아래에 `## 케이스` 제목과 마크다운 테이블을 둔다. 케이스마다 별도 테스트로 변환되고, 실패도 케이스 단위로 보고된다.

테이블 규칙:

- **첫 열** = 케이스 이름 (사람이 읽는 식별자). 선언한 모든 케이스는 생성 테스트에 반드시 존재해야 한다 — 변환에서 케이스가 빠지면 정합성 검사가 실패한다
- **마지막 열** = `유형` — `성공` / `실패` / `경계` 중 하나
- **가운데 열들** = 변수. 열 이름이 곧 변수 이름이고, 본문에서 `{변수이름}`으로 치환된다
- `—` = 그 케이스에서 변수를 설정하지 않음 (미설정)

```markdown
## 케이스

| 케이스 | coupon | expected_total | 유형 |
|---|---|---|---|
| 유효 쿠폰 | WELCOME10 | 9,000원 | 성공 |
| 만료 쿠폰 | EXPIRED99 | — | 실패 |
| 쿠폰 없음 | — | 10,000원 | 성공 |
```

### 변수 규칙

- 변수는 **케이스 테이블에서만 정의**한다. 본문에서는 `{coupon}`처럼 치환해 쓰기만 한다.
- 본문에서 새 변수를 만들거나 값을 바꾸는 문법은 없다 (의도된 제한 — §8 첫째 항목).
- 케이스 테이블이 없는 여정에는 `{변수}`를 쓸 수 없다.

## 6. 재사용 블록 — 공통 절차를 한 번만 정의

로그인처럼 여러 여정에 반복되는 절차는 블록으로 뽑는다. 블록을 고치면 참조하는 모든 여정의 테스트에 반영된다 (변환 시 공유 helper 함수가 된다).

### 블록 정의

위치: `e2e/journeys/blocks/<블록 id>.block.md`. 단계 id는 `b` 접두를 쓴다.

```markdown
---
ditto_block: v1
id: blk-login
name: 로그인
params:
  - user
  - password
---

1. [b1] 방문: /login
2. [b2] 입력: "이메일" 칸에 {user}
3. [b3] 입력: "비밀번호" 칸에 {password}
4. [b4] 클릭: "로그인" 버튼
5. [b5] 대기: url contains /dashboard
```

- `ditto_block: v1` — 블록 문법 버전 표식
- `id` — `blk-` 접두 권장, **파일 이름(stem)과 반드시 일치** — 불일치는 정합성 검사가 거부한다
- `params` — 블록이 받는 매개변수 선언. 본문에서 `{user}`처럼 쓴다

### 블록 참조

여정 본문에서 `블록` 동사로 호출하고, 매개변수를 `(이름=값, …)`으로 넘긴다. 값 자리에는 고정값도 `{변수}`도 올 수 있다.

```markdown
3. [s3] 블록: blk-login (user=buyer@example.com, password=secret123)
```

참조하는 블록은 front-matter의 `uses_blocks`에도 반드시 올린다 — 코드 변경의 영향 추림이 이 선언을 본다.

## 7. 선언적 조건 — 단계를 케이스에 따라 켜고 끄기

단계 앞에 `(조건)`을 붙이면 그 단계가 조건이 맞는 케이스에서만 실행된다. 허용되는 조건은 **딱 3형**이다.

| 조건 | 뜻 | 예 |
|---|---|---|
| `(<변수> 있음)` | 이 케이스에서 변수가 설정됨 (`—` 아님) | `(coupon 있음) 입력: "쿠폰 코드" 칸에 {coupon}` |
| `(<변수> 없음)` | 이 케이스에서 변수가 미설정 (`—`) | `(coupon 없음) 확인: "할인" 줄 hidden` |
| `(케이스: <이름들>)` | 나열한 케이스에서만 실행 (쉼표로 여러 개) | `(케이스: 만료 쿠폰) 확인: "만료된 쿠폰입니다" visible` |

**이 3형으로 표현 못 하는 분기는 여정을 분리한다.** 중첩 조건, "A이면서 B가 아닐 때", 단계 결과에 따른 갈림 같은 흐름이 필요해지면 — 그것은 하나의 여정이 아니라 두 개의 여정이다. 각각 파일을 나눠 단순하게 쓴다. DSL에 if/else를 흉내 내지 않는다.

## 8. 전체 예시 — 쿠폰 적용 결제

블록 2개 참조 + 케이스 4개를 갖춘 완전한 여정. (`e2e/journeys/checkout-coupon.journey.md`)

```markdown
---
ditto_journey: v1
id: jrn-checkout-coupon
name: 쿠폰 적용 결제
description: 쿠폰 할인이 결제 금액에 정확히 반영되고, 잘못된 쿠폰은 거부된다. 깨지면 매출·정산에 직결.
surfaces:
  - page:/checkout
  - api:POST /api/coupons/validate
  - component:src/components/coupon/**
uses_blocks:
  - blk-login
  - blk-add-to-cart
flaky_history: []
---

1. [s1] 블록: blk-login (user={user}, password=secret123)
2. [s2] 블록: blk-add-to-cart (product=기본 티셔츠)
3. [s3] 방문: /checkout
4. [s4] (coupon 있음) 입력: "쿠폰 코드" 칸에 {coupon}
5. [s5] (coupon 있음) 클릭: "쿠폰 적용" 버튼
6. [s6] (케이스: 만료 쿠폰) 확인: "사용할 수 없는 쿠폰입니다" visible
7. [s7] (coupon 없음) 확인: "할인" 줄 hidden
8. [s8] (케이스: 유효 쿠폰, 쿠폰 없음, 최소금액 경계) 클릭: "결제하기" 버튼
9. [s9] (케이스: 유효 쿠폰, 쿠폰 없음, 최소금액 경계) 확인: url contains /orders/
10. [s10] (케이스: 유효 쿠폰, 쿠폰 없음, 최소금액 경계) 확인: "총 결제금액" contains {expected_total}

## 케이스

| 케이스 | user | coupon | expected_total | 유형 |
|---|---|---|---|---|
| 유효 쿠폰 | buyer@example.com | WELCOME10 | 9,000원 | 성공 |
| 만료 쿠폰 | buyer@example.com | EXPIRED99 | — | 실패 |
| 쿠폰 없음 | buyer@example.com | — | 10,000원 | 성공 |
| 최소금액 경계 | buyer@example.com | MIN5000 | 5,000원 | 경계 |
```

읽는 법: 케이스가 4개이므로 테스트도 4개 생성된다. "만료 쿠폰" 케이스는 s6에서 오류 메시지를 확인하고 끝난다(결제 단계 s8~s10은 그 케이스에 포함되지 않음). "쿠폰 없음" 케이스는 s4·s5를 건너뛰고 s7에서 할인 줄이 없는 것을 확인한다.

## 9. 자주 하는 실수

| 실수 | 왜 안 되는가 | 대신 |
|---|---|---|
| 범용 언어처럼 쓰기 — `만약 X면 … 아니면 …`, 본문에서 변수 대입, 반복문 흉내 | DSL은 프로그래밍 언어가 아니다. 조건은 §7의 3형뿐 | 케이스 테이블로 데이터를 나누거나, 여정 파일을 분리 |
| `e2e/generated/` 아래 생성 테스트 파일을 직접 수정 | 생성물은 DSL에서 파생된다(파일 머리의 `@ditto-generated` 표식). 직접 고친 내용은 다음 재생성 때 사라진다. 기계가 감지하는 것은 DSL 변경 후 미재생성(digest)과 단계 마커 누락이며, 모든 수동 편집이 자동 감지되지는 않는다 | 여정 `.journey.md`를 고치고 DITTO에게 재생성을 맡긴다 |
| `확인:`에 자유 문장 — `확인: 결제가 잘 된다` | 기계가 판정할 수 없어 미검증으로 남는다 | §4의 5형 중 하나로 — `확인: "주문 완료" visible` |
| `surfaces`를 비우거나 대충 적기 | 코드 변경 시 영향받는 테스트를 이 선언으로 추린다. 빠지면 깨진 여정이 검증 없이 통과할 수 있다 | 여정이 지나는 페이지·API·컴포넌트를 빠짐없이, 접두(`page:`/`api:`/`component:`)와 함께. 기계 추림에 걸리려면 `component:`에 저장소 경로/glob을 적는다 — 컴포넌트 이름만 적으면 어떤 변경과도 매치되지 않는다 |
| 블록을 본문에서 쓰고 `uses_blocks`에 누락 | 영향 추림과 변환이 선언을 본다 | `블록:` 호출과 `uses_blocks` 목록을 항상 일치시킨다 |
| 단계 id(`[s3]`) 재번호 매기기 | 실패 보고·이력이 단계 id로 연결된다 | 중간에 단계를 끼워도 기존 id는 유지하고 새 번호를 이어 쓴다 (`[s11]` 등) |
| `flaky_history`를 손으로 채우기 | flaky(간헐 실패) 판정은 사용자 확인을 거쳐 DITTO가 기록한다 | 새 여정은 `[]`로 두고 건드리지 않는다 |
| 변수 정의 없이 `{var}` 사용 | 변수는 케이스 테이블 열에서만 생긴다 | `## 케이스` 테이블에 해당 열을 추가 |
