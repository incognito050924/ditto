# Graphify 깊이 읽기 — 그리고 우리 메모리 설계와의 관계

> **이 문서를 왜 읽나**: 우리는 "코드·문서·작업기록을 에이전트가 오래 기억하고 추론하게 만드는 메모리 시스템"을 설계했다(파일: `agent-intelligence-memory-report.md`, 이하 **메모리 보고서**). 그 보고서는 Graphify라는 오픈소스 도구를 "참고만 한다"고 적었는데, 정작 Graphify가 실제로 어떻게 생겼는지는 추상적으로만 다뤘다.
>
> 이 문서는 Graphify를 **실제로 뜯어본 결과**다. 그래서 (1) Graphify가 진짜 무엇이고 어떻게 도는지 쉽게 설명하고, (2) 우리 보고서가 빌려온 아이디어가 실제로 어떻게 구현돼 있는지 짝지어 보여주고, (3) 보고서가 틀렸거나 빠뜨린 부분을 짚는다.
>
> **이 문서는 메모리 보고서를 안 읽어도 따라갈 수 있게 썼다.** 필요한 배경은 본문에서 그때그때 풀어 설명한다.
>
> **소비자**: 우리 메모리 시스템을 설계·구현하는 사람.
> **작성일**: 2026-06-09.

## 먼저: 용어 미리보기 (어려운 말 풀이)

본문에 계속 나오는 단어들이다. 여기서 한 번 풀어두고, 본문에서는 가볍게 쓴다.

| 용어 | 쉬운 뜻 |
|---|---|
| **지식그래프(knowledge graph)** | 개념·파일·함수 같은 것들을 **점(노드)** 으로 놓고, 그들 사이 관계를 **선(엣지)** 으로 이은 그림. "A가 B를 호출한다", "이 함수가 저 개념을 설명한다" 같은 걸 그래프로 표현한 것. |
| **노드 / 엣지** | 노드 = 점(무언가 하나). 엣지 = 두 노드를 잇는 관계 선. |
| **AST(추상 구문 트리)** | 소스코드를 기계가 파싱해 만든 구조. 여기서 import·함수호출·상속 같은 **사실 관계**를 정확히 뽑을 수 있다. LLM 추측이 아니라 컴파일러 수준의 확실한 정보. |
| **semantic 추출** | 문서·코드에서 "이건 무슨 개념이다", "이 둘은 같은 걸 말한다" 같은 **의미** 관계를 뽑는 것. AST로는 안 되고, LLM이 글을 읽고 판단해야 한다. |
| **provenance(출처 이력)** | 어떤 노드·엣지가 **어느 원본의 몇 번째 버전 어디서** 나왔는지 기록. "이 주장은 docs/a.md의 12번째 줄, 커밋 abc123에서 나옴" 같은 꼬리표. |
| **confidence(신뢰 등급)** | 관계가 얼마나 확실한지. Graphify·보고서 공통으로 3등급: **EXTRACTED**(원본에 그대로 적혀 있음=확실), **INFERRED**(추론함=그럴듯함), **AMBIGUOUS**(불확실=사람이 봐야 함). |
| **source of truth(원본/진실)** | "여기 적힌 게 진짜다"라고 믿는 단 하나의 기준 저장소. 나머지는 전부 여기서 다시 만들 수 있어야 한다. |
| **projection(파생물)** | 원본에서 **생성된** 부산물. 그래프, 위키 같은 것. 직접 고치는 게 아니라 원본을 고치고 다시 찍어내는 대상. (사진의 인화물 같은 것 — 음화(원본)를 고쳐야지 인화물에 덧칠하면 안 됨.) |
| **IR(중간 표현)** | 추출기와 최종 산출물 사이의 중간 데이터. "정규화된 노드·엣지 묶음". 이게 있으면 그래프 DB가 날아가도 다시 만들 수 있다. |
| **hyperedge(하이퍼엣지)** | 보통 엣지는 둘(A–B)을 잇지만, 하이퍼엣지는 **셋 이상**(A·B·C가 한 흐름에 같이 참여)을 한 번에 묶는 관계. |
| **write-back(되써넣기)** | 에이전트가 만든 답/결과를 다시 그래프에 **직접 써 넣는** 것. |
| **manifest(매니페스트)** | "이 산출물은 어느 원본 버전들로 만들었다"를 적은 작은 명세 파일. 최신인지 낡았는지 싸게 판단하게 해준다. |

> **본문의 `[SKILL:줄번호]` 표기**: Graphify가 실제로 그렇게 동작한다는 **근거**다. 로컬에 설치된 Graphify 구현 파일(`~/.claude/skills/graphify/SKILL.md`, v0.4.14, 1306줄)의 해당 줄을 가리킨다. 즉 추측이 아니라 코드를 직접 보고 적었다는 표시다.

## 1. Graphify가 대체 뭔가 (한 문단)

폴더에 아무거나 넣으면 — 코드, 문서, 논문 PDF, 스크린샷, 메모 — Graphify가 그걸 다 읽어서 **하나의 지식그래프로 만들어 주는 도구**다. "무엇이 무엇과 연결돼 있는지, 네가 몰랐던 연결"을 보여주는 게 목적이다. 형태는 Claude Code의 **스킬**(`/graphify`)이고, 속은 파이썬 패키지(`graphifyy`, MIT 라이선스)다. 만든 사람은 Safi Shamsi. 결과물은 인터랙티브 HTML 그래프, `graph.json`(나중에 다시 질의 가능), 그리고 사람이 읽는 리포트로 나온다.

기술 스택은 다음 다섯 가지로 요약된다: **tree-sitter**(코드 구조 파싱) + **LLM**(문서 의미 추출) + **NetworkX**(그래프 자료구조) + **Leiden**(군집 묶기) + **vis.js**(시각화). 서버나 DB 없이 로컬에서 돈다.

## 2. Graphify는 어떻게 도는가 (파이프라인을 순서대로)

Graphify의 처리 흐름은 7단계다. 하나씩 풀면:

1. **파일 감지(detect)** — 폴더를 훑어 코드/문서/논문/이미지/영상으로 분류한다. 너무 크면(2백만 단어·200파일 초과) 하위 폴더를 고르라고 멈춘다 [SKILL:113]. → **개인·소규모 코퍼스를 전제로 만든 도구**라는 첫 신호.

2. **추출(extract)** — 여기가 핵심이고, **두 갈래로 갈린다**:
   - **코드 → AST 추출**: tree-sitter로 함수·import·호출 관계를 뽑는다. 파이썬 라이브러리 안에서 도는 **결정적이고 공짜(토큰 0)** 작업이다 [SKILL:176-195].
   - **문서 → semantic 추출**: 여기서 Graphify는 **직접 LLM을 호출하지 않는다.** 대신 Claude Code의 **서브에이전트를 여러 개 동시에 띄워서**(한 에이전트가 파일 20~25개씩 담당) 각자 글을 읽고 정해진 JSON 형식으로 노드·엣지를 디스크에 쓰게 한다 [SKILL:197-304]. 즉 "의미 추출기"는 코드 속 함수가 아니라 **오케스트레이션 프롬프트 + 에이전트 군집**이다. (이게 왜 중요한지는 §5에서 다시 다룬다.)

3. **그래프 구축(build)** — AST 결과와 에이전트들 결과를 합쳐 NetworkX 그래프 하나로 만든다.

4. **군집화(cluster)** — **Leiden 알고리즘**으로 노드를 주제별 덩어리(커뮤니티)로 묶는다 [SKILL:405].

5. **분석(analyze)** — 연결이 유난히 많은 **허브 노드(god node)**, 서로 다른 덩어리를 잇는 **뜻밖의 연결(surprising connection)**, 던질 만한 질문을 찾아낸다 [SKILL:417-419].

6. **리포트(report)** — 위 발견들을 `GRAPH_REPORT.md`로 적는다. 토큰 비용도 함께 [SKILL:423-424].

7. **내보내기(export)** — HTML, `graph.json`, 위키, Obsidian, (선택) Neo4j용 Cypher, SVG, GraphML로 출력 [SKILL:581-660].

**한 번 만든 다음**에는 증분 갱신을 지원한다: 파일 해시(SHA256)를 캐시해 **바뀐 파일만** 다시 추출하고, 변경이 코드뿐이면 LLM 단계를 통째로 건너뛴다 [SKILL:209-231, 813-829]. 또 답을 그래프에 되써넣어(`save-result`) 다음 갱신 때 그 Q&A가 노드로 들어가는 **피드백 루프**도 있다 [SKILL:1064-1070].

## 3. 우리 메모리 보고서는 뭘 하려던 거였나 (자체 요약)

(메모리 보고서를 안 읽었어도 이 절만 보면 이후 비교를 따라갈 수 있다.)

보고서의 문제의식은 이거다: 에이전트용 메모리(위키 + 그래프 DB)를 오래 운영하면 **검색 속도가 아니라 정합성**이 깨진다. 위키와 그래프가 서로 다른 말을 하고, LLM이 추측한 관계가 사실처럼 굳고, 원본이 바뀌었는데 낡은 주장이 계속 검색된다.

그래서 보고서가 세운 핵심 규칙들:

1. **진실은 둘뿐이다 — 원본(raw source)과 승인된 메모리 이벤트.** 그래프와 위키는 진실이 아니라 거기서 **다시 만들 수 있는 파생물(projection)**이다.
2. **모든 노드·엣지에 출처(provenance)를 단다.** 어느 원본 어느 버전에서 나왔는지 없으면 메모리가 아니라 임시 메모다.
3. **확실한 관계(EXTRACTED)와 추측한 관계(INFERRED)를 섞지 않는다.**
4. **위키와 그래프는 서로를 직접 고치지 않는다.** 둘 다 공통 입력에서 단방향으로 생성만 된다.
5. **에이전트는 그래프에 직접 못 쓴다.** 항상 "제안(proposal)"을 내고, 사람/정책이 승인해야 반영된다.
6. **파생물은 직접 수정하지 않는다.** 틀리면 파생물을 손보는 게 아니라 원본·추출기·컴파일러를 고치고 다시 찍는다.

요컨대 보고서는 **"시간이 지나도, 여러 에이전트가 같이 써도 정합성이 안 깨지는 것"**을 최우선으로 둔 무거운 설계다.

## 4. 둘의 근본 차이 — 한 장으로

이게 이 문서에서 제일 중요한 대목이다.

- **Graphify의 목표**: 한 사람이 자기 자료 폴더를 **빠르게 한 번에** 그래프로 만들고, 나중에도 질의하는 것. 그래서 저장소가 **`graph.json` 파일 하나**고, 그걸 직접 고치고(`--update`), 답까지 되써넣는다. 가볍고 즉시적이다.
- **보고서의 목표**: 여러 에이전트가 **오래** 공유하는 메모리에서 정합성을 지키는 것. 그래서 원본/이벤트/IR/파생물을 **층층이 분리**하고, 출처를 달고, 쓰기는 승인을 거친다. 무겁지만 안전하다.

→ 그래서 **Graphify는 보고서가 "하지 말라"고 한 것을 설계상 그대로 한다.** 파생물을 직접 고치고, 에이전트 답을 검토 없이 그래프에 되써넣는다. 이건 Graphify의 버그가 아니라 **다른 비용을 택한 결과**다.

**결론적으로 보고서 입장에서 Graphify는 두 얼굴이다**: ① 빌려올 메커니즘이 실제로 도는 **레퍼런스**이자, ② 정합성 규칙을 빼면 무엇이 무너지는지 보여주는 **반례**.

## 5. 항목별 대조 — 보고서 아이디어 → Graphify 실제

보고서가 "Graphify에서 빌려온다"고 적은 8가지를, 실제 구현과 짝지어 본다. **"실제로는"** 칸이 핵심이다.

| 보고서가 빌려온 것 | Graphify는 실제로 어떻게 하나 | 쉽게 말하면 |
|---|---|---|
| **AST 추출** | tree-sitter로 함수·import·호출 추출, 코드 한정·공짜 [SKILL:176-195] | 일치. "import는 AST가 이미 가졌으니 LLM은 다시 뽑지 마라"고 역할까지 갈라놨다 [SKILL:265] |
| **semantic 추출** | 앱이 LLM을 부르는 게 아니라 **서브에이전트 군집**이 함 [SKILL:197-304] | 보고서는 "앱 안의 추출기"를 가정했는데, Graphify는 그 일을 **에이전트 하네스에 외주**한다 (§6-3) |
| **출처(provenance)** | 노드에 `source_file`·위치·작성자만 있음 [SKILL:303] | **반쪽.** 보고서가 정합성의 핵심으로 본 `버전/해시/추출run id`가 **없다** → 다시 만들기·낡음 판정 불가 (§7) |
| **신뢰 등급(confidence)** | 3등급 + 점수 보정 규칙 + "0.5 금지" [SKILL:294-300] | **오히려 보고서보다 빡빡하다.** 보고서가 역으로 가져갈 부분 (§6-3) |
| **증분 갱신** | 해시 캐시로 바뀐 파일만, 코드만 바뀌면 LLM 생략 [SKILL:209-231,813-829] | 보고서가 "이렇게 하면 된다"고 적은 걸 **실제로 돌려 보여줌** (§6) |
| **그래프 분석** | 허브 노드·뜻밖의 연결·질문 추천 [SKILL:417-419] | 일치. 단 보고서가 원한 "정합성 감사(고아·중복·모순 탐지)"는 **없음** |
| **구조 유사도** | LLM이 판단한 의미 유사 + Leiden 군집 [SKILL:279-283] | 보고서가 적은 **node2vec가 아니다** (§7 정정) |
| **감사 리포트** | "무엇을 발견했나" 요약 [SKILL:423-424] | "정합성이 맞나" 검사는 아님. 낡음·고아 검사는 없음 |
| **MCP 접근** | 읽기 전용 질의 도구들만 [SKILL:662-680] | **쓰기/제안 도구 없음.** 최신여부(freshness) 정보도 안 돌려줌 (§7) |

## 6. Graphify가 우리에게 가르쳐주는 것 (보고서에 더하면 좋을 것)

보고서가 안 다뤘지만 Graphify에는 있는 것들. 쉽게 풀어서.

**6-1. 하이퍼엣지 — 셋 이상을 한 번에 묶기.**
보고서의 데이터 모델은 엣지가 무조건 둘(A–B)만 잇는다. 그런데 현실에는 "이 5개 클래스가 같은 인터페이스를 구현", "이 함수들이 한 인증 흐름에 같이 참여" 같은 **셋 이상의 묶음**이 많다. Graphify는 이걸 하이퍼엣지로 표현한다 [SKILL:285-289]. 보고서 모델로는 이걸 쪼개서 잃어버린다. → **우리 모델에 하이퍼엣지를 넣을지 검토.**

**6-2. "왜(rationale)"를 뽑아내기.**
Graphify는 문서에서 "왜 이 결정을 했나"를 노드로 뽑아 대상 개념에 `rationale_for`로 잇는다 [SKILL:267]. 우리 보고서엔 "결정(Decision)" 노드는 있어도 **문서에서 이유를 자동으로 캐오는** 단계가 없다. → 추출 단계에 이유 캐오기를 추가할 만하다.

**6-3. 의미 추출을 에이전트에 맡기는 방식.**
§2·§5에서 본 것. 보고서는 "추출기를 제품 안에 둔다"고 가정했는데, Graphify는 그걸 에이전트 군집에 외주한다. **"추출을 제품 안에 둘까, 에이전트 하네스에 맡길까"는 보고서가 명시하지 않은 갈림길**이고, 우리는 이미 ditto에서 에이전트 오케스트레이션을 하므로 후자가 자연스럽다.

**6-4. "코드만 바뀌면 LLM 건너뛰기" = 검증된 패턴.**
보고서는 "코드 관계는 즉시 재추출, 의미 관계는 천천히"라고 적었는데, Graphify가 **실제로** 코드만 바뀌면 LLM을 통째로 건너뛴다 [SKILL:813-829]. → 보고서의 이 아이디어를 "이미 작동하는 것"으로 인용 가능.

**6-5. 답을 그래프에 되써넣는 피드백 루프.**
Graphify는 질의 답을 그래프에 다시 써 넣어 그래프가 스스로 자란다 [SKILL:1064-1070]. 보고서는 이걸 **금지**한다(에이전트 직접 쓰기 = 오염원). 하지만 이 루프가 주는 **가치(쌓이는 지식)** 자체는 진짜다. → 보고서는 "이 가치를, 직접 쓰기 대신 제안→승인→재생성 경로로 안전하게 재현한다"를 목표로 명시하면 더 강해진다.

**6-6. 규모 한계가 설계를 정한다.**
Graphify는 5천 노드 넘으면 시각화를 거부하고 [SKILL:543], 큰 코퍼스는 쪼개라고 한다 [SKILL:113]. 즉 **개인 규모용**이다. 우리가 노리는 "여러 에이전트 장기 메모리"에서는 Graphify의 단일 파일·매번 재생성 방식이 깨진다 — **바로 이 지점이 우리의 더 무거운 설계가 정당화되는 이유다.**

**6-7. 멀티모달 전처리.**
Graphify는 PDF 인용 채굴, 이미지 vision, 영상·오디오 Whisper 전사를 한다 [SKILL:116-159,1237-1243]. 보고서는 원본 종류를 일반화만 했지 **비텍스트 자료를 텍스트로 바꾸는 전처리 단계**를 명시하지 않았다. → 추가 검토.

## 7. 보고서에서 바로잡아야 할 사실 (실측 결과 부정확한 부분)

Graphify를 직접 보니 보고서의 서술 세 군데가 사실과 다르다.

**7-1. 유사도는 node2vec가 아니다.**
보고서는 "비슷한 구조 찾기 = node2vec"라고 여러 번 적었다. 실제 Graphify는 node2vec를 **유사도에 쓰지 않는다.** 대신 ① LLM이 판단한 의미 유사 엣지 + ② Leiden 군집을 쓴다 [SKILL:279-283,405]. node2vec는 보고서가 참고한 *다른 문서(그래프 DB 매뉴얼)* 계열의 기능이지 Graphify의 것이 아니다. → 분리해서 적어야 정확.

**7-2. Graphify는 Memgraph와 무관하다.**
보고서는 "Graphify를 Memgraph로 흉내내지 말자"는 식의 경고를 반복하는데, **Graphify는 애초에 Memgraph를 쓰지 않는다.** 저장은 NetworkX + `graph.json` 파일이고, 그래프 DB는 (선택적으로) **Neo4j** 연동이다 [SKILL:581-618]. Memgraph는 보고서가 독자적으로 고른 런타임일 뿐, Graphify와 연결고리가 없다. → "존재하지 않는 결합"을 경계하는 셈이라 프레이밍 수정 필요.

**7-3. 의미 추출은 "앱 안의 추출기"가 아니다.**
§2·§5에서 본 대로, 보고서가 그린 "애플리케이션 계층이 의미 추출기를 소유" 그림은 한 가지 구현일 뿐이다. Graphify는 에이전트 군집에 외주한다. 둘 다 가능하다는 걸 명시해야 한다.

## 8. 핵심 긴장 자세히 — 단일 가변 파일 vs 층층이 분리

보고서의 규칙 8개에 Graphify가 각각 어떻게 답하는지 정리. **"평가"** 칸만 봐도 된다.

| 보고서의 규칙 | Graphify는 실제로 | 평가 |
|---|---|---|
| 파생물 직접 수정 금지 | `graph.json`이 곧 저장소이자 산출물, `--update`로 제자리 수정 [SKILL:836-895] | ❌ **정면 위배** (원본/파생 분리 자체가 없음) |
| 모든 관계에 출처 | 파일·위치는 있으나 버전/해시/추출id 없음 [SKILL:303] | △ **반쪽** (재생성·낡음판정 불가) |
| 확실/추측 분리 | 3등급 + 점수 밴드 [SKILL:294-300] | ✅ **충족** (오히려 더 빡빡) |
| 중복 개체 정리 | id 일치 제거만, 별칭·병합검토 없음 [SKILL:341] | △ **약함** |
| 전체 재빌드 + 차이 검증 | 변경 diff는 보여줌, 정합성 parity 검사는 없음 [SKILL:870-892] | △ **반쪽** |
| 에이전트 쓰기는 제안 경유 | `save-result`가 검토 없이 즉시 되써넣음 [SKILL:1064-1070] | ❌ **정면 위배** |
| 위키↔그래프 직접 동기화 금지 | 위키·HTML은 graph.json에서 생성만(단방향) [SKILL:487-579] | ✅ **충족** |
| 질의 결과에 최신여부 동봉 | MCP는 결과만, freshness 없음 [SKILL:662-680] | ❌ **미충족** |

**한 줄 요약**: 8개 중 ✅3개 충족(확실/추측 분리는 초과 달성), ❌3개 위배, △2개 반쪽.

위배 2개(**파생물 직접 수정·에이전트 직접 되써넣기**)가 핵심이다. 이건 보고서 §10이 막으려던 바로 그 "오염 경로"의 실물이다. Graphify가 빠르고 가벼운 이유이자, 우리가 안 따라 하기로 한 이유다.

## 9. 보고서가 이미 잘한 것 (Graphify 대조로 확인됨)

비판만 하면 균형이 안 맞으니, 실측이 보고서를 **뒷받침한** 지점도 적는다.

- **"Graphify는 도구가 아니라 참고점"**이라는 판단은 옳다. Graphify는 개인 규모라 다중 에이전트 장기 메모리의 운영 의존성으로는 부적합하다(§6-6이 근거).
- **확실/추측 분리**를 우리가 스키마(데이터 규격) 차원에서 강제한 건, Graphify가 프롬프트 규율로만 한 것보다 견고하다.
- **단방향 생성**은 Graphify도 실제로 지킨다 — 우리 규칙이 비현실적이지 않다는 확인.
- **"출처 없으면 메모리 아님"**이라는 주장은, Graphify가 출처를 약하게 가져서 재생성·낡음판정을 못 하는 걸로 거꾸로 증명된다.

## 10. 한계와 버전 주의

정직하게 짚을 점들.

- **버전 어긋남.** 구현 동작은 로컬 **v0.4.14**를 줄 단위로 확인했다. 한편 PyPI 최신은 **0.8.36**(2026-06-08)으로, 지원 언어(31종)·LLM 프로바이더(Claude 외 Ollama/OpenAI/Gemini/Bedrock)·연동 에이전트 범위가 더 넓다. **다만 핵심 아키텍처(tree-sitter + LLM 추출 + NetworkX + Leiden + 3등급 confidence + 단일 graph.json)는 세 버전에 걸쳐 안정적**이라, 이 문서의 *관계·긴장 분석*은 버전과 무관하게 성립한다. 0.8.x의 내부 변화(출처 필드 강화 여부 등)는 줄 단위로 확인하지 못했으니, 거기 의존하는 결론은 재확인 필요.
- **"71.5배 토큰 절감"은 Graphify 자기보고 수치다.** 제3자 검증은 못 찾았다. 인용 시 "자기보고"로 표기할 것.
- 독립적인 정합성·보안 감사 자료는 확인하지 못했다(소개·튜토리얼 글 위주).
- 이 문서는 Graphify와 보고서를 **대조**할 뿐, 보고서의 설계 결정 자체를 재심의하지는 않았다(그건 별도 작업).
