# ADR-20260723-parity-audit-classification-principles: 옛 src 능력 분류(keep/drop/redesign) 원칙 — 기본 drop(가역)·순효능 잣대·4항 증거·flip-전-완료

- 상태: accepted
- 결정 일자: 2026-07-23
- 결정자: hskim, claude (claude-opus-4-8)
- 관련: ADR-0022 (자기호스팅·배포 생애주기 — flip의 상위 틀), ADR-20260713-dogfood-not-purpose-user-project-value (증거 인정 기준의 배경 — 자기 사용은 검증 수단이지 가치가 아님), ADR-0020 (ADR 충돌 분류 — 분류표의 ADR-충돌 판정 처리에 적용), ADR-20260624-adr-identifier-policy (식별자 정책). 촉발 WI: wi_260723czc — 이 ADR은 해당 WI 의도잠금(사용자 확정)과 계획 심의(dialectic-1 required_edits)의 **기록**이며 새 결정을 만들지 않는다. 선행 게이트: #66 순효능 감사의 keep 판정.

## 컨텍스트

zero-start 재구축(rebuild/)이 옛 src 표면(~105 core 모듈 · ~19 능력영역 · CLI 39 · skills 18 · agents 21, 2026-07-21 실측)을 대체 중이다. 전수 패리티 이식이 아니라 각 능력을 keep / drop / redesign으로 분류한 뒤 keep-only로 증분 재진입한다. 선행 순효능 감사(#66)에서 대부분 기능에 효능 측정 증거가 없음이 확인되어, 무증거 기능의 기본 처분과 keep 입증 기준을 정해야 개별 판정이 가능해졌다. 아래 원칙은 wi_260723czc 의도잠금에서 사용자가 확정했다. 개별 영역의 실제 판정은 분류표와 사용자 일괄 리뷰의 소관이며 이 ADR의 범위가 아니다.

## 결정

### D1. 기본 처분 = drop (가역)

효능 증거가 없는 기능의 기본 처분은 **drop**이다. 입증 책임은 keep 쪽에 있다. drop은 **가역**이다 — 옛 src와 git 이력이 보존되므로 복원 경로가 남는다. **각 drop 판정에는 철회 조건을 기록한다**: 어떤 사실이 확인되면 그 기능이 재진입 후보로 복귀하는지를 명시한다.

근거(비용 비대칭): keep의 비용 = flip 전 확정 재구현 의무, drop의 비용 = 미래의 잠재 복원 — 후자가 가역이라 기본값은 drop이다.

### D2. keep 잣대 = 순효능

keep 성립 조건은 효능 증거의 존재가 아니라 **순효능(효익 − 비용)**이다. **증거는 인정 요건이지 충분조건이 아니다** — 인정 증거가 있어도 유지·재진입 비용이 효익을 지배하면 keep은 불성립한다(#66 감사의 결합 판정 규칙을 per-feature로 계승).

### D3. 4항 증거 인정 기준 + 증거 상태 3분류

keep 입증에 인정되는 증거의 종류(사용자 확정, 2단 인정):

1. **결함 포착 기록** — 인정. 단 **사용자 프로젝트 일반화 논증 필수**(이 저장소 밖 임의 사용자 프로젝트에서도 같은 가치가 발현된다는 논증).
2. **'매일 쓴다'(실사용) 단독 불충분** — 결함 포착·의존 사례가 뒷받침할 때만 인정.
3. **ADR 전제 = 자동 keep 아님** — 기존 ADR이 그 기능을 전제한다는 사실은 개별 확인으로 승격시키는 신호일 뿐이다.
4. **사용자 체감 = 직접 표명만** — 에이전트가 추정한 체감은 불인정, 사용자가 직접 표명한 것만 인정.

증거 상태 3분류(dialectic-1 required_edit 축자 반영): 각 증거는 **validated**(저장소 핀으로 직접 확인됨) / **candidate**(미검증 정황) / **missing**(미발굴)으로 상태를 표기하며, **keep/redesign 근거에는 validated만 산입한다** — candidate가 검증된 근거로 세탁되는 경로를 차단한다.

### D4. #66 keep 판정의 번들 한정

순효능 감사(#66)의 keep 판정은 '옛 제품 통째 × 헌장 규율 1축'의 **번들 수준 측정**으로 부품별 기여 분해가 불가하다. 따라서 그 keep은 **로드맵 진행의 선행 게이트로만 유효**하며, **어떤 개별 능력의 keep 구제 증거로도 인용할 수 없다**. 개별 능력의 keep은 전부 D2·D3으로만 성립한다.

### D5. redesign 버킷의 flip-전-완료 의무

**redesign 판정 전부가 완료되기 전에는 flip(옛 src 은퇴)이 불가하다.** redesign은 살림(keep)과 동급의 flip-전 의무다 — 교체 직후 기능 공백을 불허하며, 교체 완료 지연을 감수한다(사용자가 필수/이연 이분 추천안을 기각하고 전부-의무를 선택). 함의: redesign 버킷이 비싸졌으므로 **애매한 기능은 redesign으로 도피할 수 없다** — drop + 철회 조건으로 처분한다.

### D6. 비사소 실증 대상 정의

flip 게이트의 실증 대상은 다음과 같이 정의한다(축자): **'비사소한 실제 개발 작업 1건을 rebuild 구동 루프로 계획→구현→검증→완료까지 완주'**. '비사소'가 게이트의 판별 조건 그 자체다 — **사소한 작업의 재완주는 게이트를 충족하지 않는다.** 기존의 유일한 실증(wi_2607201id)은 사소 작업이므로 이 게이트를 충족하지 않는다.

### D7. 부품 사슬 판별 규칙 (결정적) + 재진입 순서

**'실증 부품 사슬'** = D6의 비사소 실증 완주 경로(계획→구현→검증→완료)를 rebuild 구동 루프로 수행하는 데 **실행 경로상 필요한 능력의 집합**. 판별 기준은 결정적이다: **"그 능력이 없으면 실증 완주가 불가능한가"**. 사슬 구성원은 **확정 분류표에서 이 규칙으로 도출**한다 — 사전에 concrete target을 고정하지 않는다(keep 집합이 분류 전에는 미지이므로).

**재진입 순서** = 사슬 구성원 최우선 → 나머지 keep+redesign은 의존 관계 위상정렬. 체감 가치는 순서 기준에서 제외한다(flip 전까지 옛 src가 도구를 굴리므로 사용자 체감은 순서와 무관하게 불변).

## 변경 조건 (이 ADR의 철회 조건)

- flip(#69)이 완료되어 옛 src가 은퇴하면 이 분류 원칙은 목적을 다한 것으로 historical 처리한다.
- 대체 로드맵 자체가 중단·방향 전환되거나, #66 감사 결론(번들 keep)이 뒤집히면 이 원칙 전체를 재검토한다.
