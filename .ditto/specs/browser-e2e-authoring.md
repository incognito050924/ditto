## 1. 기능

- **이름**: 웹 서비스 실브라우저 E2E 테스트 작성 기능 (slug: `browser-e2e-authoring`, wi_260610p9h)
- **층위**: DITTO 기능 4축 중 **E2E 축**의 확장. 기존 일회성 journey 실행 계층(`ditto:e2e` 스킬 → `playwright-e2e` agent → `ditto e2e run`) 위에 "영속 테스트 작성" 계층을 추가한다.

## 2. 요약

웹 서비스를 개발하는 DITTO 사용자가 **사용자 여정(user journey)을 DSL(전용 마크업)로 작성하면**, DITTO가 그것을 실제 브라우저에서 반복 실행 가능한 **영속 Playwright 테스트 파일**(`*.spec.ts`, 필요 시 `playwright.config.ts` 포함)로 변환해 대상 저장소에 작성한다. 작성된 테스트는 DITTO 없이도 표준 `npx playwright test`로 실행 가능한 회귀 방어 자산이다.

**왜 사용자 작성 DSL인가**: 에이전트가 코드와 요청만 보고 자율 작성한 E2E 테스트는 UI/UX·설계·기획 맥락을 반영하지 못해 사실상 의미 없는 테스트가 된다. 그렇다고 사용자가 Playwright 스크립트를 직접 짜기는 어렵다. 그래서 **의도(여정)는 사람이 DSL로 선언하고, Playwright 구현은 DITTO가 담당**하는 분업이 이 기능의 핵심이다.

**주 소비자는 autopilot 검증이다**: autopilot으로 개발된 기능이 정말 의도대로 동작하는지, 기존 사용자 여정·스토리가 깨지지 않았는지를 사용자 선언 여정에서 파생된 테스트로 증명한다. 사용자가 필요할 때 직접 호출하는 것은 보조 경로다. [산출물=영속 Playwright 파일, 입력=사용자 DSL: 사용자 결정, 2026-06-11 — 대안은 §10]

## 3. 배경 [장]

> 근거: 코드 조사(ditto:researcher, 2026-06-11) + memory projection `proj_9fdb287b9115` (fresh).

**현황 — DITTO의 E2E 축은 일회성 journey 실행기다.** `ditto:e2e` 스킬이 journey 스펙(JSON: steps + 기계 판정 가능한 assertions)을 즉석에서 구성하고(`skills/e2e/SKILL.md:18`), `playwright-e2e` agent가 `ditto e2e run` CLI(`src/cli/commands/e2e.ts:32-93`)를 통해 headless Chromium을 구동해(`src/core/e2e/browser.ts:257-264`, 캡처 러너 `src/core/e2e/playwright-runner.mjs`) 스크린샷·trace·console·network를 캡처한다. 산출물은 단일 `e2eJourney` 아티팩트(`src/schemas/e2e-journey.ts`)로 `.ditto/local/runs/<run-id>/journey.json`에 남고, CompletionContract의 AC를 닫는 증거로 쓰인다(`skills/e2e/SKILL.md:25-33`).

**문제 1 — 테스트가 영속되지 않는다.** 매 실행 시 러너 설정을 즉석 생성하고(`browser.ts:243-254`), 대상 저장소에 남는 것은 gitignore된 캡처 아티팩트뿐이다. 검증이 끝나면 같은 여정을 다시 검증할 수단이 사라진다 — 회귀 방어 자산이 없다.

**문제 2 — 테스트 의도의 출처가 에이전트다.** journey 스펙을 에이전트가 코드와 요청만 보고 즉석 작성한다. 에이전트는 UI/UX·설계·기획 맥락을 사람처럼 반영하지 못하므로, 자율 작성된 테스트는 사실상 의미 없는 테스트가 된다(사용자 진술, 2026-06-11). 사람의 의도를 담는 작성 표면이 현재 없다 — 그렇다고 사용자가 Playwright 스크립트를 직접 짜기는 어렵다.

**선행물 — 절반의 재료는 이미 있다.**
- ACG `JourneySpec` 스키마(`src/schemas/acg-journey-spec.ts:18-44`): 사용자 여정의 선언적 카탈로그(steps/surfaces/fixtures/evidence_requirement). 그러나 **schema-only** — 생산·소비 코드가 없다.
- `e2eJourney.journey_id` 필드(`src/schemas/e2e-journey.ts:76-82`): JourneySpec 연결용으로 이미 예약됨.
- `src/acg/journey/journey-run-adapter.ts:21-24` 주석이 "JourneyRun authored directly"를 미래 과제로 자기 언급.
- assertion 술어 문법(`<selector> contains <text>` / `visible` / `hidden`)과 분류기가 존재.

**autopilot 통합 현황.** `e2e`는 autopilot 노드 kind이자 per-AC 판정을 반환하는 judging node다(`src/schemas/autopilot.ts:20,92`, `src/core/autopilot-graph.ts:15`). planner는 고영향 웹/여정 변경 시에만 e2e 노드를 추가한다(`agents/planner.md:18`) — 즉 현행 트리거는 **planner 재량의 자동 판단**이며, 사용자에게 묻는 단계가 없다. 본 기능은 이 트리거를 사용자 제안형으로 바꾼다(§4). 웹 프로젝트 여부는 자동판정된다(`src/core/e2e/applicability.ts:18-69`).

**Playwright 의존성 정책.** 설치 스크립트가 Chromium을 준비하고(`scripts/install-plugin.mjs:293-306`), 런타임은 절대 자동 설치하지 않으며 부재 시 schema-legal `blocked`로 강하한다(`browser.ts:129-173`).

**이전 시도: 없음.** 영속 Playwright 테스트 파일을 작성하는 기능은 코드베이스 어디에도 없다(skills/agents/src 전수 grep, 2026-06-11).

## 4. 목표

1. **사용자 작성 DSL → Playwright 변환.** 사용자가 사용자 여정을 DSL(전용 마크업)로 선언하면, DITTO가 이를 실행 가능한 영속 Playwright 테스트 파일(`*.spec.ts`)로 변환해 대상 저장소에 작성한다. 작성된 테스트는 DITTO 없이 표준 `npx playwright test`로 실행된다.

2. **대화형 시나리오 작성.** DSL 작성은 사용자 단독 작업이 아니라 에이전트와의 대화형 협업이다 — 의도(여정·기대 동작·기획 맥락)는 사용자가 내고, 코드 사실(셀렉터, 라우트, 페이지 구조)은 에이전트가 채운다. 한 work item에서 테스트는 여러 개일 수 있다.

3. **제안형 트리거 (자동 추가 아님).** 변경에 웹 표면 변화(프론트엔드 페이지/컴포넌트 또는 백엔드 API)가 포함되면, autopilot이 사용자에게 E2E 브라우저 테스트 작성을 **제안**한다. 사용자가 거절하면 일반 DoD·TDD 검증으로 진행하고, 수락하면 E2E 시나리오 작성 노드를 활성화한다. 현행 "planner 재량 자동 추가"를 이 사용자 확인 기반 모델로 대체한다.

4. **기존 테스트 수명주기 관리.** 코드 변경으로 기존 E2E 테스트가 무의미해지거나 깨지는 경우를 다룬다: 의미 없어진 테스트의 삭제, 변경에 맞춘 기존 테스트의 갱신을 작성 흐름의 일부로 지원한다(삭제·갱신은 사람이 선언한 의도의 산물이므로 사용자 확인을 거친다).

5. **회귀 게이트 — 영향받는 기존 테스트의 통과를 AC로 강제.** 기존 E2E 테스트 전체가 아니라 **이번 변경의 영향권에 있는 부분집합을 추려서** 검증 노드에서 실행하고, 그 통과를 AC에 포함시킨다(전체 실행은 느리고 flaky·외부 데이터 의존 실패가 섞여 비현실적 — 사용자 결정, 2026-06-11). 추림 기준: DSL이 여정마다 대상 표면(페이지/라우트/API)을 선언하고, 이번 변경의 diff와 교차해 영향 여정을 산출하며, 목록은 사용자가 조정할 수 있다. **추려진 목록 안에서는 회피 불가** — 에이전트가 실패를 "이번 수정 범위가 아니다"라며 넘기는 행위를 구조적으로 차단하고, 실패의 처리(수정/갱신/삭제/flaky 판정)는 사용자 확인 없이 닫히지 않는다.

6. **재사용 블록과 데이터 케이스.** DSL은 두 가지 1급 구성요소를 지원한다. (a) **재사용 블록**: 공통 시퀀스(예: 로그인)를 한 번 정의해 여러 여정에서 참조 — 반복 작성을 없애고, 공통 절차가 바뀌면 블록 한 곳만 고친다(변환 시 공유 fixture/helper로 대응). (b) **데이터 케이스**: 한 여정을 여러 데이터로 반복 검증하는 케이스 테이블 — 케이스마다 초기 변수(테스트 중 사용할 값)와 기대 결과를 선언해 성공·실패·경계값 케이스를 같은 여정 정의에서 다룬다(변환 시 케이스별 파라미터라이즈드 테스트로 대응).

7. **스크립트 결함 회복 경로 — 과적합 차단.** 스크립트가 잘못되었을 때 의도대로 개발된 기능이 "테스트를 통과시키기 위한 불필요한 수정"으로 변질되는 것을 막는다. E2E 실패 시 에이전트는 기능 코드를 수정하기 전에 실패 원인을 분류(기능 결함/스크립트 결함/환경·데이터/flaky)하고 사용자 판정을 받아야 한다. 사용자가 Playwright 코드를 몰라도 판정할 수 있도록, 실패는 DSL의 step 어휘로 보고하고(목표 1의 대응 표기 재사용), 실패 장면을 직접 볼 수 있는 수단(headed 모드 재생 + trace 뷰어)을 제공한다. 스크립트 결함으로 판정되면 스크립트 쪽이 회복(수정·재생성)된다.

## 5. 비목표 (변경 경계) [장]

1. **에이전트 단독 자율 테스트 작성은 하지 않는다.** 사용자 DSL 없이 에이전트가 알아서 테스트를 창작해 커밋하는 경로는 만들지 않는다 — 이 기능의 존재 이유가 그 경로의 부정이다(§2).
2. **테스트 실행기를 재발명하지 않는다.** 영속 테스트의 실행은 표준 Playwright(`npx playwright test`)가 담당한다. DITTO 전용 실행 엔진·리포터를 만들지 않는다.
3. **기존 일회성 journey 실행 표면은 건드리지 않는다.** `ditto:e2e` 스킬·`playwright-e2e` agent·`ditto e2e run`·`e2eJourney` 스키마는 현행대로 공존한다. 이번 변경은 그 위의 새 계층이지 대체가 아니다.
4. **사람이 손으로 작성한 기존 테스트는 관리 대상이 아니다.** 수명주기 관리(삭제·갱신)와 회귀 추림은 DSL에서 파생된 테스트에 한정한다. 사용자가 직접 짠 `*.spec.ts`를 DITTO가 수정·삭제하지 않는다.
5. **브라우저·Playwright를 런타임에 자동 설치하지 않는다.** 기존 정책 유지(설치는 설치 스크립트의 몫, 런타임 부재 시 blocked 강하 — §3).
6. **대상 웹 서비스의 수명주기를 관리하지 않는다.** dev-server 기동·배포·테스트 데이터 시딩은 대상 프로젝트의 책임이다. DITTO는 Playwright `webServer` 설정을 생성해 줄 수는 있어도 서버를 직접 띄우고 내리지 않는다.
7. **Playwright 외 프레임워크(Cypress, Selenium 등)는 지원하지 않는다.**
8. **CI 통합은 범위 밖이다.** 생성된 테스트를 CI 파이프라인에 배선하는 것은 사용자의 몫이다.
9. **flaky 자동 감지·재시도 엔진은 만들지 않는다.** flaky 판정은 사용자 확인 경유의 수동 판정이다(§4-5). 자동화는 후속 과제.
10. **DSL을 범용 프로그래밍 언어로 문법화하지 않는다.** 변수 대입·임의 표현식·명령형 제어 흐름(if/else·for 문법) 같은 범용 언어 능력을 추가하지 않는다. 단, **개념으로서의 조건부 흐름은 배제하지 않는다** — 데이터 패턴에 따라 다른 필드를 검사하거나 이후 단계를 건너뛰는 흐름은 선언적 구문(단계에 조건 부착, 데이터 케이스별 시나리오 분리·예시 테이블 등)으로 표현한다. 경계의 기준: "이 구문을 평가하려면 인터프리터가 필요한가?"가 아니라 시나리오 선언으로 읽히는가다. 재사용 블록(§4-6)과 선언적 조건이 추상화의 상한이다. **재논의 조건**: 실사용에서 선언적 구문으로 표현 못 하는 여정이 반복 확인되면(§7의 표현력 한계 측정) 이 경계는 재논의한다 — 이번 빌드의 경계이지 영구 금지가 아니다.
11. **시각적 회귀(스크린샷 픽셀 diff) 테스트는 범위 밖이다.**
12. **결정론적 DSL→Playwright 변환기를 만들지 않는다.** 파서·코드 생성기 같은 컴파일러를 개발하지 않는다 — 변환은 **에이전트가 DSL을 읽고 스크립트를 작성**하는 방식으로 한다(변환기 개발 부담이 과대 — 사용자 결정, 2026-06-11). 에이전트 변환의 정합성은 게이트로 보장한다(§7 위험 참조).

| ac-7 | 변경 diff와 여정의 표면 선언을 교차해 영향받는 기존 테스트 목록이 산출되고, 목록 내 테스트의 실패는 검증 노드의 AC 실패로 이어진다 — "이번 수정 범위 아님"으로 닫을 수 없다 | test |
| ac-8 | DSL 파생 테스트와 사람이 작성한 테스트가 기계적으로 식별되며, 갱신·삭제는 DSL 파생 테스트에 한해 사용자 확인을 거쳐 수행된다 | test |
| ac-9 | Playwright 또는 Chromium이 설치되지 않은 환경에서 변환·검증 실행을 시도하면, 설치 명령을 호출하지 않고 결과를 `blocked`로 반환한다(기존 정책 유지) | test |
| ac-10 | DSL 작성 가이드 문서가 존재하고, 재사용 블록·데이터 케이스 테이블·선언적 조건(케이스 분리/단계 조건)의 문법과 예시를 포함한다 | doc |
| ac-11 | E2E 실패가 DSL의 step 어휘로 보고되고("어느 여정의 어느 단계에서 무엇을 기대했는데 무엇이 나왔다"), 사용자가 실패 장면을 관찰할 수 있는 재생 수단(headed 모드 재실행 명령 + trace 뷰어 안내)이 제공된다 | browser |
| ac-12 | E2E 실패를 이유로 한 기능 코드 수정은 실패 원인 분류(기능 결함/스크립트 결함/환경·데이터/flaky)와 사용자 판정 전에는 진행되지 않는다 — 판정 없이 기능 코드를 고치는 경로가 존재하지 않는다 | test |

## 8. 계획 (Plan) [단]

> ⚠ **비구속(non-binding) 설계 힌트.** 구현 시 참고만 하며 더 나은 설계로 대체할 수 있다. 권위 있는 계약이 아니다. 데이터·API는 모양을 암시하는 수준까지만 — 정밀 스키마는 설계 단계의 산출물이다.

- **DSL 형태 [DECIDED 추천]**: Markdown 기반 전용 마크업. front-matter + 본문(제한 어휘의 단계 목록 + 기대 결과). 기계가 파싱하는 건 front-matter(추림·digest용)까지고, 본문은 에이전트가 읽는다 — §5-12(변환기 비개발)와 정합. ACG `JourneySpec`의 surfaces 어휘를 차용하되 스키마 의존은 두지 않는다. 최종 형태는 §7-①에 따라 설계 단계에서 사용자 확인.
- **데이터 케이스 [DECIDED]**: 여정 정의 안에 케이스 테이블을 선언한다 — 각 케이스 = 케이스명 + 초기 변수 맵(입력 값·시드 데이터 참조) + 기대 결과. 성공/실패/경계값을 한 여정에서 다루고, 변환 시 케이스별 파라미터라이즈드 테스트(`for (const c of cases)` 또는 `test.describe` 분기)로 생성한다. 실패 보고·판정(ac-11)도 케이스 단위로 식별한다.
- **front-matter 필드 [DECIDED]**: 회귀 추림(§4-5)에서 사용자와 에이전트가 "어떤 테스트인지"를 보고 고를 수 있도록 식별 필드를 둔다 — **id**(안정 식별자, `e2eJourney.journey_id`와 연결 가능), **이름**(사람이 보고 식별하는 테스트명), **설명**(테스트의 목적·가치 — 이 여정이 왜 깨지면 안 되는가), **표면 선언**(페이지/라우트/API — 추림 교차 키), **재사용 블록 참조**. 추림 목록은 id가 아니라 이름·설명으로 제시한다.
- **파일 위치 [DECIDED]**: DSL과 생성 테스트 모두 대상 저장소에 git-tracked. 예: `e2e/journeys/*.journey.md`(DSL), `e2e/generated/*.spec.ts`(생성물). 생성물 헤더 주석에 `@ditto-generated` 마커 + 원본 DSL 경로 + digest — ac-4(미재생성 감지)·ac-8(파생/수동 식별)을 한 메커니즘으로 해결.
- **표면 [DECIDED — 작성/변환 분리]**: 두 개의 실행 주체를 만든다.
  - **DSL 작성 스킬**(가칭 `ditto:e2e-author`): 사용자와의 대화형 DSL 작성을 주도 — 의도는 사용자가, 코드 사실(셀렉터·라우트)은 에이전트가 조사해 채움. 사용자 대화가 본질이므로 main session 스킬.
  - **변환 subagent**(가칭 `ditto:e2e-scripter`): DSL을 입력받아 Playwright 스크립트(+공유 fixture)를 작성하는 구현 전담 subagent. 컨텍스트 격리 가능한 작업이므로 위임 — 입력은 DSL + 대상 코드 사실, 출력은 spec.ts + step 대응 표기. 정합성 검증(ac-3)은 작성자가 아닌 fresh-context 검증(기존 reviewer/verifier 패턴)으로.
  - 사용자-facing 표면은 작성 스킬 + DSL 가이드 문서로 캡 유지(변환 subagent는 내부 표면).
- **제안 트리거**: 기존 applicability 판정(`src/core/e2e/applicability.ts`)을 diff 기반으로 확장 — 변경 파일에 웹 표면(페이지/컴포넌트/API 라우트)이 포함되면 제안. API 감지 구체 로직은 **미정**(§7-②).
- **회귀 추림**: DSL front-matter의 표면 선언 × 변경 diff 경로 교차 → 영향 여정 목록 → verify 노드가 해당 spec 파일만 `npx playwright test <files>`로 실행. 목록은 실행 전 사용자에게 제시(조정 가능).
- **실패 보고·재생**: Playwright JSON reporter 출력을 step 대응 표기로 매핑해 DSL 어휘로 보고(ac-11). 재생 수단은 명령 제공으로 충분: `npx playwright test <file> --headed` + `npx playwright show-trace <trace>`.
- **실패 판정·회복 흐름 [DECIDED 추천]** (verify 노드에서 테스트 실패 시):
  1. **정지**: 실패 즉시 기능 코드 수정 경로를 잠근다(ac-12). 추림 목록의 나머지 테스트는 끝까지 실행해 일괄 보고한다(실패 1건마다 사용자를 부르지 않음). flaky 확인용 재실행은 테스트당 최대 1회.
  2. **에이전트 1차 분류(추정)**: trace·reporter 출력·step 대응 표기로 실패 지점을 DSL step에 매핑하고, 4분류(기능 결함/스크립트 결함/환경·데이터/flaky) 중 **추정 + 근거**를 만든다. 에이전트 단독 확정 금지.
  3. **사용자 판정**: DSL 어휘 실패 보고 + 추정 분류 + 재생 명령을 제시하고 사용자가 분류를 확정한다 — 이 시점이 사용자에게 알리는 시점이다.
  4. **분류별 처리**: 기능 결함 → implement 노드로 회귀(이때만 기능 코드 수정 허용). **스크립트 결함 → 회복 절차**: e2e-scripter가 DSL 기준으로 스크립트를 수정·재생성하고 ac-2(1회 실행)·ac-3(대응 검사) 게이트를 재통과한 뒤 재실행; DSL 자체가 낡았으면 e2e-author로 사용자와 DSL을 갱신한 후 재생성. 환경·데이터 → blocked로 보고하고 해당 AC는 미검증으로 남긴다(성공처럼 처리 금지). flaky → 판정 기록 후 이번 실행에 한해 예외 처리, front-matter에 flaky 이력 표시(후속 추적).
- **시나리오 작성 노드**: 대화형(사용자↔에이전트)이므로 subagent로 위임 불가 — main session에서 진행하는 user-interaction 노드. autopilot 루프 안에서의 정확한 처리 방식(노드 kind·중단/재개)은 **미정** — 설계 단계 산출물.
- **스키마**: DSL manifest·추림 결과·판정 기록이 스키마가 필요하면 zod로 `src/schemas/`에 (ADR-0002).

## 10. 기각된 대안 [장]

1. **에이전트 자율 테스트 작성** — 기각. 에이전트는 UI/UX·설계·기획 맥락을 반영하지 못해 코드와 요청만으로 짠 테스트는 사실상 무의미하다(사용자, 2026-06-11). 이 기각이 기능 전체의 출발점이다(§2).
2. **DITTO 관리 JourneySpec 카탈로그 단독(테스트 코드 비영속)** — 기각. 선언 스펙만 .ditto에 두고 실행은 DITTO 러너가 하는 방안은, 산출물이 DITTO 없이 실행 불가하고 표준 Playwright 생태계(reporter·trace·CI)와 단절된다. 산출물 형태 결정에서 영속 파일 선택(사용자, 2026-06-11).
3. **하이브리드(카탈로그 SoT → 테스트 파일 컴파일)** — 기각. 두 계층 유지 + 스펙↔코드 drift 관리 비용이 과대. 단, 그 동기(선언 계층의 거버넌스)는 DSL front-matter(id·이름·설명·표면 선언)가 경량으로 흡수했다(§8).
4. **결정론적 DSL→Playwright 변환기 개발** — 기각. 파서·코드젠 개발·유지 부담 과대(사용자, 2026-06-11). 에이전트 변환 + 게이트 3종(실행 검증·대응 추적·digest)으로 대체(§5-12, ac-2·3·4).
5. **기존 E2E 테스트 전체 실행 강제** — 기각. 느리고, flaky·외부 데이터 의존 실패가 섞여 "빨간불 무시"를 학습시킨다(사용자, 2026-06-11). 영향 부분집합 추림 + 목록 내 회피 불가로 대체(§4-5).
6. **현행 planner 재량 자동 e2e 노드 추가 유지** — 기각. 사용자 모르게 추가/생략되어 의도 주입 시점이 없다. 제안형 트리거로 대체(§4-3).

## 11. 마일스톤 [단]

순서 힌트(추정 미정):

1. **M1 — DSL 정의**: 문법(front-matter·단계 어휘·재사용 블록·데이터 케이스) + 작성 가이드(ac-10) + DSL 형태 사용자 확인(§7-①).
2. **M2 — 작성·변환 파이프라인**: e2e-author 스킬(대화형) + e2e-scripter subagent + 정합성 게이트(ac-1~5b).
3. **M3 — autopilot 통합**: diff 기반 웹 변경 감지 + 제안형 트리거 + 시나리오 작성 노드 + 실패 판정·회복 흐름(ac-6, ac-11, ac-12).
4. **M4 — 회귀 게이트·수명주기**: 표면×diff 추림 + 목록 실행 + 갱신·삭제 흐름(ac-7, ac-8).
