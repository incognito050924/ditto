종합

큰 방향은 좋습니다. 기존 work-item, completion, evidenceRef 재사용과 신규 sidecar 분리는 과하지 않습니다. 다만 main↔subagent 실제 경계인 delegation/
result, approval-gated mutation 판정, user-owned decision 대기 상태가 아직 schema로 닫히지 않았습니다. 이 셋은 v0 M2 runtime을 좌우하므로 M0.2에 포함시키
는 게 맞습니다.

검증은 문서와 현재 Zod 스키마를 read-only로 대조했습니다. 테스트는 실행하지 않았습니다.

› schema 를 기준으로 보는 것은 계약이 잘 되었는지가 아니라 계약서에 문제가 없는지 보는거잖아. 계약(interface) 설계가 잘 되었는지 누락은 없는지 보려는거야.
ditto 를 사용하는 동안 가능한 모든 동작은 엄밀한 interface 에 의해 동작하길 원해. 마치 프로그래밍에 OOP 의 SOLID 원칙처럼 말이야


• 맞습니다. 이전 답변은 “문서가 현재 Zod 스키마와 맞는가”에 치우쳤고, 당신이 묻는 핵심은 “DITTO의 모든 동작 경계가 엄밀한 interface로 닫혀 있는가”입니다.

결론부터 말하면, 현재 계획서는 artifact schema 중심이고, runtime interaction interface 중심 설계는 아직 부족합니다. 즉 intent.json, autopilot.json,
completion.json 같은 저장 산출물은 꽤 잡혀 있지만, “누가 누구에게 무엇을 넘기고, 어떤 결과를 받아, 어떤 상태 전이를 할 수 있는가”는 일부가 자연어 계약에
남아 있습니다.

핵심 결함

1. DITTO 전체 동작을 지배하는 공통 envelope가 없습니다.
   모든 hook, skill, main loop, subagent, gate, store mutation이 공통적으로 따르는 DittoEvent, DittoCommand, DittoDecision, DittoResult 같은 상위
   interface가 없습니다.
   지금은 sidecar별 JSON은 있지만, “행동 단위”의 interface가 없습니다.
2. main ↔ subagent 경계가 가장 약합니다.
   계획서에는 6-section packet과 output contract가 있지만, 엄밀한 interface로 승격되어 있지 않습니다.
   최소한 다음은 독립 계약이어야 합니다:
    - DelegationPacket
    - DelegationResult
    - AgentCapability
    - FileScope
    - ToolScope
    - DoneWhen
    - EvidenceClaim
    - SubagentFailure
3. gate 결과 interface가 공통화되어 있지 않습니다.
   completionGate, convergenceGate, approvalGate, questionGate, readinessGate, highRiskAssumption이 모두 다른 모양으로 결과를 낼 가능성이 큽니다.
   공통으로:

   GateResult =
   | { decision: "pass"; reasons: Reason[]; evidence_refs: EvidenceRef[] }
   | { decision: "block"; blockers: Blocker[]; required_action: RequiredAction }
   | { decision: "defer"; reason: Reason; owner: "user" | "agent" | "external" }

   같은 표준이 필요합니다.

4. state mutation interface가 부족합니다.
   AutopilotStore는 언급되지만, 어떤 mutation이 허용되는지 명령형 interface가 없습니다.
   직접 status를 바꾸는 대신:
    - StartNode
    - CompleteNode
    - FailNode
    - BlockNode
    - RecordDecision
    - RequestApproval
    - ResumeFromHandoff

   같은 command interface가 있어야 합니다. 그래야 상태 전이가 닫힙니다.
5. approval과 question이 runtime blocking primitive로 설계되지 않았습니다.
   현재는 approval_gate.status와 QuestionGate.decision이 있지만, 실제 DITTO 동작에서 “사용자만 답할 수 있어 멈춤”을 표현하는 공통 interface가 없습니다.
   필요합니다:
    - UserDecisionRequest
    - ApprovalRequest
    - UserDecisionResolution
    - BlockedByExternal
    - SafetyBoundaryHit
6. evidence interface가 pointer 중심이고 claim 중심이 아닙니다.
   evidenceRef는 “어디에 근거가 있는가”이고, DITTO가 필요한 것은 “이 근거가 무엇을 입증하는가”입니다.
   별도로:

   EvidenceClaim {
   ref: EvidenceRef
   proves: "acceptance" | "failure" | "risk" | "decision" | "verification"
   criterion_id?: string
   freshness: "fresh" | "stale"
   result: "pass" | "fail" | "inconclusive" | "not_run"
   }

   이 있어야 completion/gate/subagent가 같은 언어를 씁니다.

7. enum은 닫힌 것과 열린 것을 구분해야 합니다.
   지금 enum들이 대부분 단순 union처럼 보입니다. 하지만 interface 설계상 enum은 두 종류입니다:
    - closed enum: 상태 전이 안전에 필요. 예: workItemStatus, nodeStatus, approvalStatus, gateDecision
    - open/extensible enum: provider, agent role, node kind, evidence kind처럼 확장 가능해야 하는 영역

   예를 들어 node.kind는 v0에서 닫으면 확장성이 떨어지고, 열면 owner mapping이 깨집니다. 해결은 kind를 열기보다 capability 기반 dispatch를 두는 것입니다.

SOLID식으로 보면

- SRP: 현재 sidecar는 비교적 좋지만, autopilot이 graph/state/policy/approval/continuation을 너무 많이 품습니다. 분리 필요.
- OCP: node.kind → owner 고정 매핑은 확장에 약합니다. kind보다 required_capabilities 기반이 낫습니다.
- LSP: 모든 subagent가 같은 DelegationResult를 반환해야 대체 가능합니다. 현재는 역할별 자연어 계약이라 약합니다.
- ISP: subagent packet은 좋은 방향입니다. 다만 실제 schema가 없어 강제되지 않습니다.
- DIP: orchestrator가 특정 agent 이름에 의존하지 말고 AgentCapabilityRegistry 같은 interface에 의존해야 합니다.

정리하면, 현재 계약서는 “저장되는 문서의 스키마”는 꽤 의식하고 있지만, 당신이 원하는 수준의 “DITTO의 모든 동작이 interface로만 연결되는 구조”에는 아직 못
미칩니다. 다음 수정의 핵심은 artifact schema 추가가 아니라 interaction interface layer를 M0에 first-class로 올리는 것입니다.


› 그런데 이것은 분명히 트레이드오프가 있을거야. 어느 수준이 적절한지 판단이 필요해


• 맞습니다. “모든 동작을 interface로”를 문자 그대로 밀면 DITTO 자체가 작은 타입 시스템/워크플로 엔진이 됩니다. 그건 v0에는 과합니다.

적절한 기준은 이겁니다.

엄밀한 interface가 필요한 곳

1. context boundary를 넘는 것
   main → subagent, subagent → main, hook → core, skill → store는 반드시 schema가 있어야 합니다. 여기서 자연어만 쓰면 drift가 바로 납니다.
2. state를 바꾸는 것
   work-item, autopilot, completion, convergence를 직접 수정하는 행위는 command/result interface가 있어야 합니다.
   예: StartNode, CompleteNode, RecordBlocker, RequestApproval.
3. continue/block/done을 결정하는 것
   gate 결과는 공통 GateResult가 있어야 합니다. 완료, 승인, 질문, 수렴, readiness가 각자 다른 모양이면 Stop hook과 orchestrator가 금방 복잡해집니다.
4. 나중에 재생·검증·handoff해야 하는 것
   evidence, decision, blocker, user-owned decision은 사람이 읽는 문서만으로 부족합니다. 기계 판독 가능한 claim/record가 필요합니다.
5. 권한·위험·범위에 관한 것
   file scope, tool scope, approval scope, risk flags는 엄격해야 합니다. 실수 비용이 큽니다.

엄밀한 interface가 과한 곳

1. LLM 판단 과정 자체
   “왜 이 노드를 골랐는가”의 사고 과정까지 schema화하면 비용만 큽니다. 결과와 근거만 구조화하면 됩니다.
2. skill 내부 절차
   skill 본문은 prompt/procedure로 두는 게 맞습니다. 단, skill이 내놓는 산출물은 schema로 닫습니다.
3. 작은 사용자-facing 문서
   autopilot.md, handoff.md, review.md 같은 human view는 schema의 projection이면 충분합니다.
4. 미래 확장 surface 전체
   post-v0 기능까지 v0 schema로 다 닫으면 enum과 store가 조기에 굳습니다. v0가 실제 실행하는 경계만 닫고, 나머지는 reserved/extension point로 둡니다.
