# D.I.T.T.O 의 목적

## 목적
- coding agent harness 로 범용 개발 작업에 도움을 주는 하네스
- 중점 가치
  - 사용자의 인지 비용을 줄이는 것을 목표로 한다.
  - 할루시네이션을 방지한다. (모든 출력과 추론에는 논리적일뿐만 아니라 확실한 근거가 있어야 한다. 그런 것이 아니라면 모른다고 혹은 근거 부족이라고 사실대로 답해야만 한다.)
  - 사용자의 의도와 벗어나 LLM 이 멋대로 추론 및 작업을 하는 것을 구조적으로 제한한다.
  - LLM 의 자기 확신을 깨고 자신의 잘못된 기존 가설을 깨고 올바른 방향으로 나아가고, 종종 창의성을 발휘하여 혁신적인 아이디어를 도출할 수 있도록 한다.
  - LLM 의 고질적인 문제인 Context Rot을 해결한다.
  - 장기간 실행되는 작업에도 아무런 문제없이 처음 의도한 바, 목적대로 끈질기게 완수해낸다.
  - Token 비용을 낭비하지 않는다.

## 핵심기능
- 모든 액션에는 감사 기록이 누적된다. (기본적으로 세션 단위, 새 세션에서 이어서 작업할 때를 위한 핸드오프 지원)
- 주요 결정 및 변경사항 영속화 (아래 지식베이스 관리 섹션 참고)
- Context Rot 이슈 해결을 위해 서브 에이전트를 적극 사용한다.
- 사용자 인지 비용 최소화
  - Agents 와 사용자 간의 입출력은 Matt Pocock 이 주장한 Ubiquitous Language (상호 합의된 용어 및 언어) 를 사용해야함.
  - 사용자에게 질문 또는 의견 수집을 할 때에는 사용자가 답변 도출 혹은 판단을 위한 충분한 컨택스트를 반드시 같이 제공해야 함. 사용자는 이전의 대화에서 참고하거나 소스코드를 직접 보지 않고도 현재 출력만으로 답변 가능해야함.(다만, 너무 관련 지식이나 컨택스트가 방대한 경우에는 어떤 파일 또는 URL 을 참고하라고 해야함)
  - 정제된 출력 메시지: 사용자와 협의되지 않은 용어 그대로 사용 및 약어 사용, 핵심 내용 이외에 불필요한 데이터 및 표/리스트/수치/등, 과대 표현, 추측성 결과, 잘못된 내용 또는 거짓 정보, 어떤 내용의 단락 번호/ID/제목/위치만 언급하여 컨택스트 파악 저해 등이 포함되어 있는지 확인 (자가 점검 가능하다면 출력전 절차 추가) 
  - 사용자의 욕구/의도 파악, 모호성 해결(clarify) 등 이외로 불필요하게 사용자에 질문하지 않는다.
    - 무의미한 질문 범주
      - 자명하거나 이미 최선의 선택지가 있는데 사용자에게 선택을 요청한다. (사용자에게 책임 전가하는 질문 또는 요청)
      - 무엇을 도와드릴까요?, 안부 인사, 등의 질문의 의도와 결과가 아무런 이득이 되지 않는 질문 
    - 사전 답변 가능성 타진: 사용자 질문 또는 요청 전에 반드시 스스로 질문에 대해 답변이 가능한지 확인한다. (단, 코드베이스, 산출물, 기타 접근가능한 모든 파일 및 정보 또는 웹 검색 활용하여 명확한 근거가 있어야 함. 추측이나 가정만으로 넘겨 짚는 행위 절대로 하지 말 것)
    - 본 목적의 범위에서 벗어난 자동 꼬리물기 질문 및 강요 금지: 다음 추천 작업 N 건이 있습니다. 진행하시겠습니까?
- 사용자 인터뷰: 사용자의 의도를 명확히하여 이해 상태를 동기화하고, 사용자가 놓치거나 알지 못했던 부분들을 밝히는 (문자 그대로 사용자가 언급한 지점 외에도 모든 면에 대해 고려해야함) 질문을 던진다(필요한 경우에만 사용자 의견 수집)
- 오케스트레이션: 문제 정의 / 현황 파악 ~ 개발 완료(증분 적용 완료) + 자동 회고와 행동 교정까지 체계적인 흐름으로 진행해야 함.
  - oh-my-openagent / oh-my-opencode-slim / oh-my-claudecode / oh-my-codex 의 핵심 오케스트레이션 참고
  - 각 단계의 진입과 출력은 반드시 정규화된 interface 또는 문서 양식이 있어야 하며, 이 계약을 통해 단계 간의 상태 전이, 연계 및 참조가 가능하다. (각 단계의 산출물은 다른 단계에서는 절대 수정 불가)
  - 문제가 해결될 때까지(목표를 달성할 때까지) 멋대로 중단하지 않고 끝까지 끈질기게 완수해낸다.
- 멀티 모델 정반합 기반의 적대적 검토 및 발제 도구: 가급적 멀티 모델 (claude: opus/sonnet, chatgpt: gpt-5.5/gpt-5.4)
- E2E 테스트 도구: 단순 TDD 를 위한 테스트 코드 작성이 아니라 웹 서비스에서 사용자 시나리오 및 사용자 스토리, journey, 등을 E2E 로 테스트할 수 있는 브라우저 자동화 도구 필요 (MCP 말고)
- 개발은 Matt Pocock 의 Deep Module 사고 — 좁은 interface 위에 깊은 구현, shallow 추상화 회피 - 로 구현하여 Agent Coding 에 보다 적합한 패턴으로 개발한다. (명확한 인터페이스 기반으로한 개발 패턴: 해당 계층은 사람과 coding agents 가 공통으로 관리, 실제 구현은 주로 coding agent 가 담당하며, 중요하거나 필요한 경우 사람이 개입하여 리뷰 또는 직접 수정 및 수정 지시)

## 추가 고려사항
- git 을 사용한 개발 프로세스 (best practise 적용)
  - 특정 제품은 여러 개의 git 저장소로 구성되어 있을 수 있음: 여러 저장소를 워크스페이스를 두고 하위에 clone 하여 작업하도록 권장, 컨택스트 파악, 하네스 적용에 용이
  - git worktree 를 사용한 개발 프로세스 적극 지원
- 대부분의 소프트웨어 개발은 혼자서 하지 않는다 (팀으로 진행)
- 팀업을 위한 방법론 (scrum)과 도구들을 지원한다.
  - 백로그 관리: github issues & github project
  - 문서 및 산출물 관리: confluence & 지식 베이스 관리 저장소
- 사용자 세션 관리
  - 도중에 새 세션 혹은 또 다른 PC 및 기기에서 이어서 작업 가능
