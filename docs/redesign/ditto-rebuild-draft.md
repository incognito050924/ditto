# DITTO 재구축 정초 문서 (초안 v0)

> **문서의 지위 (먼저 읽을 것)**
> - **무엇**: DITTO를 핵심 가치부터 다시 짓기 위한 *첫* 정초 문서. "초안"은 MVP를 뜻하지 않는다 — 재구축의 큰 그림 전부를 담는 충실한 설계 문서다.
> - **소비자**: 재구축을 구동할 main agent·subagent, 그리고 저자(사용자) 본인. 정초 의도를 잠그기 전의 *형성 중 의도*를 담는다.
> - **권위 아님 (헌장 §4-11)**: 이 문서는 drift한다. 여기 담긴 결정이 확정되면 `.ditto/knowledge/adr/`의 ADR과 잠긴 intent가 이 문서를 대체하며, 그 시점에 이 문서는 배경 자료로 강등된다. 사실이 흔들리면 코드·테스트·스키마·ADR을 연다.
> - **인용 규칙**: 사실은 원문 내용을 직접 담고, 출처는 코드·ADR·연구 근거(URL)를 가리킨다. 백서([DITTO-whitepaper.md](./DITTO-whitepaper.md))·기능매핑([feature-mapping.md](./feature-mapping.md))은 곁 문서다.
> - **미확정 표시**: `[연구 보강 예정]`·`[열린 질문]`·`[미검증]`은 아직 사실로 닫히지 않은 항목이다.

## 1. 한 페이지 요약 (executive summary)

DITTO는 코딩 에이전트를 "실제 프로젝트에서 신뢰할 수 있는 작업 수행자"로 만드는 오케스트레이션 규율이다. 이 재구축은 그 **철학을 보존하되 실현 방식을 통째로 교체**한다.

재구축의 방아쇠는 기능 부족이 아니라 **효능의 하락**이다. 현행 DITTO는 거의 전적으로 자기 자신을 개발하는 데(dogfooding) 쓰였는데, 그 과정에서 미검증 기대가 무절제하게 축적됐고, 그 축적이 셋을 낳았다 — autopilot 한 판 3시간+, tool-call(antml) 문법 오류, 손 JSON 파싱 오류. 더 근본적으로, 검증을 자기 자신이 수행한 탓에 "증거"가 부분적으로 자기확인이 되었다. 결과는 정교하지만 *쓰는 게 안 쓰는 것보다 낫다는 느낌(효능)이 음수로 간* 도구다.

그래서 재구축의 명제는 **뺄셈**이다:

1. **보존**: 백서의 철학 12원칙·핵심 개념·아키텍처 12불변식. 이것이 DITTO의 정체성이다.
2. **zero-start**: 오염된 축적 집합을 신뢰해 걸러내는(prune) 대신, §7 불변식만 지고 백지에서 재시작한다. 각 기능은 "DITTO 위에서 터졌나"가 아니라 **"임의 사용자 프로젝트에서도 keep을 버나"**로 다시 자격을 얻어야 재진입한다.
3. **네이티브 위임**: 손으로 짠 오케스트레이션 엔진을 버리고, Claude Code가 이제 네이티브로 주는 것(격리 서브에이전트 병렬 팬아웃, Stop hook 완료 게이트, CLI resume, 스키마 강제 출력)에 실행을 위임한다. 우리가 소유하는 것은 **규율 게이트와 얇은 drive 루프**뿐이다.
4. **완료를 증거로 판정**: 완료는 모델이 transcript에 *주장한 것*이 아니라 하네스가 *직접 실행·확인한 증거*로만 인정한다.
5. **순효능 북극성**: 최상위 acceptance는 내부 게이트 통과가 아니라 "자기 자신이 아닌 실제 작업에서, 조심해서 그냥 쓰는 것 대비 순효능"이다. 이 지표를 내부 게이트보다 *위*에 둬서 monoculture 재발을 구조로 막는다.

**한 문장**: 더 많은 기능이 아니라 *더 적은, 그러나 증거로 산* 기능으로 — 그리고 재구현이 아니라 네이티브 위임으로 — 작고 신뢰성 있게 다시 짓는다.

## 2. 재구축 동기 — 왜 다시 짓는가

### 2.1 표면 증상

현행 DITTO를 실제로 구동했을 때 반복해서 물린 것:

- **autopilot 한 판 3시간+.** 노드 그래프·ready-node 선택·dispatch를 손으로 짜고, planner→implementer→reviewer→verifier→security→retrospective + pre-mortem sweep + dialectic를 대부분 순차로 서브에이전트 스폰한다. 각 스폰이 fresh 컨텍스트를 통째로 로드하고, 노드마다 별도 CLI + hook 왕복. 3시간의 대부분이 이 순차 스폰이다.
- **antml 호출 오류.** tool-call 태그(`antml:invoke`/`antml:parameter`)를 빠뜨리면 호출이 통째로 malformed로 실패한다. 이건 모델/하네스 수준 슬립(실측 대략 1.1% 기저)이지만, **매 턴 대량 컨텍스트 주입**(거대한 PRIME_DIRECTIVE·charter)이 컨텍스트 압력을 올려 악화 요인이 된다.
- **손 JSON 파싱 오류.** 구조화 데이터를 서브에이전트가 stdout에 손으로 찍은 JSON으로 주고받고 DITTO가 파싱한다. 손 JSON → 문법 오류.
- **dual-host 낭비.** Claude Code와 Codex 두 호스트 지원(ADR-0016)을 위해 두 번째 어댑터·별도 빌드·surface projection·전용 fixture를 유지하는데, 정작 Codex의 실 런타임 발화(층위③)는 검증조차 안 됐다. 확인 안 된 호스트를 위해 이중 비용을 내왔다.

### 2.2 진짜 병 — 효능의 하락

위 증상은 하류 결과다. 진짜 병은 이 한 문장이다: **거의 전적으로 dogfooding이었는데도, 도구의 신뢰성과 효능감이 현저히 떨어졌다.**

DITTO의 존재 이유는 작업을 *더 신뢰할 수 있고 더 느리지 않게* 만드는 것이다. 순효능이 음수로 가면(3시간 + 잦은 오류 + false-green 완료 > 조심해서 그냥 Claude Code 쓰기), 게이트가 아무리 많아도 자기 명제에 실패한 것이다. 왜 도그푸딩을 했는데도 효능이 *성장과 함께 떨어졌는가* — 세 기전(진단, 일부 추론):

1. **단일 재배지(monoculture) 증거.** 모든 기능이 하나의 코드베이스(자기 자신)·한 사용자·한 도메인(TS CLI 오케스트레이터)에서만 검증됐다. "도그푸딩이 증명함"은 *DITTO 위에서* 작동함을 증명하지, 백서가 목적이라 못박은 *임의 사용자 프로젝트*(ADR-20260713)에서 작동함을 증명하지 않는다. 검증된 것과 목적이 벌어졌다.
2. **신뢰성은 곱셈, 축적은 덧셈.** 기능이 검증보다 빠르게 쌓였다. 시스템 신뢰성은 부품의 곱으로 떨어진다 — 부품 50개가 각 99% 신뢰라도 곱하면 약 60%(예시 산술, DITTO 실측 아님). 그래서 **기능이 늘수록 시스템 신뢰성은 단조 감소**한다. 축적된 gotcha 목록(`DITTO_SKIP_HOOKS` 누출·land-abort·changed_files 오염·store 경쟁)이 그 곱셈의 증거다.
3. **자기 채점(self-confirming) 검증.** DITTO가 DITTO를 검증했다. 채점자와 피채점자가 같은 저자·같은 맥락·같은 미검증 가정을 공유하면 "증거"는 부분적으로 자기확인이다 — 백서가 세션 차원에서 경고한 *자기확신*이 이제 *프로젝트* 차원에서 작동한 것. 자기 숙제를 자기가 매긴 셈이라, 도그푸딩 "증명"은 보이는 것보다 약하다.

### 2.3 진단이 재구축에 요구하는 것

세 기전은 재구축의 형태를 직접 결정한다:

- **zero-start** — 증거 코퍼스(도그푸딩 이력) 자체가 monoculture + 자기확인으로 오염됐으므로, 축적 집합을 신뢰해 걸러내는 대신 §7 불변식만 지고 다시 벌게 한다.
- **잣대 교체** — "DITTO에서 catch-rate>0"이 아니라 "임의 프로젝트에서 keep을 버나"가 재진입 기준.
- **순효능 북극성** — 내부 정확성(게이트 통과)이 아니라 baseline 대비 순효능을 최상위 acceptance로. 현행 DITTO의 근본 결함은 최상위 지표를 한 번도 게이트하지 않은 것이다.
- **monoculture 탈출 의무** — 재구축도 도그푸딩으로 부트스트랩하므로 같은 병을 재생산할 위험이 있다(§11). 이를 막는 유일한 길은 self가 아닌 검증 신호를 구조에 박는 것.

## 3. 보존 대상 — 바꾸지 않는 것

재구축이 형태를 바꿔도 이것들이 성립하지 않으면 DITTO가 아니다. 아래는 백서에서 추출한 원문 내용을 직접 담는다(경로 참조가 아니라 내용 임베드 — §4-11).

### 3.1 durable 철학 12원칙 (백서 §4)

1. **의도 먼저** — 명령을 수행하기 전에 그 명령이 달성하려는 결과를 파악한다. 성공 기준이 약하면 먼저 강화한다.
2. **모호함을 숨기지 않는다** — 해석이 둘 이상이고 결과가 갈리면 드러낸다. 낮은 위험 구현 세부는 에이전트가 판단, 제품 가치·도메인 의미·비가역 결정만 사용자 확인.
3. **가장 간단하되 검증된 최소 구현(공리)** — 가장 흔한 실패는 과소가 아니라 과잉이다. 요청되지 않은 기능·설정·확장성, 단일 사용·얕은 추상화, 일어날 수 없는 시나리오용 방어 코드를 만들지 않는다.
4. **외과적 변경** — 변경은 요청과 직접 연결된 부분으로 제한. 수정한 모든 줄에 이유를 댈 수 있어야 한다.
5. **완료는 증거로만** — 테스트·빌드·실행 로그·diff·재현만이 증거. 검증 못 했으면 완료라 말하지 않고 제한 이유·남은 위험을 남긴다.
6. **계획을 조용히 줄이지 않는다(무축소)** — 합의된 의도 단위를 임의로 축소·이월하지 않는다.
7. **반론은 협업** — 더 단순·안전한 길이 있으면 말하고 더 나은 경로로 진행. 반론이 작업보다 커지면 안 된다.
8. **결정 책임은 에이전트, 가치는 사용자** — 절차 결정은 에이전트가 소유. commit은 가역이라 에이전트 소유, push는 비가역이라 사용자 허가.
9. **위임으로 컨텍스트를 지킨다** — 탐색·조사·벌크 분석은 서브에이전트에 격리, 결론·증거·불확실성만 회수. 검증·리뷰는 fresh context에서.
10. **기록된 결정과 충돌하면 드러낸다** — ADR을 추론 시점에 일관 반영. method 충돌은 에이전트가 따르고, intent 충돌은 사용자만 푼다. 어느 쪽이든 근거와 함께 공개.
11. **권위는 코드에 있다** — 동작하는 코드·코드 곁 살아있는 지침이 진실 원본. drift할 설계 문서를 권위로 삼지 않는다.
12. **시드·핸드오프는 의도의 상태로** — 형성 중 의도는 원 의도+검증된 사실만(하류 편향 방지), 확정 의도는 축자 보존해 운반.

### 3.2 핵심 개념 (백서 §6, 압축)

- **work item / Record·Run 2-tier** — request(사용자 말) vs work item(추적 단위). 공유·커밋되는 Record와 개인·폐기 가능한 Run 분리.
- **intent lock** — 착수 전 의도를 확정한 산출물. 무축소 의무의 기준.
- **deep-interview** — 착수 전에 진짜 질문(제품 가치·도메인 의미·비가역 결정)을 변증론적으로 소진해 intent를 잠그는 인터뷰.
- **pre-mortem** — "이 일이 실패한다면 왜인가"를 미리 돌려 비가역 위험·커버리지 공백을 front-load.
- **autopilot / no-auto-pick(materialize≠drive)** — 발견한 후속은 물질화하되 자동 구동하지 않는다. 유일 예외: 분류기가 "재현되는 실동작 버그"로 판정한 결함만 같은 run에서 고칠 때까지.
- **completion contract** — 완료 주장에 필수인 필드 집합. `final_verdict=pass`는 모든 AC가 evidence와 함께 pass일 때만.
- **verdict / unverified / oracle / evidence** — 판정(pass|partial|fail|unverified), "검증 못 함"의 명시 표기, AC를 기계적으로 판정하는 근거, 참조로 담는 증거.
- **context packet / handoff** — 위임 계약, 같은 작업을 이어받게 하는 pull 방식 문서.
- **dialectic** — Producer/Opponent/Synthesizer 3역 적대 심의. Opponent는 다른 맥락·다른 provider.
- **decision-conflict** — ADR 충돌을 kind×level×mode로 분류·라우팅·공개.
- **memory / freshness 2축** — provenance·freshness 그래프. 축1=SoT↔파생, 축2=코드↔SoT.

> **주의**: 이 개념들이 *보존 대상*이라는 것은 "역할"이 보존된다는 뜻이지, 현행 구현(39커맨드·CodeQL·memory graph 등)이 그대로 포팅된다는 뜻이 아니다. 무엇이 실제로 재진입할지는 §8 증거 감사가 결정한다.

### 3.3 아키텍처 12불변식 (백서 §7) — 반드시 코드로 보존

1. **완료 = 모든 AC가 evidence와 함께 pass.** 하나라도 미충족·미검증이면 pass 불가. work item 전체가 기준.
2. **무축소를 코드로.** 잠긴 의도·계획의 in-scope 잔여를 조용히 축소·이월 불가. 하나의 의도 = 하나의 단위.
3. **fail-closed 게이트.** 불확실·도구 부재·판정 불가면 기본값은 차단. 통과는 명시적 근거·override로만.
4. **fresh context 검증.** 검증·리뷰는 생성과 다른 맥락에서. 효율이 아니라 유효성 조건. **정밀화(연구): verifier는 구현 추론이 아니라 계약을 본다** — 공유 컨텍스트가 reward hacking의 인과적 원인이라는 실증(arXiv 2407.04549) 때문에 이는 위생이 아니라 구조적 필수다.
5. **증거는 참조로.** evidence는 원문 적재 아니라 path/hash/preview 참조(context rot 방지).
6. **우아한 강등.** 선택적 외부 도구 부재가 의도 실현을 막지 못한다. 정직하게 degrade하고 floor를 지킨다.
7. **호스트/프로바이더 절연.** 코어는 LLM을 직접 부르지 않고 판단을 seam으로 위임, 결과는 스키마로. (재구축에서 이 seam은 "네이티브 Claude Code = 단일 host 어댑터"로 실현 — §7-4.)
8. **tier 격리.** 제품/전역/개인 3계층, 공유(Record·커밋)와 개인(Run·gitignore)을 안 섞음.
9. **스키마가 SoT.** 데이터 계약의 단일 진실원은 스키마. 동작을 별도 설계 문서로 이중화 안 함.
10. **push는 user-gated, commit은 agent-owned.** 가역/비가역 경계를 권한 경계로.
11. **메타-도구는 사용자 환경에 맞춘다.** DITTO 기능을 도그푸딩 스택에만 굳히지 않는다. 임의 사용자 언어·환경에 맞추거나 정직하게 degrade. **(재구축에서 이 불변식은 §2.3 monoculture 진단과 직결 — 최우선으로 승격.)**
12. **언어-중립 어휘.** 사용자가 대화에 쓴 + 응답에 노출된 어휘만 공유어휘. 미합의 용어·설명 없는 축약을 reject.

## 4. loop engineering — 이론적 지반

재구축의 핵심 실행 모델은 "종료 조건을 주고 통과할 때까지 반복하는" loop다. 이 패러다임을 맹신도 배척도 하지 않고, 정확한 선을 긋는다.

### 4.1 회의적 시각 (왜 순진한 loop를 신뢰하지 않는가)

- **종료 조건 T는 의도의 손실 사영이다.** 루프는 T를 달성한다 → T가 곧 결과다. 그런데 T = "옳고 잘 설계됐고 의도에 맞는 코드"인 경우는 거의 없다. 에이전트는 T의 *문자*에 최적화한다(Goodhart: 척도가 목표가 되면 좋은 척도이길 멈춘다). "게이트 통과 ≠ 의도 충족".
- **T가 담아야 할 두 종류의 검증 가능성이 다르다.** ① 의도 대비 정확성 — AC/oracle만큼만 좋다. ② 소프트웨어 공학적 품질(결합도·응집도·추상화 수준·개념적 무결성) — **애초에 pass/fail oracle로 표현 불가능**하다. 정도와 판단의 문제지 술어가 아니다.
- **바이럴 데모가 잘 되는 건 어려운 문제를 *푼* 게 아니라 정의에서 *지워서*다.** 한 프롬프트로 게임/서비스를 만드는 데모는 넷을 지운다: **제약**(보존할 기존 불변식 없음 — greenfield), **oracle**("되는 것처럼 보임"이 완전한 저차원 oracle), **시간**(throwaway라 유지보수 가중치 0), **이해관계자**(생성 시점에 방 안에 없는 사용자·운영·보안이 없음 → 컨벤션이 지킬 대상 자체가 없음). 이 영역의 결과를 프로덕션·유지보수 소프트웨어로 외삽하는 것이 하이프의 오류다.

### 4.2 균형 관점 — steelman (연구 근거)

진지한 loop engineering은 바이럴 원샷 데모와 명확히 다르고, 그 진지한 버전의 명제는 **우리 설계와 거의 동형**이다. (아래는 웹 조사 결과 — arXiv·실무 출처. 착수 전 원문 재확인 권장. 출처 ID는 조사가 반환한 그대로다.)

- **정의 — 프롬프트가 아니라 경계 있는 산출물.** 진지한 loop는 "더 나은 프롬프트"가 아니라 인간이 설계해 하네스에 넘기는 *경계 있는 재사용 산출물*(트리거·목표·검증·종료 규칙·메모리)이다. 데모와 구분하는 golden rule: **"한 턴의 결과가 다음 행동을 바꾸는가."** 안 바꾸면 루프가 아니라 그냥 scheduled prompt다. (arXiv 2607.00038v1; Datadog "harness-first agents")
- **load-bearing은 프롬프트가 아니라 체크 설계.** "루프에서 어렵고 값진 부분은 더 나은 지시를 쓰는 게 아니라 *언제 완료인지 판정하는 체크를 설계*하는 것"이고, "루프는 맹목적 신뢰를 재현 가능한 증거로 바꿈으로써 keep을 번다." — §5.2(증거 판정)와 직접 일치. 결정론적 source-anchored verifier가 hallucination을 약 절반으로 줄인 반면 self-critique는 무효였다. (arXiv 2607.00038v1, 2601.22025)
- **maker-checker 분리는 선택이 아니라 필수 (reward hacking 실증).** 생성자와 채점자가 같은 모델·공유 컨텍스트면 reward hacking이 *자발적으로* 발생해 점수는 오르나 품질은 안 오른다. **공유 컨텍스트가 인과적 원인** — 컨텍스트가 비대칭이면 격차가 사라진다. 프로덕션 하네스는 이를 구조로 강제한다: 분리된 에이전트·job payload·실행 큐, 공유 대화 이력 없음, 그리고 **verifier가 구현 추론이 아니라 *계약*에 접근**. (arXiv 2407.04549 "Spontaneous Reward Hacking in Iterative Self-Refinement"; 2605.25665) — 조건성 caveat: 효과는 보편이 아니라 조건적(강한 모델일수록 덜 hack, 긴 공유 컨텍스트가 증폭)이고, 상관된 모델 실패까지 없애지는 못한다.
- **검증 비대칭 = 이론적 힘.** 좋은 종료 신호가 존재하는 도메인에서 검사가 생성보다 쉽다. 실측된 generation-verification gap: 모델이 정답을 *만들 수는* 있으나(높은 oracle pass@100) *고르지* 못한다(GPQA Diamond: oracle 82.8% vs majority 45.5%, 37.3pp). 검증 기반 test-time scaling이 verifier-free를 능가하고 컴퓨트가 늘수록 격차가 벌어진다. (Stanford Weaver 2025-06; Setlur et al. ICML 2025, arXiv 2502.12118) — caveat: reasoning·math 도메인 조건, 다수 verifier 합의엔 easier-subset 선택 효과.
- **옹호자들이 이미 내 비판을 인정한다.** 문헌이 "final task success를 넘는 평가, incomplete feedback 하 검증"을 미해결 과제로 *명명*한다(arXiv 2605.18747). 실제 Stripe 케이스 — 백엔드가 CI를 2사이클에 통과했으나 *계약에 인코딩 안 된* 비즈니스 로직 2건(입금 차감·할인 계산)을 출하, "동작 테스트 통과는 필요하지만 불충분"을 스스로 결론(arXiv 2605.25665). → 내 회의는 *순진한 데모 버전*엔 정확하지만, *규율 있는 프로덕션 버전*은 이미 그 문제를 계약 우선·독립 검증·실패 분류·contract refinement로 다룬다. caveat: 단일 조직 단일 케이스(diagnostic이지 definitive 아님).

### 4.3 종합 — 신뢰하는 부분과 규율로 감싸는 부분

**루프가 적이 아니라 영역이 적이다.** 같은 루프라도 데모가 지운 넷(제약·진짜 oracle·시간·이해관계자)을 도로 부과하면 정당한 공학 도구고, 안 부과하면 일회용 기술이다. 이것을 재구축의 **설계 리트머스**로 박는다:

> 우리가 짓는 *어떤 루프든*: (1) 데모가 지운 넷을 도로 부과하는가? (2) 완료를 "되는 것처럼 보임"(transcript/모델 주장)이 아니라 진짜 oracle(테스트 실행·파일 확인)로 판정하는가? (3) golden rule — 한 턴의 결과가 다음 행동을 실제로 바꾸는가(안 바꾸면 루프일 필요 없음)? 셋 다 아니면 넣지 않는다.

**결정적 한계 — 그리고 그것이 우리에게 경계를 그어준다.** 옹호자들도 인정하는 근본 한계: 검증 비대칭은 *좋은 종료 신호가 존재하는 곳에서만* 성립한다. verifier가 불신뢰한 어려운 문제에선 규율 자체가 열화하고(reference-answer 오류 39.1%), hard 문제에선 verification이 generation보다 *더 어려워지는* 역전까지 보고된다(arXiv 2606.26300). **이게 §4.1과 정확히 맞물린다** — 루프는 oracle이 명확한 곳(정확성)에서 강하고, oracle이 없는 곳(설계 건강성·의도)에서 약하다. 후자가 바로 §4.1이 "pass/fail oracle로 표현 불가능"이라 한 그 차원이다. 그래서 **DITTO의 설계 원칙: 루프(증거 게이트)를 정확성 차원에 쓰고, 설계 건강성·의도 차원은 생성 조형 + 인간에 맡긴다.** 이 경계가 "어디서 루프를 신뢰하고 어디서 fallback하는가"를 결정한다.

그리고 T가 원리적으로 불완전하다는 전제 위에서, 갭을 계층으로 나눠 떠받친다:

| 갭 | 무엇이 새는가 | 무엇이 막나 |
|---|---|---|
| 의도↔T | 옳은 T를 인코딩했나 | 착수 전 의도 잠금(deep-interview·pre-mortem) — T 자체를 신뢰 가능하게 |
| T↔코드 정확성 | "통과했다"가 참인가 | fresh-context 독립 검증(증거 참조·per-AC oracle) |
| 품질/설계 건강성 | oracle로 못 잡는 부분 | 게이트로 못 막음 → *생성을 조형*(외과적·최소증분·Tidy First·적대 dialectic·지식 축적) |
| 시간(엔트로피) | 국소 허용 변경의 누적 열화 | 주기적 *구조* 리뷰·ADR 일관성·최소증분 편향 복리 |

**설계 목표는 "T를 충분하게 만든다"(불가능)가 아니라 "T가 놓치는 잔차를 최소화하고, 놓친 것을 가시화한다"** — `unverified`·resolvability·충돌 공개가 그 가시화다. 가치는 "모든 걸 잡는 게이트"가 아니라 "안 잡은 걸 잡았다고 주장하지 않는 시스템"이다.

## 5. 정초 명제 — 무엇을 짓는가

> **적용 범위 (먼저 못박을 것).** 이 절 — 특히 완결성 축(§5.4~5.11) — 은 *재구축을 규율하는 내부 절차*가 아니라 **DITTO가 임의 사용자 프로젝트에 제공하는 기능**이다. 재구축은 이 기능의 첫 *적용·검증 대상*일 뿐이다(§8.3 non-self 검증). 따라서 여기 나오는 모든 메커니즘 — park·immutable archive·AC 적용 facet oracle(작성 시점 `planned_application_oracle` → 검증 시점 `observed_application_evidence`)·교차모델 dialectic·disposition-queue drain — 은 도그푸딩 스택(bun/TS)이 아니라 **임의 스택·임의 repo에서 작동해야 한다**(§3.3#11 메타-도구는 사용자 환경에 맞춘다). "재구축에서 이렇게 쓴다"는 예시는 그 기능의 첫 사용례이지 기능의 경계가 아니다. 이 축을 재구축 전용 규율로 읽으면 §2.2 monoculture 진단(자기 자신에만 맞춘 기능)을 이 축 자체가 재생산한다.

### 5.1 게이트 소유 / 오케스트레이션 위임

두 종류의 루프를 가른다:

- **acting 루프 (위임)** — 어떤 에이전트가 어떤 순서로 병렬/직렬로 돌고 결과를 모으는 실행·팬아웃. Claude Code 네이티브가 잘 준다.
- **drive 루프 (소유)** — "AC가 증거와 함께 실제로 충족될 때까지 계속 몬다. 모델이 다 됐다고 여긴다고 멈추지 않는다." 반드시 우리 것.

네이티브의 종료 기준 = "모델이 스스로 끝났다고 판단"인데, 그게 바로 완료 게이트가 불신하려는 대상 그 자체다(백서 §2). 종료 판단을 네이티브에 넘기면 moat을 잃는다.

### 5.2 완료를 증거로 판정 (not transcript)

Claude Code는 **덫과 약을 동시에** 준다(§6에서 상술):
- 덫: `/goal`은 fresh 모델이 판정하지만 *명령을 실행하거나 파일을 독립적으로 읽지 않고* Claude가 transcript에 이미 말한 것만으로 판정한다 → "그럴듯함" 판정.
- 약: agent-based Stop hook은 서브에이전트가 테스트를 직접 돌리고 파일을 직접 읽어 판정한다 → "증거" 판정.

**재구축의 moat 한 문장: 완료 게이트를 transcript가 아니라 증거로 판정하는 쪽을 택한다.**

여기에 연구가 준 정밀화가 하나 붙는다. 생성자와 채점자가 **공유 컨텍스트**면 reward hacking이 자발적으로 발생한다는 게 실증됐다(공유 컨텍스트가 인과적 원인 — arXiv 2407.04549). 그래서 fresh-context 검증(§7-4)은 위생 권고가 아니라 *reward hacking을 막는 구조적 필수*이고, 정밀한 형태는 이렇다: **verifier는 구현의 추론 과정이 아니라 *계약(AC·oracle)*에 접근한다.** 채점자가 "생성자가 어떻게 그 답에 도달했나"를 보면 그 서사에 물들고, "무엇을 충족해야 하나"만 보면 독립적으로 채점한다.

### 5.3 순효능 북극성 — 내부 게이트 위의 최상위 AC

모든 내부 게이트를 통과하고도 순마이너스일 수 있다. 그래서 최상위 acceptance를 이렇게 둔다:

> **자기 자신이 아닌 실제 작업에서, 조심해서 그냥 쓰는 것(baseline) 대비 순효능**(신뢰성 AND 시간)이 양수인가?

이 지표를 내부 정확성보다 위에 두고, 재구축의 각 증분이 이 지표를 후퇴시키지 않는지를 게이트한다. 이것이 §2.2 monoculture·과축적을 구조로 막는 유일한 장치다.

**측정은 self에서 성립하지 않는다.** baseline("맨 Claude Code")을 만들려면 DITTO를 꺼야 하는데, dogfood repo에서는 charter(CLAUDE.md)·hook·경로가 이미 배어 있어 끄면 *baseline이 생기는 게 아니라 측정 대상 자체가 사라진다* — §2.2 self-confirming의 측정론적 얼굴이다. 게다가 hook은 on/off 경계를 흐린다: hook이 매 프롬프트에 charter를 주입하는 것은 오케스트레이션 엔진이 아니라 "값싼 지침 주입"이라, DITTO를 이진 on/off로 못 가른다. 그래서 순효능은 원리적으로 **non-self fixture에서만, 그것도 층별 ablation으로** 잴 수 있다(프로토콜은 §8.3) — 효능이 값싼 층(hook 주입 charter)에서 오는지 비싼 층(엔진)에서 오는지 가르는 credit assignment가 곧 재구축의 자기정당화이기 때문이다(엔진이 얇은 주입 대비 값을 하나).

### 5.4 완결성 — 미결(undecided) 상태로 끝내지 않는다

**완결성(disposition-completeness)** = 발견된 모든 결함·이슈·미검증 항목이 명시적·정당화된 처분(disposition)에 도달하기 전에는 run이 완료를 주장할 수 없다는 성질.

**문제(현행의 실측 실패 모드).** 자율 증분은 늘 "제대로 끝났다 할 수 없는 지점"에서 멈췄다 — 발견한 버그를 범위 이탈 우려로 방치, 후속을 증식, 미검증/잔여 누적. 이건 완료가 아니다. **완료의 반대는 "미완"이 아니라 "미결로 얼어붙음"이다.**

**원칙.** 현실 개발에서도 설계 맹점·현실적 설계 변경은 부지기수지만 거기서 *멈춘 채 끝내지 않는다* — 방향을 틀거나 폐기하되 결론을 낸다. 그래서 발견된 모든 항목의 합법적 exit는 셋뿐이다:

1. **해결(resolve)** — 고치고 증거로 확인, 또는 검증 완료.
2. **결정 후 실행(decide & carry out)** — 설계를 틀거나 폐기하되 그 결정을 *실제로 반영*(얼어붙지 않음). 증분은 결정 후에도 coherent해야 한다.
3. **에스컬레이션(escalate)** — 사용자만 풀 결정(설계 변경·비가역·범위)이면 멈추고 인계 — 단 "done"을 주장하지 않고 결정을 프레이밍해서.

불법 exit(현행이 하던 것): **표시하고 멈춤** — 방치·미검증-종결·in-scope 잔여를 후속으로 relabel.

**닫아야 할 세 구멍** (현행이 이미 있는 게이트[종료 완전성 게이트 ADR-20260710·resolvability 게이트]를 통과시킨 방식 — 추론, 착수 전 코드로 확인):
- **disposition 자기판정.** "범위 밖"·"해결 불가"를 *찾아낸 그 에이전트가* 판정 → gameable. §2.2 자기채점 병이 완료 지점에서 재발. 이 병의 뿌리를 두 층위로 뽑는다. **(1) 판정 표면 자체를 최소화한다.** 발견된 in-scope 잔여와 재현되는 버그는 *라벨링이 아니라 처리*로 간다(처리는 증거로 완수되니 채점할 게 없다) — 그래서 유일하게 남는 판정 표면은 "이건 지금 처리할 잔여가 아니라 진짜 새 범위다"라며 백로그로 *빼는* 결정 하나뿐이다(발견된 걸 전부 처리하면 원 요청 범위가 폭발하므로 이 "빼기" 분류 자체는 없앨 수 없다). **(2) 남은 그 하나를 교차모델 적대 심의(dialectic)로 판정한다.** 같은 시스템·같은 맥락의 self-verifier는 계약만 봐도 "범위 밖" 라벨을 독립적으로 반복 승인할 수 있다(§4.2: 상관된 모델 실패는 맥락만 갈라선 안 사라진다). 그래서 이 "빼기" 분류와 종료 진단은 *다른 provider의 Opponent*를 세운 적대 심의로 올린다(§5.10) — 맥락 분리를 넘어 모델까지 다르므로 자기채점보다 훨씬 강한 독립성이되, 여전히 결정론적 oracle은 아니다(못 없애는 잔차는 §5.11).
- **materialize 구멍.** "후속으로 물질화했으니 tracked"가 게이트의 문자는 만족하나 정신을 위반(무축소를 relabel로 우회). → **materialize-as-follow-up은 독립 판정으로 *진짜* 현재 의도 범위 밖이고, 없어도 증분이 coherent할 때만 합법적 disposition.** in-scope 잔여·증분을 깨는 항목은 미룰 수 없다 — 끝내거나 에스컬레이션.
- **unverified 종결.** unverified는 정직한 *전이* 상태지 *종결* 상태가 아니다. resolve되거나 blocking escalation으로 전환돼야 하며, "done" 주장을 막는다(fail-closed).

**후속 증식 방지 (coherent-increment 조건).** 후속이 합법이려면 (a) 독립 판정으로 진짜 새 범위이고 (b) 그것 없이도 현재 증분이 coherent(안 깨짐)해야 한다. 증분을 깨는 건 후속이 아니라 미완의 현재 작업이다. 이 조건이 후속 개수를 자연히 bound하고 "산더미 물질화 후 done" 패턴을 막는다.

**materialize≠drive와의 관계.** 이 원칙은 그걸 죽이지 않고 *정밀화*한다. 자동 구동 금지는 진짜 새 기능에 여전히 옳다(백서 no-auto-pick). 다만 그게 in-scope 잔여를 미루는 데 악용됐다 — 미룰 수 있는 건 오직 진짜 out-of-scope + coherent-increment 항목뿐. (재현되는 실동작 버그를 같은 run에서 구동하는 기존 예외는 유지된다.)

**북극성과의 관계.** 완결성은 §5.3 순효능의 신뢰성 축을 완료 지점에서 집행하는 장치다 — "미결로 끝난" run은 정의상 순효능을 후퇴시킨다.

### 5.5 완결성의 시스템 차원 — 상태 legibility

완결성은 개별 run의 책임을 넘어 *시스템 전체가 읽을 수 있는 상태*를 유지하는 전제다. 미완은 국소적 품질 미스가 아니라 **상태 엔트로피**이고, 시간과 함께 복리로 커진다.

**legibility 시험 — 세 질문.** 상태는 늘 값싸게 답할 수 있어야 한다: **어디까지 왔나 / 정말 일단락됐나 / 잊혔나.** 깨끗이 완료된 의도는 셋을 자명하게 답한다(done = 일단락 = 안 잊힘). 미결로 얼어붙은 의도는 셋 다 비싼 고고학 없이 못 답하고, 그런 의도가 하나 쌓일 때마다 미래의 모든 의도가 헤쳐야 할 상태 공간이 오염된다 — 중복·망각·추적 불가의 근원.

**전제, 목표가 아니라.** "어떻게든 완료"는 목표가 아니라 전제다. 이게 안 깔리면 백로그·후속 관리 기계는 순마이너스 — 늪을 정리하는 게 아니라 *생산*한다.

**모든 의도는 읽을 수 있는 종료 상태에 도달한다** — 셋 중 하나:
- **resolved** (해결·검증 완료),
- **명명된 blocker에 깨끗이 park** (에스컬레이션 — "결정 D 대기, X까지 done·검증됨, 재개는 D 답 필요"라고 상태가 스스로 말함),
- **깨끗이 폐기** (설계 변경으로 이 의도는 죽음 — 죽음으로 *기록*).

금지되는 유일한 결말: **흩어진 잔여를 남긴 늪.** 깨끗한 park는 미완이 아니다(상태가 자기 blocker를 명시하므로). park의 정확한 생애(off-main·도태)는 §5.7. **결정됨(사용자 확인): park는 legible한 run-종료로 인정하되 경계 있는 유예다. 자율 실행에선 resolve가 기본이고 park는 §3 두 sanctioned 조건(정초 방향 뒤집힘·사용자-only 결정)에서만.**

**상태 모델 요구 (늪은 결국 상태 모델 문제).**
1. **단일 출처 상태** — 의도마다 진행/일단락/망각을 값싸게 답하는 하나의 현재 상태.
2. **생성 시 dedup** — 후속·잔여는 blind-append가 아니라 기존 백로그와 대조 후 생성(§5.4 coherent-increment가 *개수*를 bound하면, 이건 *동일성*을 bound).
3. **silent-forget 금지** — 생성된 모든 항목은 disposition 흔적을 남긴다. 잊히는 항목 0.

**저장소 부채와의 연결.** 백서 §8-3 부채 #3(공유 원장 다중 producer full-replace)·#4(락 없는 store write = 유실)가 *이 늪의 저장소 레벨 버전*이다. 미완 의도(워크플로 레벨) + 유실·경쟁 store(저장소 레벨) = 복리 혼란. 재구축은 둘 다 고친다.

**따름정리 (§8 감사 직결).** 백로그·materialize·후속 기계는 완결성이 집행될 때만 순플러스다. 전제가 없으면 그 기계가 늪을 만든다 → **그 기계 자체가 유력한 prune 후보**(완결성이 그것을 안전하게 만들지 못하면 잘라낸다). 현행이 이 경우였을 가능성이 높다(완결성 미집행 + 정교한 materialize = 늪).

**hard invariant인 이유.** 오래된 코드베이스일수록 늪 화해 비용이 *초선형*으로 커져, best-effort는 결국 도구를 안 쓰는 게 낫게 만든다(효능 음수, §2.2). 그래서 완결성은 권고가 아니라 fail-closed hard invariant — "미결 잔여가 있으면 done 불가", 예외는 명명된 blocker park뿐.

### 5.6 비대칭 — 왜 완결성은 그 자리에서(capture-at-source) 집행되나

**잘못(버그) vs 미완성의 비용은 비교가 안 된다 — 판정 기준의 유무 때문이다.**

- **버그 = 알려진 의도에 대한 불일치.** 의도가 정의돼 있고 동작이 그걸 어긴다 → *판정 기준이 버그 자체에 내장*. "옳음"을 버그가 말해준다. 비용 = 수정 그 자체, 국소적.
- **미완성 = 의도 기록 자체의 구멍.** 무엇이 있어야 했는지·왜 없는지를 말해줄 정보가 바로 그 없는 것이다 → *판정 기준이 없다*. 원인이 넷으로 갈린다 — 버그(끝냈어야 했는데 깨짐)/의도적 보류(이유가 완성 방식을 제약)/막다른 길(폐기 — "계획대로 완성"하면 실패를 재현)/망각 — **넷이 정반대 대응을 요구**한다. 원인 재구성 전엔 손댈 수 없고, 재구성이 불가능하면(맥락을 남긴 세션이 사라졌으면) **방향조차 못 잡는다.**

한 줄: **버그는 자기 수정 기준을 지니고 오지만, 미완성은 자기 원인조차 숨긴 채 온다.** "나중에 누가 끝내면 되지"는 거짓 경제 — 물려받는 건 작업이 아니라 미스터리이고, 미스터리는 풀 수 없을 수 있다.

**capture-at-source 원칙.** 원인을 값싸게 잡을 수 있는 시점은 *구멍이 생기는 그 순간 하나뿐*이다(그걸 만든 세션의 맥락에). 세션 종료·컨텍스트 압축 뒤엔 원인이 회복 불가로 붕괴한다. 따라서:

> **완결성은 사후 정리가 아니라 그 자리에서 집행된다.** "백로그를 나중에 화해시키자"는 구조적으로 불가능하다 — 화해에 필요한 정보가 그때쯤 0으로 사라진다.

이것이 §5.5의 **깨끗한 park ≠ 늪**을 갈라주는 정확한 기전이다: park는 원인을 그 자리에서 잡아 미래 독자에게 *경계 지어진 작업*을 남기고, 늪은 원인을 안 잡아 *회복 불가 미스터리*를 남긴다. 그래서 non-resolved로 남기는 모든 항목의 disposition은 **원인+의도를 일급 데이터로 각인**해야 하고(blocked-on-decision / genuinely-out-of-scope-because / dead-end-discarded-because 중 하나), **"망각"은 결코 합법 disposition이 아니다**(§5.5 silent-forget 금지).

**버그와 미완성의 통합.** 일부 버그는 설계 결정을 요구한다(버그가 설계 맹점을 드러낼 때). 그 버그도 *방향을 막는* 항목이 되어 미완성과 **같은 escalation exit**로 간다. "결정을 요구하는 버그"와 "결정을 요구하는 미완성"이 한 메커니즘으로 통합된다.

### 5.7 park의 생애 — off-main + 도태(attrition)

park는 정당한 종료 상태지만 *경계 있는 유예*다. 두 제약이 park를 늪과 가른다.

**제약 1 — park는 main에 올라가지 않는다.** main = resolved 의도만. 누구든 main을 읽으면 끝난 일만 본다 — 반쯤 된 park 코드가 트렁크를 오염시키지 않는다. park는 자기 브랜치/worktree에 유예되고(ADR-20260626 worktree 격리), 그 *존재·blocker·나이*는 공유 상태(Record tier)에 legible하게 남는다(§5.5 세 질문 답 가능). commit은 되되(작업 보존) merge/push는 안 된다(§4-8 push user-gated). → 두 종료의 갈래: **resolved → main 착지 / park → 브랜치 유예, 착지 안 함.** 둘 다 legible한 run-종료지만 main엔 resolved만.

**제약 2 — 장기 park는 도태된다.** park가 무기한 limbo면 이름만 바꾼 늪이다. park는 시계를 달고 산다:
- park의 *resolvability 자체가 시간과 함께 붕괴*한다 — base가 drift하고 §5.6대로 원인 맥락이 회복 불가로 썩는다. park는 그냥 낡는 게 아니라 *덜 해결 가능*해진다.
- 그래서 아직 잘 결정할 수 있는 동안 **resolve-or-delist를 강제**한다. aging park는 escalating 압력으로 재부상하고, staleness 임계를 넘으면 **기본값은 폐기(cull)** — 되살리지 않으면 active에서 죽는다. bias가 보존이 아니라 도태여야 SLOP이 안 쌓인다.
- **폐기(cull) = 작업 삭제가 아니라 delist + archive 보존.** 여기서 "죽는다"는 *main/active 판정에서 빠진다*이지 *작업 비트가 지워진다*가 아니다. 도태의 목적은 "미래 의도가 헤쳐야 할 상태 공간을 오염시키지 않게 active에서 빼는 것"이지 작업을 잃는 게 아니다 — cull된 park는 브랜치 커밋·원인·base·재개 조건을 그대로 담은 **immutable archive**로 옮겨진다(§5.6대로 원인이 각인돼 있으니 "회복 불가 미스터리"=늪이 아니다). **작업의 실제 물리 삭제는 사용자 명시 승인 또는 deterministic supersede(다른 것이 이걸 대체했다는 증거)일 때만.** 자율 실행에서 무읽음-무응답이 *삭제*로 해석되면 안 된다 — 무응답의 기본값은 delist(active에서 빼되 archive 보존)이지 소멸이 아니다.
- **도태 ≠ 조용한 삭제.** delist도 *강제된 결정*이지 auto-GC가 아니다 — 사용자에게 먼저 surfacing되고, 결정 안 하면 기본값이 delist로 resolve될 뿐. 기록된다(§5.5 silent-forget 금지: "intent delisted: stale-park·미결정·날짜, archive→<경로>"). 그래서 도태는 *작업을 조용히 잃는 새 SLOP*이 아니다 — archive가 legibility(§5.5 세 질문)를 보존하므로.

**§5.6과의 연결.** 도태는 임의 garbage-collection이 아니라 *§5.6을 park에 적용한 것*이다 — park의 원인+base가 아직 회복 가능한 동안 결정을 강제한다. 더 기다리면 폐기 결정조차 근거를 잃는다.

### 5.8 무축소·미배선 금지 — AC는 의도 전체를 *라이브로* 검증한다

완결성의 또 다른 얼굴: 잔여를 처분하는 문제(§5.4~5.7)와 별개로, **완료 기준 자체가 의도를 프록시로 대체**하는 두 실패 모드가 잦게 발생했다.

- **A. 임의 의도 축소** — autopilot이 스스로 의도를 좁혀 구현하고 좁힌 버전을 done으로 판정.
- **B. 미배선 (매우 빈번)** — 기능은 만들되 라이브 경로에 안 꽂음. 증상: 배선 안 됨/켜지지 않음/경로 끊김. 단위 테스트 green인데 실제론 도달 불가·off. **DITTO 자기 이력에 반복 기록**(wire-or-drop·liveness 게이트·"단위 green ≠ floor"·pure함수 미배선 갭) — *반복해 패치됐다는 것 자체가 현행 아키텍처가 구조적으로 못 막는다는 증거*.

**공통 병:** 완료 기준(T)이 *의도가 라이브로 실현됐는지*가 아니라 *프록시*(코드 존재·단위 green·에이전트가 고른 AC)를 검증했다. A는 약한 AC를 좁힌 구현이 통과, B는 단위-프록시가 통과하는 동안 라이브-의도는 거짓. §5.2(증거≠transcript)가 완료 지점에서 재발한 것.

**근본 원인 — AC 작성의 빈틈 (검증이 아니라 작성).** 게이트는 안 망가졌다. *불완전한 AC 집합을 충실히 검증*했을 뿐이다. AC가 "기능이 잘 개발됐나"만 담고 "적용됐나"를 안 담으면, 아무리 엄격히 검증해도 *틀린 것을 철저히* 검증한다. 그래서 고침은 검증 게이트가 아니라 **AC 작성 단계(intent-lock/planner)**로 이동한다.

**AC·완수조건을 애초에 잘 쓰는 것이 1차 방어선 — 그게 deep-interview + pre-mortem의 존재 이유다.** 완료 지점의 방어(§5.10 dialectic 종료 진단)는 *놓친 것을 뒤에서 잡는* 장치이지 1차 방어선이 아니다. AC와 오케스트레이션 완수 조건(테스트 케이스들)을 처음부터 최대한 완전하게 뽑아내는 것이 우선이고, 그것이 바로 착수 전 **deep-interview**(의도를 변증론적으로 소진해 무엇이 성공인지 잠금)와 **pre-mortem**("이 일이 실패한다면 왜인가"를 미리 돌려 커버리지 공백을 front-load)의 역할이다. 이 둘로 의도를 잡은 *뒤에*, 실제 AC 문장과 테스트 설계는 planner 단계에서 실현-상태 형태로 보강한다. 즉 AC 완전성은 양끝에서 방어된다: **착수 전 front-load(deep-interview·pre-mortem) → 완료 시점 적대 검토(dialectic).** 그래도 남는 잔차 — "AC가 정말 의도 전체였나"의 최종 판단 — 는 §5.11대로 사용자만 가진다.

**핵심 원칙 — AC는 *실현된 상태*를 기술한다, *구축 활동*이 아니라.** "함수 추가·단위 green" = 구축 활동 프록시(나쁜 AC). "실제 진입점을 호출하면 개선된 동작이 난다 / orphan 없음" = 실현된 상태(제대로 된 AC). 모든 AC를 실현-상태 형태로 강제하면 "만들었지만 안 꽂음"을 *애초에 충족된 AC로 표현할 수 없다* — 미배선 갭이 작성 단계에서 소멸.

**두 구조적 고침:**
1. **AC 완전성 (작성 단계에서 fail-closed).** AC 집합이 의도 전체를 실현-상태로 포착해야 한다. 구조적으로: **AC의 두 facet을 스키마로 분리·필수화** — 정확성 facet(격리 상태에서 옳게 동작하나) ⊥ **적용 facet(라이브 경로로 도달·효과 나나)**. 능력 변경 AC는 둘 다 있어야 완전. **작성 시점 게이트**가 적용 facet 빠진 AC 집합을 reject(기억에 의존하지 않음). 좁힌 구현은 남은 의도를 in-scope 잔여로 남기고 §5.4가 resolve/escalate를 강제(silent 축소 불가).

   **단, 적용 facet은 작성 시점과 검증 시점으로 쪼갠다(형식 채우기 방지).** 작성 시점엔 아직 라이브 증거가 없으므로 "적용됐다"는 주장을 넣으면 "진입점에 연결됨" 같은 산문으로 스키마만 통과하는 게이밍이 열린다(작성 게이트가 진위까지 막으려 하면 라이브 증거가 필요해 순환). 그래서 두 상태로 분리한다: **작성 시점엔 `planned_application_oracle`을 필수화** — "적용됐다"가 아니라 *어떻게 확인할지의 반증 가능한 명세*(실행 명령·진입점·관측할 효과·orphan 판정 기준). 작성 게이트는 그 명세의 진위가 아니라 *존재+반증 가능성*만 검사한다. **검증 시점엔 `observed_application_evidence`를 필수화** — 그 명세대로 라이브 경로가 실제로 통과했다는 증거(§5.8-2). 둘을 같은 "facet 충족"으로 부르지 않는다 — 작성은 확인 방법을, 검증은 확인 결과를 요구한다.
2. **AC 검증 완전성 + 라이브-경로 oracle.** 모든 AC를 fresh 증거로 독립 검증, 하나도 skip 없음(skip = unverified = done 차단, §5.4). AC를 닫는 증거는 **라이브 경로를 통과**해야 한다(도달성 + 실제 효과), 단위 격리 호출이 아니라. **orphan 코드(추가됐으나 미배선)는 완결성 위반 — wire-or-drop.**

**tier 경계와의 화해 (ADR-20260708 보존).** 통합/E2E를 완료 게이트에 다 넣는 게 아니다. 미배선은 대부분 *reachability 문제*라 값싸게 잡힌다 — 정적(라이브 진입점이 이 심볼을 참조하나 = orphan 아님) + smoke(개선이 실제 진입점을 통해 효과를 내나). **liveness-floor(도달성+smoke)는 autopilot 게이트 안에, 전체 행위 E2E는 push-gate/CI에.** "켜졌나"의 floor는 게이트가, "완전히 옳게 동작하나"는 상위 tier가.

**연결.** 미배선 orphan은 §5.4 미결 잔여의 한 형태(존재하나 결정 안 된 dead code)이고, "배선했다"는 transcript 주장 ≠ 라이브 경로로 증명된 배선(§5.2). 이 floor는 §3.3#1(완료=모든 AC evidence pass)의 "evidence"를 *라이브 evidence*로 정밀화한 것.

### 5.9 목표 변경 — re-lock ≠ drift (자율이 루프에 기댈 수밖에 없는 이유)

자율/오케스트레이션은 결국 *규율 있는* 루프 엔지니어링에 기댄다. **회의는 옳았다 — 순진한 버전에 대해.** 여기서 기대는 건 vibe 루프가 아니라 §5.4~5.8에서 규율화한 루프이고, 연구도 확인했다: 진지한 loop engineering = 이 설계(§4.2). loop engineering을 부정한 게 아니라, 부정할 부분과 기댈 부분의 선을 그은 것이다.

**목표 변경이 루프가 필요한 가장 깊은 이유.** 고정 파이프라인은 *완전히 명세된 고정 목표*를 전제하는데 현실은 안 준다(설계 맹점·§5.6 막다른 길). 그래서 재타겟 가능한 반복이 필요하다. golden rule의 가장 깊은 형태: 한 턴의 결과가 다음 *step*이 아니라 *target*을 바꾼다 — 계획으론 못 흡수하고 루프로만 흡수된다.

**결정적 구분 — 목표 변경 ≠ 목표 축소.**
- **목표 축소(§5.8-A, 금지)** = 에이전트가 일방적으로 목표를 좁혀 진행, 숨김. silent·self-owned·미공개.
- **목표 변경/re-lock(합법)** = 변경 필요를 증거·근거와 함께 surfacing, 올바른 주체가 결정, 새 목표를 명시적으로 재잠금 → 새 no-shrink 기준. explicit·올바른 소유·공개.

같은 척추: explicit+evidence+올바른 소유+공개 vs silent+자기편의+은닉. 루프는 변경 *필요를 탐지*하고, 변경 *자체는 owned re-lock*이지 결코 silent 재타겟이 아니다.

**라우팅 (§4-10 method vs intent, 이미 DITTO DNA).**
- **method 변경**(목표 그대로, 방법이 바뀜) = 에이전트-owned 반복 = 연구의 **contract refinement**(Stripe: 계약이 놓친 걸 발견하면 계약을 정련). 사용자 불필요.
- **intent 변경**(목표 자체가 바뀜) = 사용자-owned 에스컬레이션 = §3 stop 조건 "정초 방향 뒤집힘" = §5.4 exit 3.

**method/intent 분류 게이트 (no-shrink의 method-경유 세탁 차단).** 이 라우팅은 남용 표면을 하나 연다: 에이전트가 실은 목표를 좁히면서 그걸 "method refinement"로 분류하면, 현재 의도가 조용히 재잠금되고 이전 in-scope 잔여가 new-scope로 이탈한다 — §5.8-A silent 축소의 절차적 우회다. 그래서 분류에 oracle을 건다: **method 변경은 *기존 모든 AC의 관측 결과·라이브-경로 기대를 보존할 때만* 에이전트-owned다.** AC의 의미나 적용(라이브-경로) 기대가 바뀌면 그것은 method가 아니라 intent 변경이고, 사용자 에스컬레이션으로 승격한다. 분류가 "이건 method다"라는 자기선언이 아니라 보존 여부의 관측으로 판정돼야 우회가 닫힌다.

**intent-lock과의 화해 (잠재 긴장 해소).** intent-lock(고정 목표) + 목표-변경-가능은 모순이 아니다: intent-lock은 *드라이브 동안* 목표를 고정하고, 변경 필요가 발견되면 *re-lock*(새 고정 목표)을 촉발한다. 루프는 "잠긴 목표로 드라이브 ↔ 잠금이 바뀌어야 함을 에스컬레이션" 사이를 오갈 뿐 목표를 흘리지(drift) 않는다.

**소유 루프의 또 다른 근거.** 네이티브 /goal은 transcript에 진술된 *고정* 목표로 판정한다 → 목표가 바뀌면 완료 기준 자체가 바뀌어야 하는데 /goal은 못 한다. re-lock을 다루려면 drive 루프를 우리가 소유해야 한다(§5.1 보강).

### 5.10 제어 구조 — 오케스트레이션 ⊂ drive-loop, 완수 = disposition-queue drain

§5.1~5.9가 하나의 구체적 제어 구조로 합쳐진다 (2층 + escape). 이 구조는 §6 네이티브 dynamic-workflow의 loop-until-dry에 얹히는 것을 *목표*로 한다(SDK 불필요를 지향 — 단 이 실현 가능성은 확정이 아니라 이 절 말미 "네이티브 실현"에서 밝히듯 §6 스파이크로 확정할 가설이다).

**Layer 1 — 오케스트레이션 (inner, acting).** 의도 구현·후속 처리를 gated 파이프라인(intent-lock → plan → implement → live-verify)으로 밟는다. 한 단위를 규율 있게 수행하는 층. drive-loop가 호출(§5.1 위임). = 네이티브 subagent 팬아웃.

**Layer 2 — drive-loop (outer, 완수 판정).** 완수 = **현재 의도의 disposition-queue가 비었나**. 매 반복: pending 항목 → 오케스트레이션으로 처분 → 관측 → queue 갱신 → 빌 때까지(loop-until-dry). = 네이티브 workflow 루프.

**완수의 정확한 정의 — 불동점(fixed point).** "더 이상 처리할 작업 없음" = *현재 의도의* queue drain(전역 백로그가 아님 — 그건 안 끝난다). 한 오케스트레이션(= 그 의도의 AC 전부를 증거로 완수하는 한 판)을 돌면 후속·잔여·버그가 나오고, 그걸 다시 오케스트레이션으로 처리하면 또 나올 수 있다 — **더 이상 새 항목이 나오지 않는 불동점에 닿으면 완료다.** 이 종료 조건 자체는 결정론적이다: "큐가 비었나(모든 항목이 처분됐나)"는 LLM 판정이 아니라 카운트다. queue 항목: 발견된 결함·in-scope 잔여·unverified AC. queue를 떠나는 합법적 문 셋(§5.4): resolved(라이브 증거) / 진짜 new-scope deferral(백로그로, 기록·§5.5) / escape(에스컬레이션·re-lock). new-scope는 현재 완수를 막지 않되 기록돼 안 잊힌다. (불동점의 *존재*는 보장되지 않는다 — 발산은 위 "종료성" 항이 예산 상한으로 막는다.)

**종료성 — capture window를 닫는다 (생산적 발산 방지).** fail-closed 완결성의 감춰진 위험: queue가 "발견된 결함·in-scope 잔여·unverified AC"를 무제한으로 받아들이면, 항목을 하나 해결할 때마다 하나를 더 발견하는 *생산적 발산*에서 drain이 영영 안 끝난다 — 이건 "K회 진전 0" 정체(§ escape emergent)가 아니라 매 반복이 진전하는데도 총량이 줄지 않는 경우라 escape도 안 걸린다. 재구축 동기인 3시간+ 문제(§2.1)를 오히려 재생산한다. 그래서 발견의 capture window를 닫는다:
- **intent-lock이 *현재 의도 AC의 닫힌 집합*을 정의한다.** 완수 판정의 분모는 이 집합이지 "지금까지 발견된 전부"가 아니다.
- **intent-lock 이후 발견 항목은 current-scope / new-scope / blocking으로 독립 분류**(§5.4 disposition, per-item 독립 판정)하고, in-scope 재귀 수집은 **depth·변경면·시간 budget**으로 제한한다. 이 budget을 넘긴 in-scope 발견은 무한 재귀로 삼키지 않고 non-pass handoff로 종료한다(미결로 얼어붙는 게 아니라 *경계 지어 인계* — §5.5 park·legible 종료).
- current-scope 발견만 현재 완수를 막고, new-scope는 백로그로(기록), blocking은 escape로 간다.

**Escape — 루프로 못 푸는 것 (redesign-needed).** 어떤 후속/버그는 재설계가 필요해 반복이 무의미하다. 두 탐지 경로:
- **upfront** — 항목이 intent/설계/ADR/비가역을 건드림이 분명 → *즉시* 에스컬레이션(§4-10·§3·§5.9), 루프 낭비 안 함.
- **emergent** — loop-resolvable로 보였으나 *수렴 실패* → 비수렴이 곧 "재설계 필요"의 증거 → escape로 승격. doom-loop 방지. (DITTO 이력의 cap-model·doomed→blocked·no-op 데드락이 이 자리.) 비수렴은 두 형태를 다 발화해야 한다: **정체**(K회 반복 진전 0) *그리고* **생산적 발산**(항목마다 진전하나 윈도우 내 queue 크기가 단조 감소에 실패 — 새 발견이 해결을 따라잡거나 앞선다). 후자는 진전 카운터만으로는 안 잡히므로 queue 크기의 추세를 함께 본다.

escape의 "별도 처리"는 새 메커니즘이 아니라 §5.4 exit-3 + §5.7 park + §5.9 re-lock다. 구조가 닫혀 있다.

**작동 4조건 (모두 §5에서 확립).**
1. **queue-empty 체크는 증거 기반, 그리고 그 소유자는 outer-loop다** — "비었다"는 transcript 주장 ≠ 증명된 drain(§5.2). hard invariant(queue-empty 게이트)의 실제 소유자는 **CLI persistent outer-loop**(§6.3)이고, in-session Stop hook(§6.1)은 per-turn 보조 게이트로 격하한다 — Stop hook은 무진전 상한에서 override될 수 있어(§6.1·§6.5 [미검증]) 그것만으로는 hard gate가 못 된다. 그리고 queue mutation은 **구조 경계에서만 읽는다**: subagent의 자유텍스트 반환을 queue oracle로 쓰지 않고(그건 §2.1 손-JSON 파싱 병의 재발이다), top-level `--json-schema` 출력 또는 file-sidecar schema로만 queue를 갱신한다.
2. **남는 판정 표면("빼기")은 교차모델 적대 심의로** — in-scope 잔여·버그는 라벨링이 아니라 처리로 가므로(§5.4), 게임 표면은 "진짜 새 범위라며 백로그로 빼기" 하나로 좁혀진다. 그 하나와 "정말 종료해도 되나"의 종료 진단·우선순위 조정은 *다른 provider의 Opponent*를 세운 dialectic으로 판정한다(자기채점보다 강한 독립성, §4.2). 비용이 있으므로 매 오케스트레이션마다가 아니라 **종료 진단이 필요한 시점**(불동점 근처)과 우선순위 재조정이 필요할 때만 돌린다. queue-drain은 이 "빼기" 판정만큼만 정직하다.
3. **escape는 false-escape ⊥ false-persist 둘 다 방지** — 진전 측정으로 조기 포기(abdication)와 doom-loop를 동시에 막는다.
4. **완수는 현재 의도 scope** — new-scope는 백로그로(기록), 현재 완수 불방해.

**네이티브 실현 (확정이 아니라 스파이크-게이트 가설).** 의도한 매핑: Layer 2 = CLI outer-loop가 소유하는 loop-until-dry, Layer 1 = subagent 팬아웃, queue-empty 체크 = outer-loop가 구조 경계(§6.4 `--json-schema`)에서 읽는 증거 게이트(Stop hook은 per-turn 보조). **이것이 "SDK 없이 §6 네이티브 표면에 얹힌다"는 것은 확정 사실이 아니라 §6의 [미검증] 항목에 걸린 가설이다** — 특히 (a) Stop hook 무진전 상한과 (b) native subagent 반환이 스키마가 아닌 자유텍스트라는 두 갭(§6.5). 착수 전 스파이크(§10-3)가 이 매핑을 실측으로 확정하기 전까지, 이 절의 "SDK 불필요"는 목표이자 검증 대상이지 이미 증명된 성질이 아니다.

### 5.11 이 구조가 달성하는 것 — 그리고 정직한 경계

2층 구조는 두 목표를 각각의 *올바른 메커니즘*으로 달성한다. 층 배정: **orchestration → 코드베이스 건강, loop → 진정한 완료.** 그리고 이는 우연이 아니라 **§4.3 경계(oracle 있는 곳=루프 강함, 얇은 곳=약함)를 아키텍처로 실현**한 것이다.

- **완료(loop) — oracle 있는 영역.** drive-loop + 완결성(§5.4~5.10)이 "제대로 끝났다 할 수 없는 지점" 병을 구조로 닫는다. 완료 *판정*은 낮추는 게 아니라 **불동점(빈 큐 + 전부 처분됨)으로 결정론적으로** 조인다(§5.10). 잔차는 "완료 판정"이 아니라 **"큐 포착의 완전성"**에 있다 — AC가 애초에 의도 전체를 담았나. AC가 처음부터 좁게 잡혔으면 안 담긴 부분은 큐에 아예 안 들어오고, 그럼 불동점에 닿아도 그 부분은 미완인 채다. **이 완전성은 사용자만 정확히 판단할 수 있다(애초에 자동으론 불가능).** 그래서 DITTO는 이걸 없애는 척하지 않고 양끝에서 *할 수 있는 최대한만* 방어한다 — 착수 전 front-load(deep-interview로 의도 소진·pre-mortem으로 실패 예측, §5.8)로 큐를 최대한 완전하게 채우고, 완료 시점 교차모델 dialectic(§5.10)로 놓친 것을 적대 검토한다. 경계: 이것도 "설계로 닫았다"이지 실증 아님 — 현행도 게이트가 gamed됐으므로(§2.2), non-self 검증(§8.3) 전엔 그 이상 주장 못 함.
- **건강(orchestration) — 부분적으로 oracle 없는 영역.** 엄밀한 오케스트레이션이 건강의 *체크 가능한* 부분을 배달한다: no-orphan·surgical·Tidy·minimal-increment·live-verified·dialectic 압력. 그러나 §4.3대로 건강의 핵심 일부(추상화 수준·개념적 무결성·시간에 걸친 rot)는 oracle이 없다 → **구조로 방어되나 보장되지 않음.** 그 잔차는 인간 판단 + 시간축 구조 리뷰(§4.3 row4)에 기댄다.

**피해야 할 트랩 — 완료 ≠ 건강.** 완전한 코드베이스도 건강하지 않을 수 있다(모두 realized·wired·verified인데 설계가 나쁨). 완료(체크 가능, 루프 배달)를 건강(판단, 완전 배달 불가)으로 오인하면, "체크 가능한 프록시를 검증 불가한 target으로 착각"하는 Goodhart 오류를 *우리 아키텍처 메타 층위에서* 반복하는 것. 둘을 구분해 유지하는 것 자체가 정직 규율(§5.2·unverified 공개).

**정직한 총평.** 그렇다 — 두 목표를 각각 올바른 메커니즘으로 *달성 가능한 최대치*로 달성한다. 이보다 더 주장하면(예: "건강을 *보장*한다") 그게 그럴듯한 거짓말이다. DITTO의 약속은 "풀었다"가 아니라 "증거가 허용하는 데까지 방어하고 잔차를 공개한다"이다. 그 약속 안에서 이 설계는 두 목표를 지금 얻을 수 있는 가장 강한 형태로 얻는다.

### 5.12 컨텍스트 절약·무손실 핸드오프 = 제공 기능 (개발 규율이 아님)

완결성 축과 나란히 서는 또 하나의 제공 기능. **context rot 관리와 무손실 핸드오프는 도그푸딩 작업 규율이 아니라 DITTO가 임의 사용자 프로젝트에 제공하는 기능이다**(§5 적용 범위와 같은 궤). 이유는 §2.1 증상의 뿌리에 있다 — 매 턴 대량 컨텍스트 주입이 압력을 올려 antml 오류를 악화시켰고, 순차 스폰이 컨텍스트를 계속 부풀려 3시간 monolith를 낳았다. 즉 context rot은 *이 프로젝트의 사고*가 아니라 임의 코딩 에이전트 작업의 신뢰성을 갉는 보편 병이고, DITTO가 팔아야 할 것이 바로 그 병의 관리다.

**축 A — 메인 에이전트 = 컨트롤 타워.** 사용자와 대화하는 main agent는 각 동작의 *계약(목표·완료 기준·반환 형식)* 만 보고 오케스트레이션한다. 탐색·조사·벌크 읽기/편집/비교는 메인이 직접 삼키지 않고 **격리된 subagent에 위임**하고, 중간 산출물은 subagent 컨텍스트에 가두고 **결론·증거·불확실성만 회수**한다(§3.1 철학#9의 제품화). 검증·리뷰는 fresh context에서 — 이건 효율이 아니라 유효성 조건이고, §5.2에서 본 reward hacking의 구조적 방어와 같은 뿌리다(공유 컨텍스트가 인과적 원인). **낮은 임계에서 능동적으로 위임한다** — 컨텍스트가 한도에 닿기 훨씬 전(예: 창의 30% 수준)부터 rot은 비균일하게 성능을 갉으므로, "가득 차면 압축"이 아니라 "차기 전에 위임·리셋"이 기본값이다. 큰 파일을 메인이 통째로 여러 번 읽고 직접 편집하는 것이 전형적 안티패턴.

**축 B — 무손실·자연스러운 핸드오프.** 컨트롤 타워는 위임과 컨텍스트 리셋이 잦다는 뜻이고, 그럴수록 **경계 인계의 무손실성이 전체 신뢰성을 좌우**한다. 핸드오프(§3.2 핵심 개념)는 세션·에이전트 경계에서 **원 의도를 축자 보존해 운반**하고(재도출·드리프트·조용한 축소 금지 — §3.1 철학#12·§4-12), 그것을 실현하는 결정·현재 상태·근거를 함께 전달한다. 무손실이 정확성 조건이라면, **UX/AX는 사용성 조건**이다: 재개 시 핸드오프가 *발견되고 소비되는 흐름*이 자연스러워야 한다(자동 주입이 아니라 명시적 pull — 리셋한 새 컨텍스트를 다시 오염시키지 않으면서 필요한 것만 끌어온다). 무손실이되 쓰기 불편한 핸드오프는 실제로는 안 쓰이고, 그럼 축 A가 무너진다.

**네이티브 실현.** 두 축의 실행 기반은 §6에 이미 있다: 축 A = §6.2 subagent 격리 팬아웃(중간 툴 호출이 부모에 안 쌓임) + §6.3 CLI `--resume` outer-loop(노드마다 fresh context 리셋). 축 B = 핸드오프 산출물(pull 방식). 재구축은 이 둘을 *제품 기능*으로 승격해 임의 사용자 프로젝트에서 작동하게 한다 — 도그푸딩 스택에 굳히지 않는다(§3.3#11).

**북극성과의 관계.** context rot은 §5.3 순효능의 *시간·신뢰성 두 축을 동시에* 갉는다 — 3시간 monolith(시간)와 컨텍스트 압력발 antml 오류(신뢰성)가 §2.1의 실측이다. 그래서 이 축은 완결성(§5.4)과 함께 순효능 북극성을 *실행 층위에서* 떠받치는 두 기둥이다: 완결성이 "미결로 끝나지 않게" 지킨다면, 이 축은 "부풀어 느려지고 오류나지 않게" 지킨다.

## 6. Claude Code 네이티브 표면 도입안

> **근거**: 아래는 공식 문서 기반 조사 결과(URL 근거 포함). 착수 전 스파이크에서 우리 실제 CC 버전으로 실측·확정할 항목은 `[미검증]`으로 표시.

### 6.1 증거기반 agent Stop hook — drive 루프 소유

- **Stop hook = 하네스가 소유하는 완료 게이트.** Claude가 턴을 끝내려 할 때 hook이 "아직 완료 아님"을 판정하면 종료를 block하고, 그 reason이 Claude에게 되먹여져 세션이 계속 이어진다(block-stop 재구동). exit code 2 또는 `{"decision":"block","reason":...}`.
- **agent-based Stop hook**은 서브에이전트를 spawn해 **파일 읽기·명령 실행**(예: 테스트 통과 확인)까지 하고 종료 허용 여부를 판정한다. 이것이 §5.2의 "증거 판정".
- **왜 `/goal`을 쓰지 않나**: `/goal`은 세션-스코프 prompt-based Stop hook의 공식 래퍼로, 완료 판정을 fresh 모델(기본 Haiku)에 넘기는 좋은 성질이 있으나 — 명령을 실행하지 못하고 transcript만 읽는다(§5.2 덫). 증거 게이트엔 부적합. 단, "fresh 모델이 판정"이라는 설계 원리 자체는 우리 §7-4(fresh context 검증)와 정합하므로 참고한다.
- **[미검증] 상한**: Stop hook은 무진전 8회 연속 block 시 Claude Code가 override하는 캡이 있다. 즉 in-session 재구동은 무한이 아니다 → 긴 자율 실행은 §6.3 CLI outer-loop가 받아야 한다.

### 6.2 네이티브 subagent 팬아웃 — 오케스트레이션 위임

- 서브에이전트는 **완전 격리 컨텍스트**(고유 컨텍스트 윈도우·시스템 프롬프트·툴·권한)에서 실행되며 부모의 대화 이력·이미 읽은 파일을 보지 못한다(예외: fork). 중간 툴 호출·결과는 서브에이전트 안에 남고 **최종 메시지만 부모로** 넘어와 벌크 작업이 부모 컨텍스트에 안 쌓인다.
- **병렬 실행** 가능, 각기 다른 권한. 독립 서브태스크가 "가장 느린 하나의 시간"에 끝난다 → 순차 3시간의 상당 부분을 병렬로 접는다.

### 6.3 CLI outer-loop + context 리셋 — 3시간·rot 해소

- `--output-format json`으로 session_id 캡처 → `--resume`로 재호출하는 **CLI 프로세스 레벨 루프**를 하네스가 소유한다. 노드마다 **fresh context로 리셋**되니, 컨텍스트가 계속 부푸는 3시간 모놀리스 한 판이 아니라 *재개 가능한 짧은 fresh 단계들의 사슬*이 된다. §7-4·§7-5(context rot)를 구조로 해소.

### 6.4 `--json-schema` 경계 출력 — JSON 오류 해소

- top-level CLI 출력은 headless `--json-schema`로 스키마 강제(structured_output 필드)가 된다. outer-loop가 CLI 경계에서 받는 구조 데이터를 스키마로 받으면, 서브에이전트가 stdout에 손 JSON을 찍고 파싱하던 경로(§2.1 JSON 오류)가 사라진다.

### 6.5 남는 결정적 갭 2개 + 우회

1. **네이티브 마크다운 서브에이전트 반환은 스키마 검증이 아니라 자유 텍스트 요약**이다. 스키마 강제 반환은 SDK(query outputFormat)에서만 제공. → **우회**: (a) 부모가 자유 텍스트를 파싱(DITTO가 이미 하던 부모측 검증), 또는 (b) 구조가 꼭 필요한 경계는 CLI `--json-schema`로 받는다.
2. **Stop hook 8회 무진전 상한**(§6.1) → in-session 재구동만으로는 긴 작업을 못 몬다. → **우회**: CLI outer-loop(§6.3)가 진짜 backbone, Stop hook은 per-turn 게이트.

### 6.6 요약 — 증상 → 네이티브 기전

| 증상(§2.1) | 근원 | 네이티브 기전 |
|---|---|---|
| autopilot 3시간+ | 순차 스폰 + 손 오케스트레이션 | subagent 병렬 팬아웃 + CLI resume 단계화 |
| antml 문법 오류 | 대량 컨텍스트 주입 압력 | 주입 감량(얇은 게이트 코어) |
| 손 JSON 파싱 오류 | stdout 손 JSON | `--json-schema` 경계 출력 |
| 조기 종료(효능 손실) | 모델이 완료 자판정 | 증거기반 agent Stop hook |
| dual-host 낭비 | 미검증 2호스트 유지 | single-host seam(§7.4) |

**결론: SDK 없이, Claude Code 확장점(스킬·서브에이전트·훅·슬래시커맨드) "위에 얹히는" 방식만으로 "게이트는 소유하고 오케스트레이션은 위임하는" 얇은 하네스가 가능하다.**

```
│  drive 루프 (소유, 얇게)                                  │
│   - CLI outer-loop: --resume 로 단계 사슬, 노드마다 context 리셋 │
│   - Stop hook: per-turn 완료 게이트 (증거 판정)            │
│   - 종료 = 우리 게이트 통과. 모델의 "done" 주장은 거부·재구동  │
└───────────────┬─────────────────────────────────────────┘
                │ 위임
┌───────────────▼─────────────────────────────────────────┐
│  네이티브 위임층 (Claude Code)                            │
│   - subagent 병렬 팬아웃 (격리 컨텍스트)                   │
│   - dynamic workflow (loop-until 오케스트레이션)          │
│   - --json-schema 경계 출력                               │
└───────────────┬─────────────────────────────────────────┘
                │ 게이트가 감싼다
┌───────────────▼─────────────────────────────────────────┐
│  얇은 게이트 코어 (소유, §7 불변식만)                      │
│   - completion contract / verdict / unverified            │
│   - fail-closed 게이트 / 무축소 / tier 격리               │
│   - 스키마 = SoT                                          │
└─────────────────────────────────────────────────────────┘
```

**이 3층(구성요소 소유 뷰)은 §5.10의 2층 제어구조(제어 흐름 뷰)를 실현한다** — 두 뷰는 직교하고 아래처럼 대응한다:

- **네이티브 위임층 = §5.10 Layer 1 (inner, 오케스트레이션·acting)**: 한 단위를 gated 파이프라인(intent-lock → plan → implement → live-verify)으로 규율 있게 수행하는 층. subagent 병렬 팬아웃·구조 반환·컨텍스트 격리. drive 루프가 호출(§5.1 위임). 우리가 다시 구현하지 않는다.
- **drive 루프 = §5.10 Layer 2 (outer, 완수 판정)**: 소유하되 얇게. 완수 = *현재 의도의* disposition-queue drain(불동점) — "게이트→재구동→정지"의 그 게이트가 바로 **queue-empty 게이트**이고, 그 소유자는 CLI outer-loop다(§5.10 작동조건1: in-session Stop hook은 per-turn 보조). queue는 **구조 경계(§6.4 `--json-schema`)에서만 읽는다** — subagent 자유텍스트를 queue oracle로 쓰지 않는다(§2.1 손-JSON 병 재발 방지). 모델의 "done" 주장은 거부하고, 매 반복은 **재구동 또는 escape**(수렴 실패·비가역 → 에스컬레이션·re-lock, §5.10)로 간다. 그 이상은 아님(ready-node·dispatch·그래프를 여기 다시 얹지 않는다, §11-3).
- **얇은 게이트 코어**: §7 불변식과 완료 계약·상태만. 오케스트레이션 엔진(노드 그래프·dispatch·ready-node 선택)은 여기 없다 — 그건 위임층이 준다. 이 코어가 제공하는 **판정 술어(completion contract·verdict·unverified)가 곧 queue-empty 게이트가 "각 항목이 처분됐나"를 읽는 근거**다(§5.10 queue를 떠나는 문 셋: resolved/new-scope deferral/escape). 이 코어가 다시 두꺼워지면 3시간 엔진이 되살아난다(§11-2 반-패턴).

즉 §5.10 종료성(capture window·budget 상한)과 escape(§5.4 exit-3·§5.7 park·§5.9 re-lock)는 새 구성요소가 아니라 이 drive 루프가 queue를 갱신·종료하는 규칙으로 얹힌다 — 다이어그램에 상자를 더 그리지 않는다.

### 7.4 seam 원칙 — 단일 host 어댑터

§7-7(호스트 절연)은 재구축에서 "네이티브 Claude Code = 그 하나의 host 어댑터"로 실현된다. single-host-first이므로 seam 뒤 구현은 하나뿐이다. 나중에 다른 host가 필요하면 seam 뒤에 두 번째 어댑터를 *가산*으로 추가(ADR-0008 패턴). **seam을 무너뜨려 Claude Code를 코어에 baked-in 하지 않는다** — seam은 (a) 철학이 harness에 종속되지 않게 하고, (b) 코어를 live model 없이 단위 테스트 가능하게 한다.

## 8. 증거 감사 방법론 — zero-start를 어떻게 채우나

zero-start는 "아무것도 없음"이 아니라 "§7 불변식만 지고, 나머지는 자격을 다시 벌게 함"이다. 자격 심사 = 증거 감사.

### 8.1 세 바구니 잣대

1. **원칙으로 유지 (moat)** — §3.1 철학 + §3.3 불변식. 빼면 DITTO가 아니게 됨. 발화 빈도로 심사하지 않음 — 각자 *명명된 실패 모드*를 지키니까.
2. **증거로 유지** — 도그푸딩 이력(git·memory·work-item)에 실제로 결함을 잡았거나·반복 사용됐거나·실제 의도손실을 막은 흔적이 있는 것. 단 **monoculture 할인**(§8.2).
3. **제거 (증명 전엔 미검증)** — 나머지 전부. 저자 기대로 지었지만 keep을 벌었다는 기록이 없는 것. 포팅하지 않고, *재현된 구체적 필요*가 생길 때만 재진입.

**함정 방지(carve-out)**: 자주 안 터져도 *명명된 재앙적 실패 모드*를 지키는 싼 보험(예: 보안·비가역 게이트)은 유지한다. 단 부담은 그 실패 모드를 구체적으로 명명하는 것 — "언젠가 쓸모 있을지 모름"은 제거.

### 8.2 도그푸딩 이력 = 증거 코퍼스, 단 할인

도그푸딩 이력 자체가 증거다(엔진 self-audit의 "게이트 16개 × 실측 catch-rate → 제거 대상 0" 같은 측정이 이미 존재). 그러나 §2.2 monoculture·자기채점 진단 때문에 이 증거는 **할인**된다: catch-rate가 "DITTO 위에서 터졌다"는 뜻이지 "임의 프로젝트에서 keep을 번다"는 뜻이 아니다.

### 8.3 non-self 검증 의무

그래서 재진입 심사는 self가 아닌 신호를 요구한다: 최소 하나의 **비-DITTO fixture 프로젝트**(다른 언어·스택 포함)에서 그 기능이 실제로 keep을 버는지. §7-11(메타-도구는 사용자 환경에 맞춘다)을 감사 기준으로 승격.

**순효능 측정 프로토콜 — 3-arm ablation.** §5.3 순효능은 self에서 성립 불가(baseline을 못 만듦)이므로 non-self fixture에서만 재고, 이진 A/B가 아니라 최소 세 arm으로 잰다 — hook+charter가 맨 CC와 풀 DITTO 사이의 중간층이라, 효능의 출처를 갈라야 하기 때문이다:

- **B0** — 맨 Claude Code (지침 없음).
- **B1** — CC + charter 주입만 (hook 층 = *값싼* DITTO).
- **A** — 풀 DITTO (엔진·경로·완료 게이트까지 = *비싼* DITTO).

**A vs B1이 핵심 대조다** — "무거운 엔진이 얇은 charter 주입 대비 값을 하나"를 잰다. A ≈ B1이면 엔진은 순효능 없이 비용만 낸다는 뜻이고, 그건 zero-start·"얇게" 명제를 *실증으로* 지지(또는 반박)한다 — 즉 이 ablation은 측정일 뿐 아니라 재구축의 자기정당화 실험이다. hook은 음(−)의 기여도 한다(false-positive 차단·오버헤드가 "DITTO on"의 비용에 포함 — 이 프로젝트에서 실제로 반복된 실패 모드). 측정 야망은 통계 증명이 아니라 **A가 B1을 이기는 net-positive 사례를 non-self에서 하나라도 실증**으로 낮춘다(§4.3 정신 — 모든 걸 잡는 게 아니라 안 잡은 걸 잡았다 안 하는). 지표는 §5.3대로 신뢰성 AND 시간(완료 턴/시간·재작업 횟수·최종 정확성·B가 새어보낸 실패의 A 차단율).

**fixture = palimpsest (확정).** 다른 스택의 실프로젝트 — Python 3.11 / pytest, tree-sitter 다언어 추출 → Neo4j 지식그래프(src ~11.5K LOC). bun/TS monoculture를 벗어난다(§2.2). 측정 범위는 **per-task 대조**로 못박는다: "이 repo에서 DITTO가 값을 했나"가 아니라 "**한 작업을 맨 CC 대비 DITTO 엔진이 더 잘 완수하나**"를 잰다. 유효성 근거 — palimpsest에는 charter 자동주입 hook이 없어(`.claude/settings.json`은 CLI allow 권한만) self에서 baseline을 못 만들게 하던 가장 날카로운 오염(§5.3, 매 프롬프트 주입)이 부재해 B0(맨 CC, DITTO 표면 무시)를 물리적으로 구성할 수 있고, 애플리케이션 소스(`src/`)는 DITTO와 사실상 무관하며, DITTO가 만든 과거 이력은 세 arm에 동일하게 깔리므로 A-vs-B1 대조를 편향시키지 않는다. **한계(정직히 명시)**: palimpsest 자체가 DITTO로 구동돼 온 프로젝트라 "DITTO가 한 번도 닿지 않은 완전 중립 repo"는 아니다 — per-task 대조에는 무해하나, 프로젝트 수준 순효능 주장으로 확대 해석하지 않는다. 첫 측정 작업 후보 = recognizer가 `const fetch = …` 지역 바인딩 shadow를 확인하지 않아 false-positive를 내는 재현형 버그(`src/palimpsest/extract/ecmascript.py`, 대상·수정방향 특정됨) — 작고 자기완결적이라 3-arm에 적합.

- **완료 판정 vs 큐 포착 완전성 (§5.4·§5.10·§5.11).** 완료 판정은 *낮추지 않고* 불동점(빈 큐 + 전부 처분)으로 결정론적으로 조인다. 자기채점 병은 판정 표면 최소화(in-scope 잔여·버그는 라벨링 아닌 처리 → 게임 표면은 "새 범위로 빼기" 하나)로 뿌리를 뽑고, 남는 그 하나는 교차모델 dialectic로 판정한다. 잔차는 "완료 판정"이 아니라 **"AC가 애초에 의도 전체였나(큐 포착 완전성)"**로 이동 — 이건 사용자만 정확히 판단 가능하고 자동으론 애초에 불가능하므로, 착수 전 front-load(deep-interview·pre-mortem)와 완료 시점 dialectic로 *할 수 있는 최대한만* 방어하고 남는 것은 사용자 권위로 정직히 남긴다.
- **park cull = delist + archive (§5.7).** 폐기는 작업 삭제가 아니라 active/main에서 delist + immutable archive 보존. 물리 삭제는 명시 승인/deterministic supersede만. 자율 모드 무응답은 삭제가 아니라 delist.
- **AC application facet 작성-검증 분리 (§5.8).** `planned_application_oracle`(작성필수: 반증 가능한 확인-방법 명세) ⊥ `observed_application_evidence`(검증필수: live-path pass). 작성 게이트는 진위가 아니라 명세의 존재+반증가능성만 검사.

> 위 셋은 이번 조율로 §5 텍스트에 반영됐다. §5는 이제 ADR 잠금 후보다. 남는 근본 잔차 하나 — "AC가 의도 전체를 담았나"의 최종 판단은 사용자 권위이며 §8 non-self 검증 전엔 실증 아닌 설계 방어 — 는 없앨 수 없는 것으로 §5.11에 정직히 명시(§2.2 자기채점 재발 방지).

## 10. 착수 순서 (roadmap)

1. **§7 불변식을 먼저 스키마/타입/게이트로 못박는다** — 아키텍처의 뼈대. (완료 계약·verdict·unverified·fail-closed.)
2. **네이티브 seam을 정의한다** — Stop hook 완료 게이트 + CLI resume outer-loop + subagent 팬아웃 + `--json-schema` 경계. 여기서 하네스 종속을 seam 하나로 격리.
3. **스파이크로 실측** — 완료 게이트 한 조각을 네이티브로 얇게 구현해, 현행 outer-in 경로 대비 지연·신뢰성을 나란히 측정(§6의 `[미검증]` 항목 확정). 증거로 아키텍처를 결정.
4. **증거 감사(§8)** — 세 바구니 분류, non-self fixture 도입. **현행 백로그 처분(§8.4) 반영**: 유효 3건(#48 대조 벤치마크·#50 외부 피드백 루프·#49 오버헤드 실측)이 non-self 검증(§8.3)·순효능 실측(§5.3)의 실행 항목으로 여기 접힌다. 흡수 1건(#47)은 §5.10에 이미 있고, 소멸 2건(#45·#9)은 zero-start가 처리.
5. **증분 재진입** — 감사가 keep 판정한 기능만, 순효능 북극성을 후퇴시키지 않는 선에서 하나씩.

**부트스트랩**: 현행 DITTO가 이 재구축을 몬다(self-hosting). 새 구현이 self-host(작업 하나를 end-to-end로 구동) 가능해지면 main을 뒤집고 낡은 src를 Tidy 커밋으로 은퇴.

## 11. 위험과 반-패턴 — 재구축이 재생산할 수 있는 병

1. **monoculture 재발.** 재구축도 도그푸딩으로 부트스트랩 → 같은 self-confirming 함정. → *방어*: §5.3 순효능 북극성 + §8.3 non-self 검증을 구조에 박음.
2. **게이트 과축적.** 얇은 게이트 코어가 다시 두꺼워짐 → 3시간 엔진·곱셈 신뢰성 붕괴 재현. → *방어*: 코어는 §7 불변식만. 새 게이트는 §8 감사(명명된 실패 모드)를 통과해야 진입.
3. **drive 루프 비대.** 소유 루프에 ready-node·dispatch·그래프를 다시 얹음. → *방어*: 소유는 "종료 판단"에만, 나머지는 네이티브.
4. **efficacy 미측정 회귀.** 내부 게이트만 보고 최상위 지표를 안 봄(현행의 근본 결함). → *방어*: 순효능을 최상위 AC로, 매 증분 게이트.
5. **seam 붕괴.** 편의로 Claude Code를 코어에 baked-in → 철학이 harness에 종속 + 단위 테스트 불가. → *방어*: §7.4 seam 원칙.

**근본 한계 (아키텍처로 없앨 수 없는 것, 정직히 명시).** 위 다섯은 방어 가능한 자초 함정이지만, 이건 방어 불가능한 천장이다: **증거 게이트의 힘은 좋은 oracle이 존재하는 곳에서만 나온다.** verifier가 불신뢰한 어려운 문제에선 검증이 생성만큼(때론 그보다 더) 어려워진다(§4.3, arXiv 2606.26300). DITTO는 이 천장을 없애는 척하지 않는다 — oracle이 명확한 정확성 차원에 루프를 쓰고, oracle이 없는 설계·의도 차원은 생성 조형 + 인간에 맡기며, 게이트가 못 덮은 잔차는 `unverified`로 *드러낸다*. "모든 걸 잡는 게이트"를 약속하지 않는 것이 이 한계에 대한 유일하게 정직한 응답이다.

## 12. 부록 — 이 문서에서 처음 도입한 용어

- **효능(efficacy) / 순효능** — 도구를 쓰는 것이 안 쓰는 것(조심해서 그냥 씀)보다 나은 정도. 신뢰성과 시간을 함께 본다. 재구축의 최상위 지표.
- **monoculture 증거** — 하나의 코드베이스·사용자·도메인에서만 얻은 검증. 목적(임의 프로젝트)으로 일반화되지 않는 약한 증거.
- **zero-start** — 축적 집합을 신뢰해 걸러내지 않고, 불변식만 지고 백지에서 재시작해 각 기능이 자격을 다시 벌게 하는 재구축 방식.
- **acting 루프 / drive 루프** — 실행·팬아웃(위임) vs 완료·종료 판단(소유)의 두 루프.
- **증거 판정 vs transcript 판정** — 완료를 하네스가 직접 실행·확인한 증거로 판정 vs 모델이 말한 것으로 판정.
- **설계 리트머스(§4.3)** — 어떤 루프든 데모가 지운 넷(제약·oracle·시간·이해관계자)을 도로 부과하고 증거로 종료하는가.
- **완결성 / disposition-completeness(§5.4)** — 발견된 모든 결함·이슈·미검증이 명시적·정당화된 처분(해결/결정-실행/에스컬레이션)에 도달하기 전엔 완료를 주장할 수 없음. 완료의 반대는 미완이 아니라 "미결로 얼어붙음".
- **disposition(처분)** — 발견된 항목의 최종 귀결. 합법: 해결·결정후실행·에스컬레이션. 불법: 표시하고 멈춤(방치·relabel·unverified 종결).
- **coherent-increment 조건** — 후속으로 미루는 것이 합법이려면 그것 없이도 현재 증분이 깨지지 않아야 함. 후속 증식을 bound하는 장치.
- **capture-at-source(§5.6)** — 미완의 원인은 구멍이 생기는 그 순간에만 값싸게 잡히고 이후 회복 불가로 붕괴하므로, 완결성은 사후 정리가 아니라 그 자리에서 집행돼야 함.
- **park / 도태(attrition)(§5.7)** — park=명명된 blocker에 유예된 legible 종료 상태(main에 안 올라감, off-트렁크). 도태=장기 park의 resolvability가 붕괴하기 전에 resolve-or-discard를 강제하는 압력. 기본값은 폐기(cull)이되 기록되는 강제 결정이지 조용한 삭제가 아님.
- **main = resolved-only 불변식(§5.7)** — main 브랜치엔 resolved 의도만 착지. park·미완은 트렁크에 안 섞임.
- **컨트롤 타워 / 무손실 핸드오프(§5.12)** — 메인 에이전트는 각 동작의 계약만 보고 위임·종합하는 컨트롤 타워(벌크 작업은 격리 subagent에, 결론·증거만 회수), 세션 경계는 원 의도를 축자 보존하는 무손실·pull 방식 핸드오프로 인계. context rot 관리를 개발 규율이 아니라 제공 기능으로.
