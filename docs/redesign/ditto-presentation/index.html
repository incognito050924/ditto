DITTO

코딩 에이전트를 실제 프로젝트에서 신뢰할 수 있는 작업 수행자로 만드는 작업 조율 계층

의도 · 증거 · 위임 · 기억
목차

오늘의 흐름

오른쪽 위 `목차` 버튼으로 발표 중 언제든 원하는 장표로 바로 이동할 수 있습니다.

배경

바이브 코딩은 빠른 출발점

강한 곳

아이디어를 바로 만져보고, 낮은 위험의 도구나 프로토타입을 빠르게 만든다.

약한 곳

오래 유지될 제품 코드에서는 품질, 보안, 결정의 이유, 변경 범위가 남아야 한다.

빠르게 만드는 능력과 오래 유지하는 능력은 같은 문제가 아닙니다.

배경

제품 코드는 계속 살아남아야 한다

오늘 동작사용자가 기대한 결과를 낸다.
내일 변경다음 요구사항을 받아들일 수 있다.
이유 보존왜 이렇게 만들었는지 남아 있다.
품질 유지보안, 성능, 구조가 무너지지 않는다.

에이전틱 코딩의 기준은 “만들었는가”에서 “계속 다룰 수 있는가”로 올라가야 합니다.

권위

무엇을 믿고 다음 변경을 할 것인가

실행되는 코드최종 동작의 기준
테스트와 설정동작과 환경의 기준
결정의 이유코드만으로 복원하기 어려운 맥락
낡은 문서검증 없이 믿으면 위험한 기억

문서가 문제가 아니라, 낡은 문서가 권위처럼 쓰이는 상황이 문제입니다.

문서

문서는 셋 중 하나여야 한다

1살아있는 기록코드 변화와 함께 갱신된다.
2결정의 이유왜 이 선택을 했는지 남긴다.
3임시 산출물소비한 뒤 정리하거나 폐기한다.

이 구분이 없으면 에이전트는 낡은 맥락을 근거로 그럴듯하지만 틀린 결정을 내립니다.

품질

에이전트가 잘 일하는 코드베이스

이해 가능사람도 에이전트도 구조를 빠르게 잡는다.
적절한 추상화복잡도를 줄이고 변경 지점을 좁힌다.
명확한 규칙우연한 스타일과 즉흥 변경을 줄인다.
남는 결정코드 뒤의 이유가 다음 작업으로 이어진다.

좋은 코드베이스는 한 번 만드는 산출물이 아니라 계속 가꾸는 상태입니다.

루프

반복은 출구를 강하게 만들지만, 과정 전체를 보장하지는 않는다

잘 맞는 반복

성공 기준이 분명하고, 자동으로 확인할 수 있고, 기존 코드와 충돌이 적은 작업.

부족한 반복

제품 판단, 설계 품질, 보안, 장기 유지보수처럼 통과/실패만으로 닫기 어려운 작업.

반복 실행에는 의도 확인, 범위 제한, 결정 기록, 독립된 검증이 붙어야 합니다.

의도

요구사항을 받아 적기 전에 의미를 고정한다

목적사용자가 정말 얻고 싶은 결과와 피하고 싶은 상태를 분리한다.
제약바뀌면 안 되는 구조, 일정, 운영 조건, 사용자의 우선순위를 확인한다.
질문 지도모호한 표현을 질문으로 바꾸고, 답이 필요한 지점을 드러낸다.
반대 관점그럴듯하지만 틀릴 수 있는 해석을 먼저 세워보고 깨뜨린다.

DITTO의 의도 인터뷰와 관점 점검은 “무엇을 만들까”보다 “성공이 무엇인가”를 먼저 묻습니다.

위험

완료 기준은 잠재 실패를 먼저 볼 때 높아진다

가장 그럴듯한 실패이 변경이 틀어진다면 무엇 때문인지 먼저 적는다.
먼 영향보안, 성능, 데이터, 배포, 운영 흐름까지 영향을 넓혀 본다.
숨은 기대사용자가 말하지 않았지만 깨지면 안 되는 흐름을 찾는다.
판단 분리자동 검증, 사람 판단, 후속 과제로 남길 일을 나눈다.
성공 기준 보강위험을 확인 가능한 완료 조건으로 바꾼다.
미확인 항목 표시확인하지 못한 것은 통과로 처리하지 않고 남긴다.

사전 실패 점검은 불안을 늘리는 절차가 아니라, “완료”라는 말의 기준을 올리는 절차입니다.

연결

문제의식이 실제 표면으로 이어진다

의도가 흐림모호한 요청을 질문 지도와 확정된 목표로 바꾼다.ditto:prism · ditto:deep-interview
작업이 흩어짐요청을 작업 항목으로 만들고 상태, 범위, 후속 항목을 추적한다.ditto work
완료가 모호함실행한 명령과 브라우저 흐름을 완료 조건의 증거로 남긴다.ditto verify · ditto e2e
세션이 끊김다음 세션이 같은 의도와 상태에서 이어받게 한다.ditto handoff
결정이 사라짐용어, 결정, 코드 관계를 다음 작업에서 다시 찾을 수 있게 한다.ditto knowledge · ditto memory
운영 경계가 위험함설치 상태, 분리 작업 공간, 원격 반영 경계를 점검한다.ditto doctor · ditto worktree · ditto push-gate

DITTO 설명은 추상 가치가 아니라 사용자가 실제로 만나는 표면에서 출발해야 합니다.

문제

그럴듯함이 옳음을 이기는 순간

DITTO가 막는 실패

의도 왜곡, 증거 없는 완료, 긴 대화로 인한 판단 흐림, 변경 흩어짐, 지식 휘발.

요청을 조용히 줄이지 않는다. 검증하지 않은 완료를 통과시키지 않는다. 자기 맥락에서 자기 작업을 채점하지 않는다. 세션이 끝나도 결정과 용어를 남긴다.

DITTO는 모델의 실패 양식을 구조로 다루는 제품입니다.

사상

의도, 범위, 증거, 기억

의도를 잠근다명령보다 결과를 먼저 확인한다.
작게 바꾼다요청 밖 변경과 얕은 추상화를 줄인다.
증거로 말한다완료는 테스트, 빌드, 로그, 재현 결과로만 인정한다.
기억을 남긴다결정, 용어, 근거를 코드 곁에 축적한다.

빠르게 코드를 뱉는 에이전트를 의도에 충실하고 증거로 말하는 협업자로 바꿉니다.

대상

길고, 흔적이 필요하고, 실패 비용이 있는 일

잘 맞는 경우

기능 단위 위임, 장기 리팩터링, 여러 세션에 걸친 프로젝트 관리, 팀 단위 에이전트 도입.

과할 수 있는 경우

5분짜리 일회성 코드 생성, 버릴 프로토타입, 성공 기준을 따질 필요가 없는 실험.

DITTO는 모든 프롬프트에 붙이는 장식이 아니라, 책임 있는 작업 단위에 붙이는 실행 규율입니다.

사용법

사용자는 표면을 따라 작업을 닫는다

1ditto work요청을 작업 항목으로 만들고 상태, 목표, 후속 항목을 추적한다.
2ditto:prism모호한 요청은 질문 지도와 설계 문서로 정리한다.
3ditto verify테스트, 빌드, 실행 명령을 완료 조건의 증거로 기록한다.
4ditto work done증거, 미확인 항목, 남은 위험을 묶어 종료한다.

복잡한 작업은 중간에 의도 인터뷰, 자동 실행, 브라우저 흐름 검증, 인계 표면이 추가됩니다.

기능

사용자가 실제로 만나는 DITTO

install / setup설치와 프로젝트 연결

플러그인, CLI, 프로젝트의 `.ditto/` 상태를 준비한다.

work작업 항목 생명주기

시작, 기준 설정, 상태 확인, 완료, 포기, 재개, 후속 항목을 다룬다.

prism모호한 요청 정리

열린 쟁점을 사용자 언어로 보여주고, 승인된 설계를 작업 항목으로 나눈다.

deep-interview착수 전 의도 확정

성공 기준을 쓰기 어려운 요청에서 질문과 사전 실패 점검을 거친다.

verify / e2e증거 기록

명령 실행 결과와 실제 브라우저 사용자 흐름을 완료 근거로 남긴다.

handoff / knowledge이어받기와 기억

세션 경계의 맥락, 합의한 용어, 중요한 결정을 남긴다.

DITTO는 하나의 버튼이 아니라 작업 전후를 감싸는 여러 사용자 표면의 묶음입니다.

경로

신중한 경로와 빠른 경로

신중한 경로

work → prism 또는 deep-interview → 자동 실행 → work done

모호함, 되돌리기 어려운 위험, 여러 화면이나 모듈에 걸친 장시간 작업

빠른 경로

work start → work set-criteria → verify → work done

작고 되돌릴 수 있으며 완료 기준이 명확한 작업

완료 계약

성공 기준별 판정, 변경 파일, 검증 명령, 확인 못 한 항목, 남은 위험, 이어받기 정보.

전체 완료는 모든 성공 기준이 검증 기록과 함께 통과했을 때만 인정됩니다.

사용법

표면별로 무엇을 보장하는가

표면사용자가 하는 일DITTO가 붙이는 보장
ditto work작업을 만들고, 상태를 보고, 완료하거나 재개한다.요청, 완료 조건, 후속 항목, 종료 상태를 한 작업 항목 안에 묶는다.
ditto:prism애매한 요청을 설계 문서와 작은 작업으로 정리한다.남은 쟁점을 사용자 언어로 보여주고, 명시 승인 전에는 쪼개거나 확정하지 않는다.
ditto:deep-interview착수 전 질문에 답해 의도를 확정한다.성공 기준, 제약, 잠재 실패를 통과 가능한 작업 기준으로 만든다.
ditto verify테스트나 빌드 명령을 실행해 기록한다.완료 주장을 방금 실행한 명령의 결과에 묶고, 빈 검증 명령은 거부한다.
ditto e2e사용자 여정을 브라우저 테스트로 만들고 실행한다.화면 흐름 변경이 실제 사용자 흐름을 깨뜨리는지 따로 확인한다.
handoff / memory이어받기 문서와 검색 가능한 맥락을 남긴다.세션이 바뀌어도 원 의도와 결정 근거를 다시 추측하지 않게 한다.

가치는 기능명 뒤에 붙는 설명이고, 설명의 기준점은 사용자가 실제로 만지는 표면입니다.

동작

규칙 엔진, 판단 연결부, 실행 환경

규칙 엔진데이터 형식, 점검 규칙, 상태, 완료 계약
판단 연결부모델에게 물을 지점과 답을 받는 형식
실행 환경Codex, Claude Code 같은 실제 작업 공간
변경 통제변경 계약, 기록된 결정과의 충돌, 원격 반영 경계

실행 환경이 바뀌어도 지켜야 할 것은 규율과 데이터 형식입니다.

Hook

Claude Code Hook은 실행 흐름의 특정 지점에 걸린다

세션 시작SessionStart · InstructionsLoaded
사용자 입력UserPromptSubmit · UserPromptExpansion
도구 사용 전PreToolUse · PermissionRequest · PermissionDenied
도구 사용 후PostToolUse · PostToolUseFailure · PostToolBatch
서브 작업SubagentStart · SubagentStop · TaskStart · TaskStop
종료와 압축Stop · StopFailure · PreCompact · PostCompact · SessionEnd

Hook은 대화 바깥의 실행 지점이다. 어떤 Hook은 컨텍스트를 더하고, 어떤 Hook은 실행을 막거나 기록만 남긴다.

Hook

Claude Code가 제공하는 Hook 표면

세션SessionStart, SessionEnd, InstructionsLoaded, CwdChanged
프롬프트UserPromptSubmit, UserPromptExpansion
도구PreToolUse, PostToolUse, PostToolUseFailure, PostToolBatch, PermissionRequest, PermissionDenied
표시·알림MessageDisplay, Notification
서브에이전트·작업SubagentStart, SubagentStop, TaskStart, TaskStop
파일·설정·MCPFileChanged, SettingsChange, MCPServerStart, MCPServerStop
정지·압축Stop, StopFailure, PreCompact, PostCompact

DITTO는 이 전체 표면을 다 쓰지 않고, 작업 조율에 필요한 지점만 골라 쓴다.

Hook

DITTO가 실제로 쓰는 Hook과 적용되는 문맥

Hook시점적용되는 문맥
SessionStart세션 시작DITTO 모드 배너, worktree 안 세션의 작업 항목 자동 연결 안내.
UserPromptSubmit사용자 입력 직후활성 작업 항목, 재개/생성 안내, 의도 인터뷰 필요 신호, 먼저 확인할 수 있는 질문 안내.
PreToolUse도구 실행 전비밀 파일, 저장소 밖 쓰기, 파괴적 명령, 변경 범위 위반, 진행 중 작업 범위 밖 편집 차단.
PostToolUse도구 실행 후실행 명령과 편집 파일을 작업 항목의 증거 기록으로 남김.
PreCompact컨텍스트 압축 전활성 작업의 의도, 상태, 다음 확인 지점을 handoff 문서로 저장.
Stop모델이 멈추려 할 때남은 작업, 검증 누락, 의도 drift, 고위험 미해결 항목을 보고 계속 진행시킴.

DITTO는 prompt hook을 쓰기보다 command hook으로 추가 컨텍스트와 차단 피드백을 모델에게 돌려준다.

전후

완료의 의미 변화

그냥 에이전트DITTO가 붙은 에이전트
“수정했습니다”성공 기준별 통과 여부와 검증 기록을 제시
긴 대화가 곧 맥락목표, 범위, 관련 파일, 검증 기록을 묶어 의도를 운반
문제 발견 시 즉흥 확장후속 과제를 기록하는 일과 지금 처리하는 일을 분리
세션이 끝나면 결정이 휘발공유 용어, 중요한 결정, 코드 관계로 축적
한계

잘 맞는 문제, 덜 맞는 문제

판정 기준이 약한 문제제품 판단과 설계 미학은 통과/실패로 닫기 어렵다.
작은 일회성 작업게이트 비용이 이득보다 클 수 있다.
도구 부재브라우저나 정적분석 엔진이 없으면 할 수 있는 검증 수준을 낮춰서 표시한다.
불명확한 목표무엇을 성공으로 볼지 합의되지 않으면 게이트도 약해진다.

잠재 사용자에게는 어디에 쓰면 좋은가만큼 어디에는 과한가도 같이 말해야 합니다.

사람

방향을 제시한다

코드를 일일이 쓰는 사람에서, 방향을 정하는 사람으로

프로젝트와 제품은 어떤 형태여야 하는가

무엇을 만들고 싶은가 · 왜 만들고 싶은가 · 어떤 모습에 가까워져야 하는가

코드는 에이전트가 많이 만들 수 있다. 방향은 사람이 말해야 한다.

평가

현재 상태를 평가한다

돌아간다 ≠ 원하는 제품에 가까워졌다

지금의 제품 · 프로젝트 · 코드베이스는 내가 그리는 그림과 맞는가

목표와 현재 상태 사이의 차이를 발견하고 줄이는 일

기능 동작 여부보다, 목표와의 거리감을 보는 일이 중요해진다.

환경

환경과 방법을 제시한다

에이전트가 더 나은 결과를 만들 수 있는 환경

명확한 목표 · 검증 가능한 완료 기준 · 읽히는 코드베이스

살아있는 문서 · 안전한 실행 경계 · 좋은 결과가 나올 수 있는 방법

DITTO는 이 조건을 의도, 실행, 검증, 기억으로 잡아 보려는 시도다.

정리

의도, 증거, 컨텍스트, 기억

의도를 잃지 말 것
증거로만 완료할 것
컨텍스트를 격리할 것
결정을 코드 곁에 남길 것

DITTO가 지키는 것은 이 네 가지 규율입니다.
