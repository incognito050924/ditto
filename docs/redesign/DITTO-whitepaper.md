# DITTO 백서 — 하네스 재설계를 위한 철학·가치·개념 정본

> **이 문서의 위치**: DITTO를 처음부터 다시 지으려 할 때 *무엇을 반드시 보존해야 하는가*를 정의한다.
> DITTO의 헌장은 스스로 "특정 모델·특정 하네스·특정 저장소 이름에 의존하지 않는다"고 못박는다. 그래서 이 백서는 현재 구현(39개 CLI 커맨드, CodeQL, zod, citty, bun 등)을 *설명하지 않는다*. 그것들은 철학을 실현한 하나의 realization일 뿐이며, 재설계에서 교체 대상이다. 여기 담는 것은 하네스가 바뀌어도 남아야 하는 **문제 정의 · 사용자 가치 · 철학 · 개념 · 불변식**이다.
> **권위는 코드에 있다(헌장 §4-11)**: 이 문서가 인용하는 결정의 원본은 `.ditto/knowledge/adr/`의 ADR과 `.ditto/knowledge/CONTEXT.md`의 glossary다. 사실이 흔들리면 그쪽을 연다. 현재 구현과의 연결은 곁 문서 [feature-mapping.md](./feature-mapping.md)에 있다.

---

## 1. 한 줄 정의

**DITTO는 코딩 에이전트를 "실제 프로젝트에서 신뢰할 수 있는 작업 수행자"로 만드는 오케스트레이션 계층이다.** 모델이 코드를 쓰게 하는 것이 아니라, 모델이 *사용자의 의도를 잃지 않고, 증거로만 완료를 말하며, 대규모·장시간 작업을 자율로 끝까지 몰고, 그 과정에서 지식을 축적하도록* 강제하는 규율과 기계를 제공한다.

---

## 2. 문제 정의 — 코딩 에이전트 단독은 왜 신뢰할 수 없는가

강한 모델을 그대로 프로젝트에 붙이면 다섯 가지 실패가 반복된다. DITTO의 모든 설계는 이 다섯을 막기 위한 것이다.

1. **의도 왜곡·조용한 축소.** 에이전트는 모호한 요청을 묻지 않고 하나의 해석으로 고정하고, 큰 요청을 임의로 줄여 "나머지는 다음에"로 미룬다. 사용자가 원한 것과 다른 것이, 다 됐다는 얼굴로 나온다.
2. **증거 없는 완료.** "수정했다 / 될 것이다 / 대충 맞다"가 완료로 통용된다. 검증하지 않은 것과 검증한 것이 구분 없이 섞인다.
3. **컨텍스트 rot와 자기확신.** 맥락이 길어질수록 성능이 떨어지고(한도 도달 전부터, 비균일하게), 쌓인 맥락이 prior가 되어 판단을 자기 서사 쪽으로 끌어당긴다. 자기 작업을 자기 맥락 안에서 검증하면 통과가 보장된다 — 검증이 무의미해진다.
4. **변경의 흩어짐.** 요청과 무관한 리팩터·포맷·"개선"이 섞이고, 변경한 줄에 이유를 댈 수 없게 된다. 리뷰와 되돌리기가 불가능해진다.
5. **지식의 휘발.** 매 세션이 백지에서 시작한다. 왜 이렇게 결정했는지, 무엇이 이미 시도되어 기각됐는지가 남지 않아 같은 실수를 반복한다.

**공통 뿌리**: 에이전트에게는 "그럴듯함"이 "옳음"을 이긴다. DITTO는 그럴듯함이 통하지 않도록 *게이트·증거·위임*을 구조로 박는다.

## 3. 사용자에게 제공하는 가치 (value proposition)

DITTO의 목적은 도그푸딩(자기 자신으로 자기를 개발)이 아니다. **DITTO가 만드는 모든 가치는 궁극적으로 사용자 프로젝트에서 발현될 수 있는 형태여야 한다**(ADR-20260713). 이 repo에서만 작동하고 남의 프로젝트로 옮겨갈 수 없는 것은 가치로서 미완이다.

사용자가 실제로 얻는 것:

- **의도를 잃지 않는다.** 요청은 착수 전에 검증 가능한 목표(acceptance criteria)로 잠기고, 그 뒤 에이전트는 그것을 임의로 넓히거나 줄이지 못한다. 큰 일이 큰 일 그대로 끝까지 처리된다.
- **완료가 진짜 완료다.** 모든 완료 주장은 fresh evidence(테스트·빌드·실행·재현)로만 인정된다. "검증 못 함"은 `unverified`로 정직하게 남는다.
- **긴 일을 맡길 수 있다.** 승인된 작업 단위 안에서 에이전트는 절차 결정을 되묻지 않고 정지 조건까지 자율로 몬다 — 사용자는 값진 결정(제품 가치·도메인 의미·비가역 판단)에만 개입한다.
- **변경이 외과적이다.** 모든 변경은 요청과 직접 연결되고, 리뷰 가능하고, 되돌릴 수 있다.
- **프로젝트가 기억을 얻는다.** 매 변경마다 결정·용어·관계가 코드 곁 지식으로 축적되어, 단편적 사고가 프로젝트 메모리로 재탄생한다.

**한 문장**: DITTO는 "빠르게 코드를 뱉는 에이전트"를 "의도에 충실하고 증거로 말하는 협업자"로 바꾼다.

## 4. 핵심 철학 (durable principles)

헌장에서 추출한, 하네스와 무관하게 보존해야 할 행동 공리다. 재설계는 이 원칙들을 *코드로 집행*할 방법을 다시 짜는 일이다.

1. **의도 먼저 (intent over instruction).** 명령을 그대로 수행하기 전에 그 명령이 달성하려는 결과를 파악한다. 성공 기준이 약하면 먼저 강화한다.
2. **모호함을 숨기지 않는다.** 해석이 둘 이상이고 결과가 갈리면 드러낸다. 낮은 위험의 구현 세부는 에이전트가 판단하고, 제품 가치·도메인 의미·비가역 결정만 사용자에게 확인한다.
3. **가장 간단하되 검증된 최소 구현 (공리).** 에이전트의 가장 흔한 실패는 과소가 아니라 과잉이다. 요청되지 않은 기능·설정·확장성, 단일 사용 추상화, 얕은 추상화, 일어날 수 없는 시나리오용 방어 코드를 만들지 않는다.
4. **외과적 변경.** 변경은 요청과 직접 연결된 부분으로 제한한다. 수정한 모든 줄에 이유를 댈 수 있어야 한다. 관련 없는 리팩터·포맷·죽은 코드 정리를 취향으로 하지 않는다.
5. **완료는 증거로만.** 테스트·빌드·실행 로그·diff·재현 결과만이 증거다. 검증하지 못했으면 완료라 말하지 않고 제한 이유와 남은 위험을 남긴다.
6. **계획을 조용히 줄이지 않는다 (무축소).** 합의된 의도 단위를 에이전트가 임의로 축소하거나 다음 작업으로 밀지 않는다. 축소가 필요하면 이유·영향도를 사용자에게 의도 차원에서 확인한다.
7. **반론은 협업이다.** 더 단순·안전한 길, 먼저 확인할 사실이 있으면 말하고 더 나은 경로로 진행한다. 다만 반론이 작업보다 커지면 안 된다.
8. **결정 책임은 에이전트, 가치는 사용자.** 구현 세부·절차 결정("이대로 진행할까요?", "커밋할까요?")은 에이전트가 책임진다. 사용자에게는 사용자만 답할 수 있는 것만 묻는다. **commit은 가역이라 에이전트가 소유하고, push는 비가역이라 사용자 허가로만 한다.**
9. **위임으로 컨텍스트를 지킨다.** 컨텍스트는 유한한 예산이자 편향의 원천이다. 탐색·조사·벌크 분석은 서브에이전트에 격리하고 결론·증거·불확실성만 회수한다. 검증·리뷰는 fresh context에서 한다 — 자기 작업을 자기 맥락에서 검증하지 않는다.
10. **기록된 결정과 충돌하면 드러낸다.** 되돌리기 어려운 결정(ADR)은 추론 시점(조사·의도·계획·리뷰)에 일관되게 반영한다. ADR이 금지한 *방법*과 충돌하면 에이전트가 ADR을 따르고, work item의 *목적 자체*가 ADR과 충돌하면 사용자만 풀 수 있다. 어느 쪽이든 충돌은 근거와 함께 출력에 드러낸다.
11. **권위는 코드에 있다.** 동작하는 코드(소스·테스트·스키마)와 코드 곁 살아있는 지침(SKILL·ADR·glossary)이 진실의 원본이다. 코드와 분리되어 동기화되지 않는 설계·기획 문서는 시간이 지나면 drift하므로 권위로 삼지 않는다. 사실을 인용할 때 drift할 문서를 경로로 가리키지 말고 원문 내용을 직접 담는다.
12. **시드와 핸드오프는 의도의 상태로 담는다.** 의도가 아직 형성 중이면(백로그·인터뷰 시드) 원 의도와 검증된 사실만 담고 내 결론·추천은 뺀다(하류 편향 방지). 의도가 확정됐으면(핸드오프) 원 의도를 축자 보존해 운반한다(재도출·조용한 축소 금지).

## 5. 어떻게 그 가치를 전달하는가 (delivery model)

철학은 네 개의 전달 기제로 실현된다. 이 기제들의 *형태*는 재설계 대상이지만 *역할*은 보존 대상이다.

### 5-1. 기능 4축 — 완전 순차 파이프라인 (ADR-0010)

DITTO가 사용자를 위해 하는 일은 네 축이며 **1 → 2 → 3 → 4 순서로** 흐른다.

1. **의도 파악 및 동기화** — 변증론적·경계 질문을 반복해 사용자가 놓친 부분까지 드러내고, 사용자 이해와 완전히 일치할 때까지 인터뷰해 산출물을 만든다. 종료 조건은 *시스템 readiness 게이트(1차) ∧ 사용자 확인(2차)* 둘 다.
2. **자율 실행 오케스트레이션** — 잠긴 의도를 스스로 구체화해 실제 코드베이스를 만든다. 의도 왜곡 없이, 대규모·장시간이라도 끝까지. *자율은 단어 그대로* — 승인된 실행 단위 안에서 절차 결정을 되묻지 않고 정지 조건까지 몬다.
3. **E2E 테스트** — 사용자 의도대로, 최초 계획대로 구현됐는지 확인한다. (진짜 브라우저 E2E 한정 — 웹 UI 없는 프로젝트는 N/A.)
4. **지식 베이스** — 매 변경마다 코드·컨텍스트를 문서화해 프로젝트 메모리를 획득한다. (가치 있는 durable 변경만 — 자동 강제 아님.)

> **두 층위 모델(중요, 재설계의 핵심 경계)**: 위 4축은 **목적 기둥**(무엇을 하는가)이고 canonical하다. 그것을 타겟에서 살아있게 배달하는 **기층**(Hooks/Skills/Agents/State 같은 substrate)은 그 아래 레이어다. **재설계에서 교체하는 것은 기층이다. 기능 4축은 substrate-agnostic이라 하네스가 바뀌어도 영향받지 않는다.** (ADR-0010 D3)

### 5-2. 표준 work-lifecycle 2경로

코드를 바꾸는 모든 작업은 등록된 work item 아래에서, 딱 두 경로 중 하나로만 진행한다.

- **정식(heavy)**: 의도 잠금(deep-interview) → 계획 잠금(pre-mortem) → 자율 구현(autopilot). 모호·비가역·다중 표면·위험 선언 작업.
- **약식(light)**: 관측 가능한 기준 세팅 → 검증 → 완료. 작고 가역적인 작업.
- **금지된 세 번째 경로**: work item 없이 콘솔에서 즉흥적으로 코드를 고치는 것. 추적·완료·회고·정리가 빠져 rot을 남긴다. TDD는 경로 *안의* 구현 기법이지 경로의 대체가 아니다.

경로 선택은 에이전트 판단(advisory, weight 라우팅)이되, 우회만 강제로 막는다. (ADR-20260626)

### 5-3. 완료 계약 (completion contract)

완료 주장에 필수인 필드 집합: acceptance verdict, 변경 파일, 검증 명령, `unverified` 항목, 남은 위험, 다음 핸드오프 경로. **`final_verdict=pass`는 모든 acceptance가 evidence와 함께 pass일 때만 허용된다.** 이것이 "증거로만 완료"를 데이터 구조로 만든 것이다.

### 5-4. 호스트/프로바이더 절연 (host-agnostic seam)

DITTO는 실제 model/tool loop를 직접 돌리지 않는다. LLM 판단이 필요한 지점은 *seam*으로 두고 호스트(claude-code, codex 등)에 위임하며, 결과는 스키마로 받는다. 덕분에 코어는 stack-agnostic하게 남고 두 호스트에서 동작한다(ADR-0016, ADR-0001). **재설계에서 이 절연 경계는 보존해야 한다 — 코어가 특정 provider에 묶이는 순간 철학이 하네스에 종속된다.**

## 6. 핵심 개념 사전 (concepts)

재설계가 다시 구현해야 할 개념들. 정의 원본은 `CONTEXT.md`/glossary.

**작업 단위·상태**
- **request / work item** — request는 사용자가 한 말, work item은 DITTO가 추적하는 정규화 작업 단위(goal·acceptance criteria·실행 기록·evidence·handoff). 한 request에서 여럿이 파생될 수 있다.
- **Record / Run 2-tier** — work item 상태를 *공유·커밋되는 Record*와 *개인·폐기 가능한 Run*으로 분리(ADR-20260706). 협업 산출물과 로컬 실행 흔적을 섞지 않는다.
- **run / run manifest** — 한 번의 provider 호출과 그 권위 기록(provider·git state·prompt·stdout/stderr·diff·verifications·unverified).
- **profile** — 실행 권한 묶음(read-only·workspace-write·networked·reviewer·isolated).

**의도·계획**
- **intent lock** — 착수 전 의도를 확정한 산출물. 이후 무축소 의무의 기준.
- **deep-interview** — 착수 *전에* 모든 진짜 질문(제품 가치·도메인 의미·비가역 결정)을 변증론적으로 소진해 intent를 잠그는 인터뷰. autopilot의 무정지 실행을 가능케 하는 짝.
- **pre-mortem** — "이 일이 실패한다면 왜인가"를 미리 돌려 비가역 위험·커버리지 공백을 front-load하는 절차. 카테고리-완전 종료 + 이진 관련성 게이트(ADR-0023, ADR-20260625).
- **declared risk** — work item이 heavy 경로를 요구하는지 판정하는 위험 신호(non_local·irreversible·unaudited + 미해결 unknown).

**실행·자율**
- **autopilot** — 축2를 구동하는 오케스트레이터. 노드 그래프를 스스로 선택·위임·검증하며 정지 조건까지 끝까지 몬다. 중간 멈춤은 사용자만 답할 결정·비가역 위험·안전 경계에서만.
- **no-auto-pick / materialize≠drive** — autopilot은 발견한 후속을 *물질화(materialize)*는 하되 자동으로 *구동(drive)*하지 않는다. 유일 예외: 자율 실행 중 분류기가 "재현되는 실동작 버그"로 판정한 결함만 같은 run에서 고칠 때까지 몬다(ADR-20260627, ADR-20260714).
- **self-answer 가드** — 사용자에게 묻기 직전, 잠긴 의도·ADR·코드로 답이 유도되는지 먼저 시도한다. 근거로 답 가능하면 답하고(출처를 드러내 거부 가능하게), 사용자만 답 가능하면 묻는다.

**검증·완료**
- **evidence** — 주장을 뒷받침하는 참조. 컨텍스트에 원문을 넣지 않고 path/hash/preview로 참조한다.
- **verdict** — acceptance/review 판정: `pass | partial | fail | unverified`.
- **completion contract** — §5-3.
- **oracle** — acceptance criterion을 기계적으로 판정하는 근거(테스트·명령·관측). gate와 score가 같은 입력에서 출발해야 한다.
- **reviewer output** — generator와 분리된 evaluator의 결과. 가능하면 다른 provider/model family(cross-provider)로.
- **unverified** — "검증 못 함"의 명시적 표기. 완료 주장과 섞지 않는다.
- **self-check** — 응답 발신 전 적용하는 결정적 lint(미합의 용어·추측 단정·미검증 완료·근거 없는 답을 reject).

**위임·컨텍스트**
- **context packet** — 위임에 넘기는 계약(goal·acceptance·git state·relevant files·last failure·what-not-to-touch·evidence pointer·expected output contract). 의도를 대화 릴레이가 아니라 계약 산출물로 운반한다.
- **handoff** — 같은 작업을 다른 세션/PC/에이전트가 이어받게 정리한 문서(pull 방식 — 자동 주입하지 않는다).

**심의·거버넌스**
- **dialectic** — Producer / Opponent / Synthesizer 3역 적대적 심의. 계획·설계·산출물을 커밋 전에 압박한다. Opponent는 다른 맥락·다른 provider로.
- **ACG (Architecture/Change Governance)** — 변경을 아키텍처 계약(ArchitectureSpec·ChangeContract)에 대어 검증하는 거버넌스. 정적 사실은 단일 정적분석 엔진에서 나온다(ADR-0006). 스펙/바인딩 2계층.
- **decision-conflict** — ADR 충돌을 kind×level×mode로 분류·라우팅(따르기/정당화/사용자확인/차단), 근거와 함께 공개(ADR-0020).

**지식·메모리**
- **glossary / CONTEXT / ADR** — 프로젝트 ubiquitous language와 되돌리기 어려운 결정의 영속 기록. ADR 식별자는 불변 파일명(ADR-20260624).
- **memory graph** — 코드·결정·문서가 어떻게 얽혔는지의 provenance·freshness를 담은 그래프. 관계는 INFERRED/AMBIGUOUS로 표시, 쓰기는 propose→approve→re-projection(에이전트가 직접 쓰지 않는다). LLM 추출은 호스트 위임(ADR-0013, ADR-0001).
- **정합성 2축 (freshness)** — 축1=SoT↔파생물, 축2=코드베이스↔SoT. code_drift(신뢰 결함, 억제)와 code_dirty(개발 중 정상, 라벨만)를 구분(ADR-0015).

**E2E**
- **Journey DSL** — 사용자가 여정을 선언하는 Markdown. 사람이 선언 + 에이전트가 스펙으로 변환 + 게이트로 검증(ADR-0014, ADR-20260702).

## 7. 아키텍처 불변식 — 재설계에서 반드시 코드로 보존할 것

철학을 말이 아니라 *기계*로 만드는 규칙들. 재설계가 형태를 바꿔도 이 불변식이 성립하지 않으면 DITTO가 아니다.

1. **완료 = 모든 AC가 evidence와 함께 pass** — 하나라도 미충족·미검증이면 `pass` 불가. 중간 체크포인트가 아니라 work item 전체가 기준.
2. **무축소를 코드로** — 잠긴 의도·계획의 in-scope 잔여를 에이전트가 조용히 축소·이월할 수 없다(종료 완전성 게이트, ADR-20260710). 하나의 의도 = 하나의 단위.
3. **fail-closed 게이트** — 불확실·도구 부재·판정 불가일 때 기본값은 *차단*이다. 통과는 명시적 근거·override로만. (예: pre-mortem 승격, push 게이트, decision-conflict.)
4. **fresh context 검증** — 검증·리뷰는 생성과 다른 맥락에서. 자기 작업의 검증을 자기 맥락(또는 compaction된 맥락)에서 끝내지 않는다. 이것은 효율이 아니라 유효성 조건이다.
5. **증거는 참조로** — evidence는 컨텍스트에 원문을 적재하지 않고 path/hash/preview로 참조한다(context rot 방지).
6. **우아한 강등 (graceful degradation)** — 선택적 외부 도구(정적분석·브라우저 등)의 부재가 사용자 의도 실현을 막지 못한다. 도구가 없으면 정직하게 degrade하고 floor를 지킨다(ADR-0018).
7. **호스트/프로바이더 절연** — 코어는 LLM을 직접 부르지 않는다. 판단은 seam으로 위임하고 결과는 스키마로 받는다. 코어는 stack-agnostic(ADR-0016, ADR-0001, ADR-0008).
8. **tier 격리** — 제품/프로젝트전역/개인 3계층, 공유(Record·커밋)와 개인(Run·gitignore)을 섞지 않는다(ADR-0012, ADR-20260706). session-rooting invariant(ADR-0011).
9. **스키마가 SoT** — 데이터 계약의 단일 진실원은 스키마다(ADR-0002). 동작·계약을 별도 설계 문서로 이중화하지 않는다(=drift).
10. **push는 user-gated, commit은 agent-owned** — 가역/비가역 경계를 권한 경계로 만든다(헌장 §4-8).
11. **메타-도구는 사용자 환경에 맞춘다** — DITTO 기능을 도그푸딩 스택(특정 언어·런타임)에만 작동하게 굳히지 않는다. 임의 사용자 언어·환경에 맞추거나 정직하게 degrade한다.
12. **언어-중립 어휘 강제** — 사용자가 대화에 쓴 + 응답에 노출된 어휘만 공유어휘로 쓴다. 미합의 용어·설명 없는 축약을 self-check가 reject한다.

## 8. 재설계 지침 — 보존 vs 교체, 그리고 고칠 것

### 8-1. 보존 (철학·개념·불변식)
§4의 12원칙, §5-1의 4축과 두 층위 모델, §6의 개념, §7의 12불변식. 이것들이 DITTO의 정체성이다.

### 8-2. 교체 가능 (기층·구현 세부)
현재의 39개 CLI 커맨드 구성, 게이트 배선 방식, 저장 레이아웃, 정적분석 엔진 선택(CodeQL), 스택(citty/zod/bun/smol-toml), 훅 메커니즘의 구체, 플러그인 조립 방식. 이것들은 철학을 실현한 *한 가지 방법*이며 더 나은 방법으로 갈아끼울 수 있다.

### 8-3. 재설계에서 고칠 것 (현 구현의 구조적 부채)
현재 39개 커맨드를 fresh context로 각각 분해했을 때 반복해서 드러난 시스템 수준 문제다(근거는 [../features/README.md](../features/README.md) 하단과 각 문서 §6/§7). 재설계는 이것들을 *처음부터* 피할 기회다.

1. **의도↔구현 drift** — 개념 정의가 코드보다 넓거나 주석·description이 실제 동작을 잘못 안내(예: context packet 개념 8필드 vs 구현 5섹션, refactor의 "auto-commit" 미구현). → 문서·주석이 가리키는 코드 경로가 실재하는지를 게이트로.
2. **CLI 표면 ≠ 실제 소비 경로** — 여러 커맨드가 CLI 직접 소비자 없이 내부 함수·훅·다른 커맨드를 통해서만 발화. "결정적 seam(측정·기록)"과 "실제 게이트(집행)"를 명시적으로 층 분리.
3. **공유 원장 다중 producer full-replace** — 같은 파일을 여러 생산기가 병합 없이 덮어써 last-writer-wins. → append/merge 또는 producer별 네임스페이스.
4. **락 없는 store write = 동시 세션 유실** — 상태 저장이 full-replace에 락이 없어 공유트리 동시 실행에서 유실 가능. → 동시성 모델을 시스템 차원에서 결정.
5. **핵심 가치가 코드가 아니라 절차(SKILL)에만 사는 지점** — 무엇을 코드로 강제하고 무엇을 규율로 둘지의 경계를 명시(집행 가능성 = delegation-enforcement-boundary, ADR-20260628).
6. **enum·taxonomy 수동 lockstep 중복** — 같은 열거·분류가 여러 곳 inline되어 주석 규율로만 동기화. → 단일 SoT로 수렴.
7. **비-JS/비-java 스택 침묵 강등** — symbol 확장·경계 분석이 특정 언어 하드코딩. → §7-11 메타-도구 원칙을 재설계 불변식으로.

### 8-4. 재설계 착수 순서 (권고)
1. §7 불변식을 *먼저* 스키마/타입/게이트로 못박는다 — 이것이 아키텍처의 뼈대다.
2. §5-4 호스트 절연 seam을 정의한다 — 여기서 하네스 종속을 차단한다.
3. 4축을 파이프라인으로 배선하되, 각 축은 seam 뒤의 교체 가능한 구현으로 둔다.
4. §8-3의 7개 부채를 각 층에서 구조적으로 배제한다(사후 패치가 아니라 설계로).

## 9. 한 줄 요약

DITTO는 *의도를 잃지 않고, 증거로만 완료하고, 위임으로 컨텍스트를 지키고, 결정을 코드 곁에 남기는* 규율을 코딩 에이전트에 강제하는 계층이다. 하네스는 이 규율을 배달하는 수단일 뿐 — 재설계에서 바꾸는 것은 배달 방식이고, 지키는 것은 규율이다.
